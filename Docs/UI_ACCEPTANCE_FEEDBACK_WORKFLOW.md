# UI 验收与反馈查验流程

本文档用于 `骰子王` 主流程 UI 修改后的人工验收。其他会话接手时，应按本文档收集截图、检查问题、记录反馈，再决定是否继续改 UI。

## 适用范围

- 主游戏场景 UI：顶部 HUD、左侧骰袋摘要、中央投掷区、结果判断区、右侧关卡信息、底部行动条。
- 投骰阶段：`Ready / Shaking / Stopping / ResultDecision / CheatEdit / Scoring / StageClear / StageFailed`。
- 与 UI 表现相关的骰子展示：未投掷类型卡、结果骰面、类型标识、状态标签、出千选中态、结算动画。
- 不用于重新判断计分规则、骰子 archetype 或市场数值；涉及这些内容时另读 `DICE_ARCHETYPES.md`。

## 启动前读取

1. 读取 `AGENTS.md`。
2. 读取 `PROJECT_CONTEXT.md`。
3. 读取 `GAME_FLOW.md`。
4. 如果验收中涉及骰子效果、计分、市场或出千规则变化，再读取 `DICE_ARCHETYPES.md`。
5. 查看 `git status --short`，确认当前脏文件范围。不要回退别人已有改动。

## 先做离线真值门禁

文字、图标、槽位尺寸或其它像素敏感资源发生变化时，不应直接进入 Unity 反复截图。先用与运行时共用的字库、映射、配置、虚拟坐标和缩放规则生成离线真值板：

1. 从已经批准的预览中截取三到五个代表区域，不用整屏反复比较。
2. 同一张板只放“批准参考 / 当前运行 / 本轮单一变量候选”；如果问题是字形识别，另做不带答案的真实文案盲读板，不再先铺整张 UI 或要求用户在多个模糊方向中二选一。
3. 同时生成 `1920x1080` 与 `1280x720` 结果；低分辨率是硬门禁，不是通过后的补查项。
4. 每轮只调整一个变量族：先粗细 / 密度，再字号 / 间距，最后颜色 / 辉光。
5. 生成 JSON 指标；物理像素、有效字高、越界、缺字或关键对比不合格时直接失败，不进入 Unity。
6. LED 字体必须依次通过四个独立门：像素清晰度、字形身份、基线排版、文字框装入。前一门通过不能替代后一门；字形不可读时不得以缩小网格满足旧框，字形通过也不能掩盖中英数混排上下漂移。

主游戏 LED 字体当前参考实现为：

```text
Assets/ArtSource/Production/ArcadeDiceKing/MainGameFlowV1/RuntimeTools/build_main_game_led_font_atlas.py
Assets/Resources/Art/MainGame/Flow/main_game_led_font_atlas.png
Assets/Resources/Art/MainGame/Flow/main_game_led_font_map.csv
Assets/Resources/Art/MainGame/Flow/main_game_led_font_styles.csv
Assets/ArtSource/Production/ArcadeDiceKing/MainGameFlowV1/LedFontRuntimeV4/
```

自 2026-07-17 起，用户通过的 V4 字形与统一基线是全局默认文字标准；没有单独说明时，主菜单、开场、主流程、设置、市场、悬浮窗、按钮、提示、结果页及旧回退界面的动态文字都必须走 `MainGameLedFont` / `DrawStandardLabel`，不得新增系统字体旁路。明确例外只有骰面实体压印数字，以及已经单独批准的 `DICE KING` 实体点阵招牌。

LED 字体方向校准首轮在 `1280x720` 下仍以 `2 px` 核心灯珠作为主验收基准，主信息与主操作使用正式 `Display` 档。全局接入后的较小文字只能降到已登记的 `Compact / Micro` 整数档：两者保留同一 `16×16` 字形与 V4 基线，使用精确 `1 px` 硬灯珠且必须通过 1:1 盲读；这不等于允许把任意字图缩成分数像素或临时造一套细字体。最终档位同时服从批准参考、信息层级和区域装入结果。

上述尺寸只属于必要条件，不足以单独批准清晰度。任何字体锐度验收还必须满足：目标分辨率完整帧不做二次缩小；代表区域提供 1:1 原始裁切；放大检查只允许整数倍 `Point / NEAREST`；低分辨率字图不得缩放到非整数物理尺寸；硬核心无抗锯齿且先在无辉光状态通过；辉光必须是独立可关闭层，并记录半径与 Alpha。总览缩略图只能导航，不得作为字体锐度通过证据。

清晰度通过后还必须单独验证字形身份：使用当前真实文案，并固定包含 `章 / 额 / 规 / 检 / 骰 / 袋 / 调 / 整 / 结 / 算 / 顺 / 序 / 掷` 等复杂字；提交原生尺寸盲读板和整数倍最近邻检查图。“映射中存在该字符”“没有越界”或“旧框装得下”都不能证明字形可识别。F021 已批准并正式接入的 V4 合同使用 `16×16` hinted 二值字形与统一底线；720p 只允许 `Display / Compact / Micro` 三档整数灯珠几何，不再接受任意连续字号或非整数缩放。

字形身份通过后必须先验证基线排版，再做文字框回排。基线板至少覆盖中文、英文大写、`0–9`、短横、间隔点、中文逗号和句号；中英数共享底线的物理偏差必须为 `0 px`，悬浮标点使用显式中线锚点。修正基线时应移动完整活动字形，并验证字形哈希和亮灯数守恒，不能借机重画已通过的字形。若可读且同线的字形超出当前区域，优先拆分标签 / 数值、扩大终端正文区、分页或增高提示带；禁止重新压回 `7×7–10×10`、分数像素或未登记的临时 `1 px` 细字体。完成回排后才允许生成完整状态真值板并运行一次自动 Unity 冒烟。

## 验收环境

离线真值门禁通过后，才使用 Unity Editor Play Mode 做最终冒烟。当前原型自启动，无需手动挂组件。

1. 打开 `Assets/Scenes/SampleScene.unity`。
2. 进入 Game 视图。
3. 至少检查两个分辨率：
   - `1920x1080`
   - `1280x720`
4. 如果时间允许，补查一个窄窗口或非整数缩放窗口，确认 16:9 安全区没有挤压。
5. 优先使用 `Assets/Editor/WabishMainGameRuntimeCapture.cs` 或等价编辑器 / 批处理入口注入确定状态并自动截帧；不把 Computer Use 鼠标操作作为标准依赖。
6. 每个关键阶段保留截图，建议放在 `Docs/QA/`，命名格式：
   - `YYYYMMDD_ui_ready_1920x1080.png`
   - `YYYYMMDD_ui_result_1920x1080.png`
   - `YYYYMMDD_ui_cheat_1280x720.png`

如果当前机器无法启动 Unity，必须在反馈中明确写明“未做 Play Mode 验证”，并只提交离线同源校准和静态检查结论。截图文件生成成功只表示捕获链路成功，不表示视觉已经通过。

## 标准操作脚本

从新游戏开始，按以下顺序走一遍：

1. 主菜单点击开始新游戏。
2. 进入主游戏 Ready 阶段，截图。
3. 按 `Space` 开始摇骰，进入 Shaking 阶段，截图。
4. 在摇晃阶段连续按几次 `Space`，观察提示和骰盅位置。
5. 等待 Stopping 阶段，截图。
6. 结果锁定进入 ResultDecision 阶段，截图。
7. 打开并关闭牌型表，确认遮罩、关闭按钮和返回状态。
8. 点击出千，进入 CheatEdit 阶段，截图。
9. 选择 1 颗、2 颗、3 颗骰子，确认选中态和计数。
10. 尝试选择第 4 颗，确认不会超过上限。
11. 点击取消出千，确认返回 ResultDecision 且原结果不变。
12. 再次进入出千，选择至少 1 颗并确认，观察是否直接进入 Scoring。
13. 观察 Scoring 阶段逐颗结算动画，截图。
14. 进入 StageClear 或 StageFailed，截图。
15. 如果通关，进入市场，确认市场骰子视觉是否和局内骰子冲突。

## 阶段检查表

### Ready

- 中央投掷区骰子必须是类型卡，不应出现空白骰面底板。
- 六颗基础骰也要有明确类型身份，不能像未加载资源或占位符。
- 左侧只显示实际信息：上场数量、路线、类型构成、骰面摘要、关键状态。
- 右侧不显示无意义的最近反馈；只保留本关规则和牌型表入口。
- 底部行动条只出现当前主动作提示，不重复出现牌型表。
- 玩家第一眼视觉重心应落在中央骰子和底部行动提示，而不是左右侧说明文字。

### Shaking / Stopping

- 骰盅位置和阴影不遮挡关键 UI。
- 摇晃提示能说明当前动作，不出现长段教程文案。
- 左右侧信息不应抢视觉焦点。
- Stopping 阶段的进度或回落提示不应与骰盅重叠。

### ResultDecision

- 每颗骰子必须显示真实点数 pip。
- 骰子类型必须能通过色条、图标或标签区分；不能只看成一排同款骰面。
- 状态标签只显示本次判断有用的信息，如 `目标 3`、`命中 +金`、`阈值 4`、`自身 x2`。
- 基础骰不应出现 `基础 / 基础` 这类重复标签。
- 结果判断区必须清楚显示牌型、倍率、本次得分、结算后进度。
- 底部主动作是直接结算，出千是次动作。
- 右侧最近反馈在结果阶段可以出现，但不能比中央结果更醒目。

### CheatEdit

- 点击可出千骰子后，选中态必须明显，且不遮挡点数和类型。
- 选择计数必须准确显示 `已选 N / 3`。
- 未选择骰子时，确认出千应禁用或明显不可用。
- 选择 3 颗后，第 4 颗不能被加入。
- 取消出千后应回到 ResultDecision，且不消耗出千机会、不改变骰面。
- 临时小骰不能被选择出千。

### Scoring

- 当前正在结算的骰子要有明确强调。
- 浮动 `+score` 不应遮住其它骰子点数或结果区关键信息。
- 龟龟临时小骰如果出现，应和实体骰视觉有差异，并标记为临时。
- 结算完成后，阶段分数和最终进度应与结果预览一致。

### StageClear / StageFailed

- 达标或失败状态必须一眼可见。
- 通关收入、账本、利息等信息不能和本次得分混淆。
- 进入市场或回主菜单的主动作应在底部清楚出现。

### Market 快速回归检查

- 市场中的骰子类型视觉不要和局内结果骰冲突。
- 买入卡应能看清类型、面组、价格和简短效果。
- 卖出区应能看清当前骰袋容量和被选骰子。
- 如果市场 UI 因局内 UI 改动显得风格断裂，应记录为后续项，不必混入本轮主流程验收。

## 分辨率检查

每个截图至少确认：

- LED 正文核心灯珠在 `1280x720` 下不低于 `2 px`，不出现单列忽隐忽现或断点。
- 复杂中文字形已通过真实文案盲读；不能只检查缺字数与装入结果。
- 中文、英文大写和数字在混排压力条中共享底线；短横与间隔点使用明确中线，不出现逐字顶对齐造成的上下跳动。
- UI 文字使用已经批准的命名样式，不再在调用处任意写入 `12 / 14 / 16` 等小字号。
- 文字没有溢出按钮、卡片或面板。
- 面板之间没有互相遮挡。
- 骰子标签不会压住骰面或相邻骰子。
- 顶部 HUD 在 1280x720 下仍能读出金币、账本、出手、出千。
- 牌型表弹窗不会超出安全区。
- 鼠标点击区域和可见按钮位置一致。

### 非整数显示压力

- 用户截图若来自 Unity Game View，必须同时记录渲染分辨率与编辑器 `Scale`；`1920×1080 @ 0.606×` 不能只记为“1080p”。
- 当前固定压力目标为：从原生 `1920×1080` 以双线性缩放到 `1164×654`，模拟反馈现场的 `0.606×` 显示。它只检查主次层级、同级等亮和识别生存，不承担灯珠锐度真值。
- 原生 `1280×720 / 1920×1080` 仍负责点阵锐度、二值核心、基线、裁切和像素装入；压力图通过不能覆盖原生失败，原生通过也不能忽略压力图中关键文字消失。
- L2 / L3 关键角色在运行时不得使用 Micro；压力图中若只有描边 / 辉光可见而核心字不可识别，仍判失败。
- 自动任务状态只证明截图与指标生成成功；与用户批准参考存在明显视觉差异时，必须继续记录为人工待确认。

## 问题分级

- `P0`：阻断操作或核心信息不可读。例如无法结算、出千按钮失效、结果骰点数看不见。
- `P1`：影响主要判断。例如骰子类型难区分、结果区优先级混乱、Ready 阶段仍像空白占位。
- `P2`：影响效率或观感。例如信息重复、间距不均、局部文字偏多。
- `P3`：风格 polish。例如色彩不够统一、装饰细节弱、动画节奏可更顺。

## 通过标准

本轮主流程 UI 验收通过需要同时满足：

- Ready 阶段没有空白骰面占位感。
- ResultDecision 阶段能同时读出点数、类型和本次有效状态。
- 左侧、右侧、底部没有设计说明类无效文案。
- `Ready / ResultDecision / CheatEdit / Scoring / StageClear` 至少在 `1920x1080` 和 `1280x720` 下无明显重叠。
- 操作路径能从开始投骰走到结算或失败。
- 如果未启动 Unity，只能给出“静态检查通过”，不能宣称 UI 验收通过。
- 像素敏感资源必须先通过离线真值板；Unity 冒烟截图与批准参考仍有明显差异时，不能用自动任务状态 `PASS` 代替视觉通过。
- LED 字体必须分别记录“清晰度 / 字形识别 / 基线排版 / 布局装入”四项结果；任一项失败都不能宣称字体或 UI 已通过。
- LED 亮度还必须单独记录“语义角色 / 同级等亮 / 局部对比 / 非整数显示压力”；不得用字形通过代替亮度通过。
