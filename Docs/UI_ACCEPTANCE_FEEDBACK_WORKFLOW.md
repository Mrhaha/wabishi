# UI 验收与反馈查验流程

本文档用于 `骰子王` 主流程 UI 修改后的人工验收。其他会话接手时，应按本文档收集截图、检查问题、记录反馈，再决定是否继续改 UI。

## 适用范围

- 主游戏场景 UI：顶部 HUD、左侧骰袋摘要、中央投掷区、结果判断区、右侧关卡信息、底部行动条。
- 投骰阶段：`Ready / Shaking / Stopping / ResultDecision / CheatEdit / Scoring / StageClear / StageFailed`。
- 与 UI 表现相关的骰子展示：未投掷类型卡、结果骰面、类型标识、状态标签、出千选中态、结算动画。
- 不用于重新判断计分规则、骰子 archetype 或市场数值；涉及这些内容时另读 `DICE_ARCHETYPES.md`。

## 启动前读取

1. 读取 `AGENTS.md`。
2. 读取 `PROJECT_CONTEXT.md`。
3. 读取 `GAME_FLOW.md`。
4. 如果验收中涉及骰子效果、计分、市场或出千规则变化，再读取 `DICE_ARCHETYPES.md`。
5. 查看 `git status --short`，确认当前脏文件范围。不要回退别人已有改动。

## 验收环境

优先使用 Unity Editor Play Mode。当前原型自启动，无需手动挂组件。

1. 打开 `Assets/Scenes/SampleScene.unity`。
2. 进入 Game 视图。
3. 至少检查两个分辨率：
   - `1920x1080`
   - `1280x720`
4. 如果时间允许，补查一个窄窗口或非整数缩放窗口，确认 16:9 安全区没有挤压。
5. 每个关键阶段保留截图，建议放在 `Docs/QA/`，命名格式：
   - `YYYYMMDD_ui_ready_1920x1080.png`
   - `YYYYMMDD_ui_result_1920x1080.png`
   - `YYYYMMDD_ui_cheat_1280x720.png`

如果当前机器无法启动 Unity，必须在反馈中明确写明“未做 Play Mode 验证”，并只提交静态检查结论。

## 标准操作脚本

从新游戏开始，按以下顺序走一遍：

1. 主菜单点击开始新游戏。
2. 进入主游戏 Ready 阶段，截图。
3. 按 `Space` 开始摇骰，进入 Shaking 阶段，截图。
4. 在摇晃阶段连续按几次 `Space`，观察提示和骰盅位置。
5. 等待 Stopping 阶段，截图。
6. 结果锁定进入 ResultDecision 阶段，截图。
7. 打开并关闭牌型表，确认遮罩、关闭按钮和返回状态。
8. 点击出千，进入 CheatEdit 阶段，截图。
9. 选择 1 颗、2 颗、3 颗骰子，确认选中态和计数。
10. 尝试选择第 4 颗，确认不会超过上限。
11. 点击取消出千，确认返回 ResultDecision 且原结果不变。
12. 再次进入出千，选择至少 1 颗并确认，观察是否直接进入 Scoring。
13. 观察 Scoring 阶段逐颗结算动画，截图。
14. 进入 StageClear 或 StageFailed，截图。
15. 如果通关，进入市场，确认市场骰子视觉是否和局内骰子冲突。

## 阶段检查表

### Ready

- 中央投掷区骰子必须是类型卡，不应出现空白骰面底板。
- 六颗基础骰也要有明确类型身份，不能像未加载资源或占位符。
- 左侧只显示实际信息：上场数量、路线、类型构成、骰面摘要、关键状态。
- 右侧不显示无意义的最近反馈；只保留本关规则和牌型表入口。
- 底部行动条只出现当前主动作提示，不重复出现牌型表。
- 玩家第一眼视觉重心应落在中央骰子和底部行动提示，而不是左右侧说明文字。

### Shaking / Stopping

- 骰盅位置和阴影不遮挡关键 UI。
- 摇晃提示能说明当前动作，不出现长段教程文案。
- 左右侧信息不应抢视觉焦点。
- Stopping 阶段的进度或回落提示不应与骰盅重叠。

### ResultDecision

- 每颗骰子必须显示真实点数 pip。
- 骰子类型必须能通过色条、图标或标签区分；不能只看成一排同款骰面。
- 状态标签只显示本次判断有用的信息，如 `目标 3`、`命中 +金`、`阈值 4`、`自身 x2`。
- 基础骰不应出现 `基础 / 基础` 这类重复标签。
- 结果判断区必须清楚显示牌型、倍率、本次得分、结算后进度。
- 底部主动作是直接结算，出千是次动作。
- 右侧最近反馈在结果阶段可以出现，但不能比中央结果更醒目。

### CheatEdit

- 点击可出千骰子后，选中态必须明显，且不遮挡点数和类型。
- 选择计数必须准确显示 `已选 N / 3`。
- 未选择骰子时，确认出千应禁用或明显不可用。
- 选择 3 颗后，第 4 颗不能被加入。
- 取消出千后应回到 ResultDecision，且不消耗出千机会、不改变骰面。
- 临时小骰不能被选择出千。

### Scoring

- 当前正在结算的骰子要有明确强调。
- 浮动 `+score` 不应遮住其它骰子点数或结果区关键信息。
- 龟龟临时小骰如果出现，应和实体骰视觉有差异，并标记为临时。
- 结算完成后，阶段分数和最终进度应与结果预览一致。

### StageClear / StageFailed

- 达标或失败状态必须一眼可见。
- 通关收入、账本、利息等信息不能和本次得分混淆。
- 进入市场或回主菜单的主动作应在底部清楚出现。

### Market 快速回归检查

- 市场中的骰子类型视觉不要和局内结果骰冲突。
- 买入卡应能看清类型、面组、价格和简短效果。
- 卖出区应能看清当前骰袋容量和被选骰子。
- 如果市场 UI 因局内 UI 改动显得风格断裂，应记录为后续项，不必混入本轮主流程验收。

## 分辨率检查

每个截图至少确认：

- 文字没有溢出按钮、卡片或面板。
- 面板之间没有互相遮挡。
- 骰子标签不会压住骰面或相邻骰子。
- 顶部 HUD 在 1280x720 下仍能读出金币、账本、出手、出千。
- 牌型表弹窗不会超出安全区。
- 鼠标点击区域和可见按钮位置一致。

## 问题分级

- `P0`：阻断操作或核心信息不可读。例如无法结算、出千按钮失效、结果骰点数看不见。
- `P1`：影响主要判断。例如骰子类型难区分、结果区优先级混乱、Ready 阶段仍像空白占位。
- `P2`：影响效率或观感。例如信息重复、间距不均、局部文字偏多。
- `P3`：风格 polish。例如色彩不够统一、装饰细节弱、动画节奏可更顺。

## 通过标准

本轮主流程 UI 验收通过需要同时满足：

- Ready 阶段没有空白骰面占位感。
- ResultDecision 阶段能同时读出点数、类型和本次有效状态。
- 左侧、右侧、底部没有设计说明类无效文案。
- `Ready / ResultDecision / CheatEdit / Scoring / StageClear` 至少在 `1920x1080` 和 `1280x720` 下无明显重叠。
- 操作路径能从开始投骰走到结算或失败。
- 如果未启动 Unity，只能给出“静态检查通过”，不能宣称 UI 验收通过。
