## 当前摘要

F004 聚焦主流程投骰操作手感。当前方向是把投骰从“按空格延长摇晃时间”改为“第一次按 `Space` 起摇，随后在有限窗口内爽快连打施加固定表现冲量，窗口结束后不再影响摇晃并进入自然回落，最后显示结果”。用户已确认连打只影响表现，不影响骰面概率；窗口结束后的额外输入只给很轻的“显示结果”提示；整体手感偏爽快连打。投骰表现参数必须配置化，不能在代码中写死，并需要支持运行时重载已修改配置，至少用于调试和手感迭代。本案已具备生成生产包的基础，剩余参数可作为首版默认值进入实现后通过配置调参。

## 原始用户意图

用户认为当前摇骰子逻辑手感不好，希望改成：第一下敲击 `Space` 后骰盅开始缓慢晃动；晃动过程中每次敲击都像施加一次固定的力；但存在一个窗口期，超过窗口后不再接受 `Space` 的施加；随后骰盅回落到停止。

## 问题框定

- 玩家问题：当前投骰操作像在管理倒计时，不像亲手敲动骰盅，缺少明确的起摇、加力、停盅节奏。
- 生产问题：需要在不改变骰面概率的前提下，把投骰输入、动画强度、音效和提示状态拆成清晰阶段。
- 当前系统张力：投骰是高频操作，如果连打能影响结果概率，会把构筑游戏变成隐性技巧输入；如果连打完全没有反馈，又会显得手感空。
- 目标体验：玩家第一次按键启动投骰，随后可以爽快连打制造强烈摇晃表现；到达窗口结束后，系统自然接管并清楚提示正在显示结果。

## 讨论时间线

### 第 1 轮：提出投骰手感问题

- 用户方向：希望第一下 `Space` 开始缓慢晃动，晃动过程中每次敲击都施加固定的力，但有窗口期，超过后不再接受施加，之后自然回落停止。
- Codex 回应：将投骰拆为起摇、加力窗口、回落停盅三段，并建议敲击不改变骰面概率，只影响表现。
- 形成的洞察：玩家需要的是“我在敲骰盅”的仪式感和反馈，而不是更复杂的随机控制系统。
- 决策状态：倾向

### 第 2 轮：确认输入边界和手感方向

- 用户方向：确认敲击只影响表现；窗口结束后给很轻的“显示结果”提示；整体偏爽快连打。
- Codex 回应：建议保留固定冲量、强度封顶和轻提示，把窗口结束后的输入反馈从“继续加力”切换为“显示结果中”。
- 形成的洞察：本案首版应优先做好输入节奏、阶段提示、动画强度曲线和音效密度，不应引入概率或数值收益。
- 决策状态：已确认

### 第 3 轮：推荐首版参数和表现层级

- 用户方向：使用 `F004` 作为编号，并要求继续往下推荐。
- Codex 回应：建议首版先按短窗口爽快连打做参数验证：起摇即时响应、约一秒多的加力窗口、固定表现冲量、强度封顶、短回落和轻量“显示结果”提示。
- 形成的洞察：本案最重要的不是数值精确，而是阶段边界清楚。首版参数应便于快速调手感，并能验证玩家是否理解“连打只影响表现”。
- 决策状态：倾向

### 第 4 轮：确认配置化和运行时重载要求

- 用户方向：所有表现参数都必须可配置，不能写死；询问运行时是否可以重加载修改后的配置。
- Codex 回应：确认可支持运行时重载，但不建议依赖 `Resources.Load<TextAsset>` 作为热调入口；建议新增外部可读配置文件，代码保留默认兜底、字段校验和手动重载入口。
- 形成的洞察：F004 的生产重点不只是改摇骰状态机，还要建立可调参边界。表现参数应从代码常量迁出，随机结果仍必须和这些参数隔离。
- 决策状态：已确认

### 第 5 轮：推进生产前流程

- 用户方向：确认继续推进流程。
- Codex 回应：将 F004 拆为配置建立、状态机改造、运行时重载、表现接入、验收验证五个步骤，并建议先做可热调的配置闭环，再精修动画和音效。
- 形成的洞察：本案如果直接先调动画，后续每次改手感都会回到代码修改。正确顺序应先搭好配置和重载边界，再用配置调手感。
- 决策状态：倾向

### 第 6 轮：补齐生产包边界

- 用户方向：继续推进流程。
- Codex 回应：将 F004 推进为可生成生产包状态，补齐首版范围、非范围、角色拆分、非阻塞默认值和分层验收口径。
- 形成的洞察：F004 的关键生产目标不是一次性调出最终手感，而是建立“表现配置可热调、随机结果隔离、投骰阶段清楚”的闭环。只要这条闭环成立，后续手感数值可以低风险快速迭代。
- 决策状态：倾向

## 已确认决策

- 本案编号为 `F004`。
- 第一次按 `Space` 进入起摇，骰盅先以低强度缓慢晃动。
- 加力窗口内，每次按 `Space` 都只施加固定的表现冲量。
- 加力窗口结束后，不再接受 `Space` 对摇晃强度、时长或结果的施加。
- 玩家连打只影响动画、音效、震动或界面反馈，不影响骰面概率。
- 窗口结束后的额外输入只给很轻的“显示结果”提示。
- 整体手感方向偏爽快连打。
- 投骰表现参数必须配置化，不能作为不可改的代码常量写死。
- 需要支持运行时重载投骰表现配置，至少用于调试、手感迭代和快速验收。

## 倾向方向

- 首版采用短而密的加力窗口，让玩家在约一秒多的时间内连续敲击，快速把骰盅推到高强度表现。
- 强度使用封顶值，不因极端连打无限增长。
- 每次有效敲击都应有可见反馈，例如骰盅瞬间偏移、桌面轻震、短促音效或提示条脉冲。
- 窗口即将结束时，可以让提示从“敲击加力”自然过渡为“显示结果”，降低玩家继续狂按的困惑。
- 回落阶段不应太长，应保持投骰节奏紧凑，避免连打后的等待感过重。

## 首版参数建议

| 参数 | 建议值 | 设计意图 |
|---|---:|---|
| 起摇响应 | `0.12s - 0.18s` | 第一下按键后立刻有低强度反馈，避免迟滞 |
| 加力窗口 | `1.25s - 1.45s` | 支持爽快连打，但不拖长每次投骰 |
| 有效冲量上限 | `7 - 9` 次 | 让玩家快速打满表现强度，不要求无限敲 |
| 单次表现冲量 | 约 `0.16 - 0.20` 强度 | 每次敲击都有可见变化，数值最终按动画调 |
| 输入去抖 | `0.05s - 0.07s` | 过滤异常重复输入，仍保留高速连打 |
| 回落停盅 | `0.38s - 0.50s` | 连打后快速进入结果，不制造等待感 |
| 超时轻提示 | `0.10s - 0.15s` 闪烁 | 告诉玩家正在显示结果，不改变状态 |

首版不建议显示“已敲击次数”数字。数字会把玩家注意力引向达成某个隐藏最优次数。更合适的是用骰盅晃动幅度、提示条脉冲、短促音效和桌面轻震表达力度。

## 表现层级建议

| 层级 | 表现 | 备注 |
|---|---|---|
| 骰盅本体 | 横向偏移、轻微抬起、旋转回弹 | 主反馈，所有有效冲量都应驱动 |
| 桌面反馈 | 极短轻震或阴影变化 | 只在中高强度后明显，避免画面过乱 |
| 音效反馈 | 每次有效敲击给短促木桌或骰盅音 | 音量和音高可轻微变化，但不暗示收益 |
| 阶段提示 | “空格起摇”到“敲击空格加力”再到“显示结果” | 提示跟状态机同步 |
| 超时反馈 | 只闪“显示结果”或给极轻音效 | 不能增加强度，避免误解 |

## 配置化要求

投骰表现首版建议新增独立配置，例如 `roll_feedback_config`。配置只控制表现，不参与随机结果、骰面概率、分数、出千或结算。

| 配置组 | 建议字段 | 说明 |
|---|---|---|
| 时间 | `start_response_time`、`input_window_duration`、`stop_duration`、`expired_prompt_duration` | 控制起摇、加力窗口、回落和超时提示 |
| 输入 | `input_debounce`、`max_impulse_count`、`allow_expired_tap_feedback` | 控制连打采样和窗口后轻反馈 |
| 强度 | `base_power`、`impulse_power`、`max_power`、`power_decay_per_second`、`stop_min_power` | 控制摇晃强度曲线 |
| 骰盅运动 | `cup_x_amplitude`、`cup_y_amplitude`、`cup_rotation_amplitude`、`cup_frequency` | 控制主视觉反馈 |
| 桌面反馈 | `table_shake_amplitude`、`table_shake_duration` | 控制屏幕或桌面轻震 |
| 提示反馈 | `prompt_pulse_scale`、`prompt_pulse_duration` | 控制底部提示条脉冲 |
| 音效 | `tap_sound_volume`、`tap_sound_pitch_min`、`tap_sound_pitch_max`、`expired_tap_volume` | 控制有效敲击和超时敲击音效 |

当前 `DiceKingDemo.cs` 中已有 `ShakeStartTime`、`ShakeTapBonus`、`ShakeMaxTime`、`StopDuration` 等硬编码常量。F004 实现时应迁移为配置字段，并保留安全默认值作为配置缺失时的兜底。

## 运行时重载建议

- 运行时重载可做，推荐先做手动重载入口，例如调试键或设置页按钮，而不是首版自动监听文件变化。
- 修改配置后，重载结果建议从下一次投骰开始生效；如果正在 `Shaking` 或 `Stopping` 中，避免即时改当前状态，防止动画强度或计时突然跳变。
- 配置读取建议使用外部可读文件作为调试入口，例如 `StreamingAssets` 或 `persistentDataPath` 下的配置文件；`Resources.Load<TextAsset>` 可作为默认配置兜底，但不适合作为运行中热调文件来源。
- 重载时必须做字段校验和范围钳制。非法值回退到上一次有效配置或默认配置，并写入日志。
- 重载不能重新随机骰面，不能重置当前小关、当前骰袋、出手次数、出千次数、金币或计分状态。

## 生产前流程草案

### 第一步：建立配置闭环

- 新增投骰表现配置文件，建议首版使用 CSV，命名可为 `roll_feedback_config.csv`。
- 代码内保留一份安全默认值，只用于配置缺失或非法时兜底。
- 启动时先加载默认配置，再尝试加载外部调试覆盖配置。
- 设置一个手动重载入口，建议调试键为 `F5`，也可以后续放入设置页。
- 重载成功后写日志，例如“投骰表现配置已重载，下次投骰生效”。

### 第二步：改造投骰状态机

- 当前 `Shaking` 不再用“延长倒计时”逻辑。
- 起摇时创建一份本次投骰配置快照，当前投骰全程只读这份快照。
- `Shaking` 阶段改为固定加力窗口倒计时。
- 有效窗口内的 `Space` 按键增加表现冲量计数和表现强度。
- 窗口结束后进入 `Stopping`，后续 `Space` 只触发轻量提示反馈。
- `Stopping` 使用配置化回落曲线衰减到结果锁定。

### 第三步：接入表现层

- 骰盅运动先接入横向偏移、纵向偏移和轻微旋转。
- 提示条接入三段文案：起摇、加力、显示结果。
- 桌面震动和音效作为第二层，不阻塞首版状态机完成。
- 所有表现读取本次投骰配置快照，不读取全局实时可变配置。

### 第四步：运行时重载策略

- 不在每帧读取文件。
- 调试键触发时重新解析配置，并生成新的全局有效配置。
- 如果当前不在投骰动画中，下一次投骰直接使用新配置。
- 如果当前正在投骰动画中，当前投骰继续使用旧快照，新配置从下一次投骰生效。
- 如果解析失败，保留上一次有效配置，不把坏配置写入当前运行状态。

### 第五步：验收顺序

1. 先验收配置能加载和重载。
2. 再验收状态机阶段边界正确。
3. 再验收连打表现爽快且不影响骰面随机。
4. 最后验收提示、音效和桌面反馈是否过强或过弱。

## 当前推荐结论

F004 当前可以进入生产包生成状态。两个参数可作为首版默认值进入实现，并通过运行时配置继续调参：

- 加力窗口首版默认值倾向 `1.35` 秒。
- 有效冲量上限首版默认值倾向 `8` 次。

这两个值不作为永久设计锁定，不阻塞生产包生成。

## 生产验收草案

### 配置验收

- 删除或改坏投骰表现配置时，游戏仍使用安全默认值。
- 修改配置并触发运行时重载后，下一次投骰使用新参数。
- 正在投骰过程中触发重载，当前投骰不跳变，新参数从下一次投骰生效。
- 旧的 `ShakeStartTime`、`ShakeTapBonus`、`ShakeMaxTime`、`StopDuration` 等硬编码不再作为主逻辑来源。

### 状态机验收

- 第一次 `Space` 起摇，进入低强度晃动。
- 加力窗口内连续 `Space` 会增加表现冲量。
- 加力强度达到上限后继续敲击，不再无限增强。
- 加力窗口结束后进入回落阶段。
- 回落结束后锁定结果并进入结果决策。

### 随机隔离验收

- 同一随机种子下，不同敲击次数得到相同骰面结果。
- 同一随机种子下，不同配置表现参数得到相同骰面结果。
- 运行时重载配置不刷新本次已生成或待生成的骰面随机状态。

### 体验验收

- 玩家能明显感到第一下起摇和后续连打加力的区别。
- 超过窗口后继续敲击，只出现轻量“显示结果”反馈。
- 连打手感偏爽快，但不要求玩家长时间高频敲击。
- 投骰总时长不显著拖慢每关节奏。

## 非阻塞默认值

| 项目 | 首版默认 |
|---|---:|
| 加力窗口 | `1.35` 秒 |
| 有效冲量上限 | `8` 次 |
| 输入去抖 | `0.06` 秒 |
| 回落时长 | `0.45` 秒 |
| 起摇基础强度 | `0.45` |
| 单次冲量强度 | `0.18` |

这些默认值用于实现起步，不需要在生产包前逐项确认。生产目标是让它们可配置、可重载、可验收。

## 已拒绝 / 已后置想法

- 已拒绝让敲击影响骰面概率，因为这会改变游戏重心。
- 已拒绝窗口结束后完全无反馈，因为玩家可能误解为空格失灵。
- 已后置旧的“按空格延长摇晃时间”模式，因为它不能很好表达施力感。

- 是否在首版接入桌面轻震，还是后置到第二轮手感调优。
- 是否需要设置页按钮触发配置重载，还是首版只保留调试键。
- 是否需要为键盘以外输入预留同等手感，例如鼠标点击或手柄按键。

## 设计理由

这次调整符合当前项目方向：投骰是每个小关的核心仪式，但游戏乐趣应来自骰袋构筑、结果判断、出千决策和市场取舍，而不是隐藏的按键技巧。爽快连打可以提升“正在摇骰盅”的参与感，固定表现冲量和概率隔离可以保证玩家不会把投骰误解为可操控随机结果的技巧系统。

## 职能备注

### 策划

需要确认首版窗口时长、有效敲击上限、强度曲线和提示文案。规则上应明确：敲击不影响随机结果。

### 美术

首版可复用当前骰盅资源，通过位移、旋转、缩放、轻微桌面震动和阴影变化表达冲量。后续如果进入正式表现，可补充不同强度下的骰盅姿态或短序列动画。

### 界面体验

需要把底部阶段提示拆成“按空格起摇”“敲击空格加力”“显示结果”三类，并让提示变化跟输入窗口同步。窗口结束后的轻提示应克制，避免让玩家以为仍能操作。

### 程序

需要把现有摇晃逻辑从“延长剩余时间”改为“固定时间窗口 + 表现强度冲量 + 回落衰减”。随机结果生成仍应在结果锁定阶段执行，且不读取敲击次数、敲击节奏或最终强度。

需要新增投骰表现配置结构和加载入口。当前 `Resources.Load<TextAsset>` 适合作为默认配置读取，但运行时重载应优先读取外部文件或调试覆盖文件。重载后若当前正在投骰动画中，建议缓存为下一次投骰生效。

实现顺序应先做配置加载和重载，再改状态机，最后接入动画、提示和音效。这样后续调手感不需要反复修改代码。

## 风险登记

| 风险 | 重要原因 | 可能缓解方式 | 状态 |
|---|---|---|---|
| 玩家误以为连打影响结果 | 连打反馈越强，越容易被理解为技巧输入 | 明确保持概率隔离，提示只写加力和显示结果，不写收益 | 已识别 |
| 连打过累 | 高频操作在长局中重复很多次 | 窗口短、上限低，达到高强度后不要求继续敲 | 待评估 |
| 回落等待感过重 | 爽快连打后如果停盅太慢会拖节奏 | 回落时长控制在短时间内，并用清晰显示结果提示承接 | 待评估 |
| 画面过乱 | 强震动和大位移可能影响可读性 | 强度封顶，骰盅和桌面震动分层控制 | 待评估 |
| 超时输入反馈过强 | 玩家可能以为窗口仍在接受加力 | 超时只闪提示或极轻音效，不增加强度 | 已识别 |
| 配置热重载打断当前投骰 | 正在摇晃时改计时或强度会导致表现突变 | 重载后下一次投骰生效，当前投骰继续使用快照 | 已识别 |
| 配置值非法 | 手感参数可能被改成负数或极端值 | 加载时钳制范围，失败则回退上一次有效配置 | 已识别 |

## 候选验收

- 第一次按 `Space` 后，骰盅立即进入低强度晃动，不直接跳到高强度。
- 加力窗口内连续按 `Space` 能明显提高骰盅表现强度。
- 同样随机种子下，不同敲击次数不改变骰面结果。
- 加力窗口结束后继续按 `Space` 不改变摇晃强度、时长或骰面结果。
- 窗口结束后的额外输入只触发轻量“显示结果”提示。
- 回落阶段能自然衰减到停止，并顺畅进入结果决策界面。
- 连打手感偏爽快，但不要求玩家无限敲击才能得到完整反馈。
- 修改投骰表现配置后，通过运行时重载能让下一次投骰使用新参数。
- 正在投骰过程中重载配置，不应改变当前骰面结果或破坏当前投骰状态。
- 配置缺失或非法时，游戏仍能使用安全默认值继续运行。
