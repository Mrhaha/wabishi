## 当前摘要

开始界面、主游戏九状态、五条动态、V4 LED 字形 / 基线和市场浅色实体键帽均已完成前序门禁与运行时接入。2026-07-22 的市场按钮反馈已收敛为“动作文案固定、整键表达状态、短时提示解释结果或阻断原因”，并完成单张 Pilot 样张：卖出键保持明亮凸起并只显示“卖出”，三枚购买键只显示“购买”且整键压暗内陷，点击受阻键时在按钮附近显示“骰袋已满”。顶部大横幅 V1 因像常驻状态条而内部退回，V2 局部提示版继续作为静态基准；提示动效补充为从触发按钮向上短促飞入、停留后继续轻微上浮并淡出。市场按钮进一步统一为“按钮家族决定基础材质、交互状态决定唯一表现”：同家族、同状态必须共享亮度、饱和度、凸凹、边光、位移和点击反馈，受阻原因只由统一提示解释。该样张保留血契船长满袋仍可购买的真实规则；尚未修改代码、运行资源、玩法、经济或正式主文档。

## 原始用户意图

用户认为 B「午夜旧机厅」可以深化，并要求先提供细节方案。当前最关心的问题是：单个游戏机如何承载开始界面的四项选项，而不是立即继续生成图片。

## 问题框定

- 玩家问题：四项功能必须一眼能懂、容易操作，不能为了旧机台气氛牺牲菜单可发现性。
- 生产问题：开始界面与主游戏必须属于同一台机、同一块显示区域和同一镜头轴线，后续不能依赖大幅重排完成衔接。
- 当前系统张力：真实街机通常没有完整的电脑游戏主菜单，但项目必须保留开始、继续、设置和退出四项通用功能。
- 目标体验：玩家像在深夜唤醒最后一台仍可运行的六骰竞技机；机器本身有故事气味，菜单仍符合现代电脑游戏习惯。

## 讨论时间线

### 第 1 轮：选择午夜旧机厅并聚焦单机台菜单

- 用户方向：优先深化第二套接触图，先解决单个游戏机承载开始界面选项的问题。
- Codex 回应：提出“一屏、两层、一个键”的主案，并保留更偏实体机械控制台的备选结构供比较。
- 形成的洞察：不应让四项功能都变成四枚同尺寸实体按钮。机台外壳负责世界感，显示屏承载游戏流程入口，机框或屏幕页脚承载系统入口，控制台只保留通用确认动作。
- 决策状态：主案待用户确认。

### 第 2 轮：确认生成三版单机台开始界面

- 用户方向：认为当前结构可行，要求保持已确定的午夜旧机厅风格，再生成三版只包含开始界面的接触图。
- Codex 回应：生成 A「CRT 存档槽」、B「机框系统键」和 C「机械记忆终端」三张独立横版样张，未加入主游戏、六骰、目标分或转场。
- 形成的洞察：A 的现代界面层级最清楚；B 把游戏流程入口与系统级实体键分层，物理街机可信度和“最后一台旧机”的故事感最强；C 的工业终端语义会暗示不存在的插卡或硬盘操作。
- 决策状态：样张待用户选型。

### 第 3 轮：选择 A 并规划 UI 切图与主菜单完善

- 用户方向：认为第一张 A「CRT 存档槽」感觉不错，允许后续细化，并希望基于该图开始准备 UI 切图与开始新游戏、继续游戏、设置、退出等主界面 UI 完善；本轮先看方案。

- 形成的洞察：当前 `DiceKingDemo.cs` 继续使用 `OnGUI` 是首轮最低风险路径；需要为主菜单增加独立街机背景、动态焦点、真正的覆盖 / 退出模态框和同机台设置页，但不应修改现有主流程和市场的羊皮纸资源。
- 决策状态：A 结构方向已确认，运行时资源与实现方案待用户批准执行。

### 第 4 轮：A 结构三版细化接触图

- 用户方向：开始细化，以已确认的 A「CRT 存档槽」为共同结构生成三版不同的开始界面接触图；本轮暂时不切图。

- 静态复核修订：A / B 的存档摘要补入“从本关开始”；B 删除容易像进度或节拍轨的顶部刻度；C 将单骰歧义文案改为“开启新的六骰挑战”，并把投币感图标改为“新记录 +”符号；三版远景伪招牌已压暗或清除。

### 第 5 轮：选择 C 并规划雨夜与 CRT 氛围动态

- 用户方向：确认 C「首次启动」可以继续细化，希望补充框外持续雨滴、撞窗后沿玻璃流动的水痕、室内灯光偶发屏闪，以及主机 CRT 偶尔成像模糊或出现轻微信号问题。
- Codex 回应：将氛围拆为持续环境、低频机器老化、交互反馈和转场保护四层；建议“雨一直在，故障偶尔来，交互永远清楚”，避免雨、灯和 CRT 同时抢戏。
- 形成的洞察：C 的留白和稳定焦点适合承受少量外围动态；雨水可持续存在，吊灯和 CRT 故障必须由同一低频调度器互斥触发，标题灯牌保持稳定，菜单文字不能参与故障变形。
- 生产判断：首轮不为当前 `OnGUI` 原型建立 `RenderTexture` 和真实 CRT 后处理；先用 clean plate、透明雨水图集、独立发光层、噪点图集和 1 到 2 像素重影 / 横条偏移模拟。真实模糊、色散和曲面扭曲后置到 CRT 内容改为独立渲染目标之后。
- 决策状态：C 视觉方向已确认；氛围细化方案待用户评审，本轮不生成新图、不切运行时资源、不修改代码。

### 第 6 轮：明确评审图与最终动态的实现方式

- 用户问题：两张评审图如何表现雨水、灯闪和 CRT 故障，以及这些效果后续究竟通过什么方式实现。
- Codex 判断：单靠两张静态图只能冻结构图、图层和故障峰值，无法可靠评审发生频率、持续时间、互斥关系和整体节奏。
- 推荐修订：下一轮主交付改为“一张 C 常态氛围定稿关键帧 + 一段 20 到 30 秒的分层合成动态预览”；六格状态图降为随动态预览一起提供的制作说明，用于标出常态、流滴、灯光掉亮、轻失焦、水平错位和交互保护状态。
- 动态预览实现：使用同机位分层源图进行非运行时合成，循环雨层、随机播放流滴序列、调节独立灯光层透明度，并用局部重影 / 横条偏移模拟 CRT；它用于导演评审，不进入 `Assets/Resources/Art/`，也不代表 Unity 已实现。
- Unity 首轮实现：clean plate + 透明雨水图集 + 独立发光层 + CRT 噪点 / 扫描线覆盖；由更新逻辑驱动帧索引、透明度和低频事件，`OnGUI()` 只负责绘制。真实局部模糊、色散和曲面弯曲仍后置到 `RenderTexture` / Shader 方案。
- 决策状态：实现路径已解释，推荐交付格式待用户确认；本轮不生成新图、视频或运行时资源，不修改代码。

### 第 13 轮：否决组合动态，单独重做吊灯断电表现

- 用户判断：现有动态版本方案不成立，后续需要逐项盯表现；第一项只处理昏黄吊灯，效果应像断电后重新亮起，而不是轻微局部明暗波动。
- 问题诊断：现有关键帧把灯芯、灯罩光晕、左墙受光和地面暖反射烘在同一张图里，后期叠加暗色径向层只能造成局部压暗，无法形成可信的“供电消失”。
- 推荐修订：将事件定义为吊灯局部线路瞬断，不是整间机厅总断电；制作一张不含吊灯暖光的灯灭底图，再独立叠加灯芯、近场光晕和墙面 / 地面反射三层。断电时三层真正归零，冷蓝雨夜与机台 CRT 仍提供环境可见度；复亮时灯芯先亮，近场光晕和环境反射延迟恢复。
- 推荐节奏：约 `0.06s` 熄灭、`0.28s` 停黑、`0.06s` 重新点亮、`0.24s` 暖光恢复，总时长约 `0.64s`；只发生一次，不增加连续抖闪、白闪、黑帧或镜头震动。
- 评审方式：下一轮只做约 5 秒单效果预览，前 2 秒常亮，中间演示一次断电与复亮，随后回到常亮；雨水、CRT 和其它灯光保持静止。
- 决策状态：组合动态样片已拒绝；吊灯单效果方案待用户确认，本轮不生成新素材、不改 Unity。

### 第 20 轮：修正降雨方向并定义运行时参数

- 用户反馈：效果一看起来像雨滴向上运动；后续希望能控制窗外雨幕的密度、速度、远近层次，以及玻璃水滴数量、汇流速度和水流长度。

### 第 23 轮：玻璃水滴改为多区域随机生成

- 用户反馈：效果二当前观感“还可以”，但只有四颗雨滴在固定位置出现并汇成固定水流，不够真实；希望多个随机位置出现水滴并汇流，或在后续 Unity 实现中通过配置控制。
- 问题判断：当前缺口主要是事件分布和重复性，不是水珠材质、撞击环、折射或下流节奏。继续制作一条固定十颗水滴的视频仍不能证明随机性，容易把一次导演演出误当成运行规则。
- 推荐结论：保留 V3 作为单个“撞击—附着—汇流”视觉语言基准；最终玻璃层改为 Unity 程序化事件，不播放固定视频。右窗有效区域内随机选择多个撞击簇，多数水滴只附着，只有相邻水滴的累计水量达到阈值时才生成主流。
- 生成规则：随机撞击使用内缩玻璃区域，默认同时保留 `8–14` 颗可见水珠，分布在 `2–3` 个松散区域；水珠之间满足最小间距，避免规则网格。邻近水珠在合并半径内缓慢吸附，达到 `3–5` 颗等价水量后才触发向下水流；同一时刻最多 `1–2` 条主流，支流为低概率事件。
- 水流规则：路径沿重力向下，加入轻微横向漂移、随机停滞和再次加速；流经其它附着水珠时吸收水量并略微加粗，最大长度使用有效玻璃高度比例控制。未达到阈值的水珠只缓慢淡出或等待后续合并，不要求每颗水滴都变成水流。
- Inspector 常用参数：`ImpactRate`、`ActiveDropletRange`、`SpawnRegion`、`ClusterSpread`、`MergeRadius`、`FlowThreshold`、`FlowSpeedRange`、`FlowLengthRange`、`MaxConcurrentFlows`、`RandomSeed` 和 `LockPreviewSeed`。参数保存在 `ScriptableObject`，不进入 CSV；运行时可随机，评审时可锁定种子复现同一序列。
- 推荐默认值：每秒约 `0.8–1.2` 次撞击、可见水珠 `8–14`、撞击区域 `2–3` 个、合并半径约 `18–32px`、触流水量阈值 `3–5`、同时主流 `1–2` 条、水流长度为有效窗高 `0.25–0.55`、支流概率约 `0.15`、停滞概率约 `0.30`。
- 验收建议：不再用固定离线视频验证随机性。进入 Unity 后录制至少 30 秒玻璃层专项样片，检查撞击位置没有明显循环、不是每颗水滴都汇流、主流不会同时铺满玻璃、固定种子可复现、窗框边界和菜单保持稳定。
- 决策状态：效果二视觉语言可作为条件基准，固定四滴分布不通过终版要求；程序化随机与 Inspector 参数方案待用户确认，尚未修改 Unity 代码或资源。

## 已确认决策

- 当前深化对象为 B「午夜旧机厅」，A 与 C 暂不继续细化，但不视为永久否决。
- 本轮只生成开始界面接触图，不生成主游戏、转场或运行时拆件。
- 开始界面仍需保留开始新游戏、继续游戏、设置和退出四项功能。
- “一屏、两层、一个键”已完成三版美术比较，A「CRT 存档槽」被冻结为本轮结构母版。
- A「CRT 存档槽」已被用户选为后续细化、UI 切图和主菜单完善的结构母版；B 与 C 不再作为本轮实现候选。
- A 结构细化稿中的 C「首次启动」已被用户选为视觉密度与氛围母版；这里的 C 不是早期被拒绝的「机械记忆终端」。
- 开始界面氛围需要包含持续窗雨、玻璃流滴、低频室内供电波动和偶发 CRT 老化感，但这些效果不得改变输入、焦点、存档或游戏状态。
- 正式菜单文字、存档摘要、焦点、禁用、按下和确认状态必须由 Unity 动态绘制，不能烘进美术底图。
- 第 30 轮已替代“主游戏必须沿同一机位推近实体机台”的旧限制：早期 B 只继续提供深海军蓝、暖琥珀、磨损金属和六节点机械语汇，不再约束主游戏的空间、机壳轮廓或 CRT 比例。
- 主游戏不得继承 `骰子 ×3`、皇冠、赔付表式右栏、具体数字或原图按钮语义；这些必须按当前 V0.2 玩法重做。
- 正式进入游戏采用过渡遮罩切换到独立主游戏场景；主游戏可以使用完整画布，不必保留窗户、吊灯、完整机壳或开始菜单 CRT。
- 主游戏常驻信息固定为章节 / 小关、剩余生命、金币、目标分、当前分、六颗实体骰、本关规则和唯一主操作；单次出手规则不再用 `1/1` 常驻显示，临时小骰、市场反馈与家族短标签按状态临时展开。
- 生命规则已确认为小关失败扣 1，仍有生命时恢复该小关开始状态重试，归零才进入 `GameOver`；当前代码尚未实现，初始生命数和恢复来源待定。
- 开始界面动态改为逐项门禁；此前雨、流滴、灯光和 CRT 同片评审的方式已被用户否决，第一项先单独确认吊灯断电与复亮。
- 右窗雨效的雨线越界与非物理磁吸完成 R1 修复后，已由用户人工回归确认基本无问题；雨效切片通过。
- `DICE KING` 顶部灯牌采用 A「点阵灯泡补亮」V2：V1 因过快被否决，V2 改为分区统一压暗、近灭停留、两次失败回亮和约 `1.63s` 供电波动；已进入 Unity 参数化实现，开始界面当前由用户判断为基本没有问题。
- 关间市场动作键的常驻文字只表达动作：卖出键显示“卖出”，购买键显示“购买”；卖价不再写进卖出键，“骰袋已满”等阻断原因不再替换购买键文案。
- 市场可用与受阻状态必须通过整枚键帽形成强对比，不能继续只靠文字颜色或把“骰袋已满”写在键帽上区分。
- 卖出结果与购买受阻原因使用快速出现、自动消失的短时浮动提示；后续同类浮动提示统一走同一个可复用控件，避免各功能各写一套表现。

## 倾向方向

- 界面体验优先时倾向 A：继续游戏主卡、开始新游戏次入口以及屏内页脚设置 / 退出最容易理解，键鼠与手柄焦点也最统一。
- 街机物理可信度和故事气味优先时倾向 B：继续 / 新游戏属于屏幕运行内容，设置 / 退出属于机框实体系统键，统一确认键属于控制台操作。
- 若继续深化 B，建议把中央大号发光卡片改为嵌入机台的单行琥珀 CRT 或机械翻牌选择窗，并把换项控制并入窗框小方向键，消除现代网页卡片感和输入来源不明。
- 无存档时，开始新游戏提升为主入口，继续游戏改为简短的无记录提示，不保留一块巨大的禁用卡片。
- 设置与覆盖存档确认都在同一块显示屏内原位切换，不增加外置弹窗或另一套面板。
- 开始界面与主游戏不再强制共用机台外壳、显示屏边框或镜头中心；二者通过穿屏过渡、共享色彩、六节点信号和声音接力保持连续性。
- 首轮运行时建议保留当前单文件 `OnGUI` 技术路径，不为了主菜单单独迁移整套界面框架；街机主菜单资源放入独立目录，避免覆盖现有主流程与市场资源。
- 有存档时，大主卡显示继续游戏与不超过两行的摘要，开始新游戏降为次入口；无存档时，大主卡显示开始新游戏，继续游戏作为禁用次入口。
- 主游戏倾向在 CRT 信号完全覆盖全屏时切换到独立“骰王竞技台”；开始界面的黑色显示区只承担过渡入口与遮罩，不再决定主游戏的安全画布比例。
- 转场后的主画面不保留窗雨、吊灯和完整机壳。视觉连续性改由深海军蓝底色、暖琥珀主反馈、低饱和青绿状态、方形点阵，以及转场阶段短暂出现的六骰信号共同承担；主游戏常态不保留六节点轨道。
- 原图上方两组数字改为顶部章节 / 小关以及右侧生命、金币；目标与当前只保留在左侧积分塔。右侧多行表改为单张可更换规则插件。
- 主游戏不再绘制悬空结算轨道、六节点状态线或骰槽结算灯。`Scoring` 只通过当前骰与底座机械抬升、局部效果产生、回落和下一骰抬升表达左到右推进。
- 六颗实体骰下方预留窄型状态槽，显示类型色、饲料 / 吞吃 / 吸附等必要短标；详细效果仍由悬浮窗承接，不常驻长说明。
- 临时小骰不占用六个实体槽。结算中从当前来源骰下方展开一条小型“余波托盘”，显示前若干颗临时骰并把超出部分折叠为汇总计数，结算结束后收回。
- 进入小关时的市场反馈使用计分轨道上方短纸带 / 点阵条临时显示，数秒后收起；不永久占用规则插件或六骰区域。
- 独立主游戏场景沿用当前已验证的信息骨架：左侧积分塔、中央六槽骰台、右侧单条规则、顶部窄状态带和底部固定 `Space` 操作条；不恢复牌型表、出千、多次出手或常驻长说明。
- 主游戏第一轮只评审 `Ready / Scoring / StageClear` 三张状态关键帧。`ResultDecision`、停转、市场和复杂家族表现先由状态矩阵约束，等三锚点通过后再补动态与市场。
- 视觉衔接倾向“外壳冷、程序暖”：环境和机壳保持雨夜深蓝，CRT 内部使用受控的暖琥珀、低饱和青绿和少量警示红；常态克制，左到右结算逐级升温，达标后才允许短时峰值。
- 通关奖励倾向从机台票据口、计数器或受控金币槽兑现，避免旧流程样张中的整屏金币堆与赌场式爆灯。
- 主游戏新版优先采用“午夜机械结算台”：六骰与结算游标承担主叙事，计分器、单条规则和阶段操作只做服务层，不再以多个等权面板争夺注意力。
- `1/1` 从主游戏常驻 UI 完全删除；“直计”只保留为计分器旁的小铭牌，不占用独立卡片。
- `ResultDecision` 的预计得分必须独立显示，不能提前改变当前已入账分数；`Scoring` 才允许机械计数器随只读展示事件逐项翻动。
- 首选过渡为“菜单信号收束—六节点依次启动—CRT 全屏覆盖—竞技台组装”，利用全覆盖帧隐藏场景替换，避免为了镜头连续而强行复刻开始界面的几何结构。
- 市场实体键帽建议拆成“可用 / 悬停 / 按下 / 受阻”四态：可用态保留浅米色通电键帽、深墨字和亮边；受阻态整键压暗、降饱和、取消亮边并保持短文案可读，不再仅改变字色。
- 受阻键仍保留一个只用于说明原因的点击命中层：点击不执行交易，只触发一次短时提示。这样视觉上明确不可执行，同时玩家仍能知道原因；重复点击同一原因时刷新当前提示，不堆叠多条。
- 统一短时提示首版建议只承载即时操作结果与阻断原因，不替代悬浮详情、市场中央反馈条、骰子机制短标签或离场效果队列。推荐文案为“已卖出 / 已购买 / 骰袋已满 / 金币不足 / 请先选择骰子”。
- 购买键是否可执行必须读取真实购买资格，不能简单使用“骰袋是否满”判断；血契船长在满袋时仍保持可用态并执行既有购买、监听和吞吃流程。

## 已拒绝 / 已后置想法

- 四项功能全部做成右侧竖排菜单：已被用户指出不符合常见主菜单阅读习惯。
- 四个等宽、等亮的实体大按钮：容易像普通网页按钮换皮，也会增强老虎机控制台联想。
- 在开始界面提前显示完整六骰、目标分、规则插件和资源信息：会让首屏重新变复杂。
- 使用滚轴、拉杆、三盏轮转灯、`777` 或赔付表结构：会把街机误读成赌场机器。
- C「机械记忆终端」不建议作为下一轮母版：两块螺丝槽更像工业终端或硬盘仓，会暗示项目中不存在的插卡操作；只保留其黑玻璃和机械分段材质语言。

## 待回答问题

- 临时骰独立展示已废弃；现有龟龟等玩法产生的临时骰如何压缩为来源骰局部反馈，后置到骰子效果专项，不阻塞当前主游戏表现接入。
- 首发初始生命数是多少，以及首发范围内是否存在恢复生命的来源；这不阻塞共用表现与通关分支接入，但阻塞配置、存档版本和失败重试闭环。

## 设计理由

开始界面继续把开始、继续视为游戏流程入口，把设置、退出视为系统入口；正式进入游戏时，CRT 只作为穿屏启动的入口和全屏遮罩。主游戏使用独立完整画布，通过色彩、点阵、六节点和继电器声音保持血缘，而不复用开始界面的机台几何。主游戏内部再按“本轮资源、关卡目标、六槽操作、关卡规则、阶段操作”分区，使氛围连续性与玩法可读性分别成立。

## 职能备注

### 策划

- 继续游戏只恢复到小关开始，存档摘要必须明确这一边界，不显示会暗示恢复到中途的旧分数或临时状态。
- 有存档时开始新游戏仍需二次确认，默认焦点应落在取消。

### 美术

- 机台保持深海军蓝、磨损金属、暖琥珀显示和雨夜橱窗；不要用皇冠、赔付表、三联灯或投币口作为视觉中心。
- 标题只在机顶灯牌出现一次，显示屏不重复堆叠大标题。
- 正式生产先补 clean plate：无菜单内容的空 CRT、无选中光的机壳、未按下的 Space 键、无动态雨线的环境底图。
- 运行资源控制在约 13 到 16 个：环境大底、雨层、机壳、顶箱发光、CRT 玻璃与遮罩、可复用菜单框与焦点层、游标、四个图标、Space 键帽与键光、共用模态框。
- 窗雨只在右侧窗框遮罩内运动：远雨常驻，近景流滴低频生成；雨线和水痕不能越过窗框、机台或菜单可读区。
- 吊灯、顶箱、CRT 外溢光和 Space 键光必须分层控制。顶箱常态保持稳定，只在选定的短招牌事件中变化；招牌、吊灯与 CRT 峰值同一时刻最多出现一种。
- `DICE KING` 的无光底、字 / 灯段遮罩和辉光必须独立；标题事件与吊灯、未来 CRT 故障互斥，不能把闪光烘进整张机台底图。
- CRT 常驻层只保留低透明扫描线、噪点和极弱亮度呼吸；偶发老化只使用轻失焦、短重影或一条局部水平同步错位，不使用黑帧、白闪、整屏熄灭或大面积彩色雪花。

### 界面体验

- 有存档默认焦点为继续游戏，无存档默认焦点为开始新游戏。
- 焦点使用稳定琥珀描边和小三角游标，不使用追光、节拍脉冲或轮流闪烁。
- 鼠标悬停、键盘方向键和手柄方向键共用同一焦点状态。
- 进入主游戏后必须等待确认键释放，避免同一次 `Space` 穿透并立即触发投骰。
- 菜单进入后默认等待 `1.00s`，任意输入后的 `0.50s`、确认弹层与设置页显示期间、转场期间均暂停吊灯事件；循环关闭时每次进入主菜单只演示一次。
- 持续雨层不参与故障互斥；吊灯当前使用约 `0.98–1.58s` 的快速随机循环。未来接入近景大流滴或 CRT 异常时，同一时刻仍最多出现一种峰值事件，避免叠加成惊吓。
- 建议以 10 秒待机录屏验收：吊灯稳定完成多次断电—复亮循环，随机间隔可感知但不形成机械节拍，焦点与文字始终比氛围变化醒目。

## 候选验收

- 开始界面四项功能在三秒内可找到，开始和继续的层级高于设置与退出。
- 无存档和有存档两种状态都不改变菜单位置，只改变主入口内容与默认焦点。
- 单张静态图可以看出菜单属于机台显示屏和机框，不像贴在场景上的网页面板。
- 开始界面不出现六骰行、目标分、规则插件、`x3`、赔付表或胜利灯效。
- 主游戏切换后使用独立完整画布，并容纳目标 / 当前、生命、金币、六骰、规则插件和短 `Space` 提示；不要求继续处于开始界面显示屏内，且不常驻显示 `1/1`。
- 转场中必须出现一个完全覆盖屏幕的信号帧，使场景替换不可见；首帧和末帧在色彩、六节点母题与声音上能够连续对应。
- 进入主游戏后三秒内能找到目标分、当前分、六颗骰子、本关规则和唯一主操作。
- `Scoring` 中一秒内能辨认当前结算槽位，前一槽局部连锁结束后才移动到下一槽。
- `ResultDecision` 不把预估分画成已入账；只有确认结算后积分塔才开始左到右增长。
- `StageClear` 的分数峰值先完成，固定金币与利息奖励随后兑现；两者不同时爆发。
- 三锚点在 `1920x1080` 和 `1280x720` 下都不重叠，右侧规则不超过一条短句，动态阶段不依赖长悬浮窗阅读。
- 满袋普通商品的购买键仍显示“购买”，但整枚键帽在不看文字时也能立刻与可购买键区分；点击只出现“骰袋已满”，不扣金、不取货、不发布购买事件。
- 满袋血契船长的购买键保持可用，点击继续执行既有特殊购买流程，不被通用满袋表现误伤。
- 卖出键始终只显示“卖出”；成功后钱包数值正常变化，并在动作附近短暂显示“已卖出”，不在键帽或提示中重复卖出金币数。
- 同一锚点连续触发同类提示时替换或续时，不纵向堆叠；提示不遮挡商品、悬浮详情或其它按钮，也不拦截鼠标。
