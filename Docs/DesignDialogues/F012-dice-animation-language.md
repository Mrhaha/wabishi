## 当前摘要

F012 聚焦后续骰子风格统一问题。当前倾向是：不要为每颗骰子各自独立设计完整动画，而是先由基础骰定义一套统一的投骰和结算表现语法，再让不同骰子类型继承相同阶段，并只在类型身份、触发反馈和结算高光上做差异插槽。该方向可概括为“资源复用、环节细分、表现配置继承”：每种骰子应拥有自己的表现配置档，但不默认拥有一整套独立序列帧。首个推荐推进切片是 `F012-01 待机到显点流程`：只处理待机身份态、起摇、旋转循环、停转显点、结果保持和重置，不把结算触发、成长入档、金币飞字或临时小骰兑现塞进同一批。该方向需要继续遵守现有边界：动画层只读真实投骰和真实结算结果，不改变骰面随机、出千、`ScoreDice()`、钱包、成长、临时小骰或存档。

## 原始用户意图

用户提出后续想解决风格统一问题：先用基础骰子定义标准的序列帧动画和结算表现，之后各类型骰子都按这套统一表现框架制作各自动画，并询问这个思路是否可行。随后用户确认希望将该讨论编号定义为 `F012`，并询问该编号是否未被占用。

## 问题框定

- 玩家问题：骰子类型越来越多后，如果每颗骰子的投骰、显点和结算表现都没有统一语法，玩家会觉得风格散、节奏乱，也难以理解哪些反馈属于基础流程、哪些反馈属于类型触发。
- 生产问题：每个骰子类型都制作完整独立动画会快速膨胀资源量和验收成本，也容易让风格失控。
- 当前系统张力：F009 已经建立固定槽位序列帧旋转的底座，F006 / F007 已经建立只读结算展示队列；F012 不能推翻这些边界，而应把它们整理成可复用的动画规范。
- 目标体验：玩家看到任何骰子时，都能识别同一套投骰节拍和结算节拍；同时又能通过类型色条、图标、短标签和触发高光，看出不同骰子的机制身份。

## 讨论时间线

### 第 1 轮：提出统一动画思路

- 用户方向：考虑风格统一，想用基础骰定义标准序列帧动画和结算表现，之后各类型骰子按统一表现制作自己的动画。
- Codex 回应：判断方向可行，但应定义为“统一表现语法”，而不是把基础骰完整动画复制给所有骰子。基础骰负责底层阶段和节奏，各类型骰子只改差异插槽。
- 形成的洞察：真正需要统一的是阶段、节奏、点数可读性、槽位稳定和结算点名规则；真正需要差异的是类型身份、触发反馈和路线高光。
- 决策状态：倾向

### 第 2 轮：确认编号

- 用户方向：询问是否可以定义为 `F012`，以及该编号是否没用过。
- Codex 回应：在仓库中搜索 `F012 / f012`，未发现设计对话、生产包、资源或文档编号占用；只有 `.git` 对象哈希中偶然包含 `f012`，不视为编号占用。随后创建本设计对话记录。
- 形成的洞察：`F012` 可作为本主题稳定编号使用，后续生产包可以沿用 `F012-骰子统一动画语法`。
- 决策状态：已确认编号

### 第 3 轮：序列帧资产粒度

- 用户方向：进一步追问，既然制定的是标准结算演示和旋转流程，后续还可能有其它流程，那么每个不同种类的骰子是否都要有一套自己的序列帧动画。
- Codex 回应：判断不应默认每种骰子都有完整独立序列帧；更合理的是每种骰子有自己的表现配置档，复用共享基础旋转和停转序列帧，再按类型或路线家族叠加身份图标、色条、短标签、触发特效和少量高光模块。
- 形成的洞察：F012 要区分“每个类型都有可识别表现”和“每个类型都有全量独立动画资源”。前者必须有，后者只应在骰子外形、机制触发或商业展示价值确实需要时才做。
- 决策状态：倾向

### 第 4 轮：不同外形骰子的旋转序列帧处理

- 用户方向：举例说明，如果要做骰子旋转序列帧动画，而不同骰子外形都不同，应如何处理序列帧或这个问题。
- Codex 回应：建议把外形差异分级处理。表面标记不同的骰子复用共享旋转序列帧和运行时叠加；同一几何家族内的骰子使用同一套旋转骨架或渲染模板批量出帧；只有轮廓、体积或机制表达明显不同的少数骰子才制作独立序列帧。
- 形成的洞察：F012 的正确生产对象不是“逐颗手画旋转条”，而是“统一旋转模板 + 几何家族 + 批量出帧流程 + 类型表现配置档”。这样可以让不同外形成立，同时避免每新增一颗骰子都重新做全套动画。
- 决策状态：倾向

### 第 5 轮：资源复用与环节细分

- 用户方向：将当前理解总结为“资源的复用 + 环节细分”。
- Codex 回应：判断这个总结成立，但建议再补上一层“表现配置继承”。资源复用解决成本，环节细分解决节奏和可读性，配置继承解决不同骰子如何在同一规范下填入自己的差异。
- 形成的洞察：F012 不只是少做素材，而是把骰子表现拆成可继承的系统：全局默认定义基础节奏，几何家族定义外形动画，路线家族定义触发效果，单个骰子类型只覆盖身份层和必要特例。
- 决策状态：倾向

### 第 6 轮：工作流示例

- 用户方向：要求按照“资源复用 + 环节细分 + 表现配置继承”的思路，给出一个工作流示例。
- Codex 回应：提出一个从单颗新增骰子到批量扩展的样例流程：先拿到骰子规则卡，再做表现分层归类，随后判断是否复用共享资源或家族资源，接着按投前、起转、停转、结果保持、结算点名、触发高光和重置这些阶段填表现表，最后由样张、配置接入和运行验证逐级通过。
- 形成的洞察：F012 的实际生产顺序不应从“画一套动画”开始，而应从“这颗骰子在哪一层覆盖默认表现”开始。资源是否新增是流程中的判定结果，不是默认前提。
- 决策状态：示例

### 第 7 轮：首个推进切片

- 用户方向：询问后续应如何推进，尤其是从待机到摇骰子再到展示点面的流程，如何拆分，如何处理不同骰子的表现，从而省资源、有好表现并保持美术风格统一。
- Codex 回应：建议先把首个切片限定为 `F012-01 待机到显点流程`，暂不包含结算触发和收益兑现。该切片先建立统一阶段契约、共享资源包、几何家族与路线家族分类、表现配置字段和三颗代表骰样张，再进入批量扩展。
- 形成的洞察：推进顺序应为“先定阶段语言，再定资源层级，再定骰子家族，再做少量样张，再批量接入”。如果先按每颗骰子出动画，会同时失去资源效率、风格统一和运行验收边界。
- 决策状态：倾向

## 探索过的方案

| 方案 | 核心思路 | 优点 | 风险 | 状态 |
|---|---|---|---|---|
| 每个骰子独立制作完整动画 | 每个类型都有自己的投骰、停转和结算动画 | 类型差异最大 | 资源量爆炸，风格容易散，普通触发也会抢戏 | 不建议 |
| 复制基础骰完整动画 | 基础骰做一套，其它骰子只换皮 | 成本低，统一性强 | 类型机制不容易被看见，后续会显得同质化 | 不建议作为完整方案 |
| 统一动画语法 + 类型差异插槽 | 基础骰定义阶段、节奏和可读规范；类型只替换身份层和触发层 | 风格统一、成本可控、可支持多类型扩展 | 需要先整理规范和配置表，否则实现仍可能分散 | 倾向 |
| 为路线家族制作少量共享特效 | 龟龟、金币、大树、奇偶、爆发等家族共享触发高光 | 资源复用好，路线身份清楚 | 家族内个体差异需要靠短标签和图标补足 | 倾向 |
| 每种骰子一个表现配置档 | 每种类型记录身份图标、色条、旋转节奏、停转强调、结算高光等级和触发特效引用 | 类型可控、资源可复用、程序可配置 | 需要先定义配置字段和默认继承关系 | 倾向 |
| 几何家族批量出帧 | 按标准方骰、刻印方骰、龟壳轮廓、大树轮廓、金币轮廓等分组，用同一相机和角度模板批量渲染序列帧 | 外形差异真实，节奏仍统一，资源生产可控 | 需要建立稳定出帧模板和命名规范 | 倾向 |
| 分层继承表现配置 | 全局默认、几何家族、路线家族、骰子类型和特殊覆盖逐层决定表现 | 既能统一风格，又能控制特例成本 | 首版需要明确哪些层可覆盖哪些字段 | 倾向 |
| 每颗骰子手工逐帧绘制 | 每颗骰子单独画完整旋转和停转条 | 理论表现最自由 | 成本极高，风格和帧间一致性风险最大 | 不建议 |

## 倾向方向

- F012 的核心不是“基础骰动画复制给所有骰子”，而是“基础骰定义统一表现语法，各类型骰子继承统一阶段并填充差异插槽”。
- F012 可以用三句话描述：资源复用决定“用哪些共享素材和家族素材”，环节细分决定“每个阶段该看什么、停多久、强调什么”，表现配置继承决定“每颗骰子在哪些字段上覆盖默认表现”。
- 表现配置继承建议按“全局默认 -> 几何家族 -> 路线家族 -> 骰子类型 -> 特殊覆盖”组织。多数骰子只需要类型身份覆盖；少数骰子才覆盖几何家族或特殊序列帧。
- 基础骰负责定义投前身份态、起转、循环旋转、停转显点、结果保持、结算点名、点数可读性和短反馈密度。
- 类型差异优先放在小角标、类型色条、机制图标、短标签、触发高光和结算高光上，不在普通投骰阶段给每种骰子大幅专属运动。
- 资源策略倾向为一套共享基础旋转序列帧、一套共享停转序列帧、每类骰子的静态类型图标，以及少量路线家族触发特效模块。
- 动画参数应配置化，避免在代码中为每个类型硬写帧率、相位、停顿、特效强度和触发文案。
- 每种骰子应有自己的表现配置档，但表现配置档不等于完整独立序列帧。配置档应优先引用共享基础帧、路线家族特效和类型身份层。
- 完整独立序列帧只适合少数情况：骰子外形明显不同、机制触发必须通过身体动作理解、关键稀有骰或后续宣传展示需要。
- 外形差异建议分为四级处理：仅表面图案不同则共享序列帧并叠加贴图；边角、刻槽、绑带等轻度几何差异则归入同一几何家族；龟壳、大树、金币等明显轮廓差异则使用家族旋转模板批量出帧；Boss 或宣传级骰子才做完整定制序列帧。
- 如果使用序列帧资源，优先建立统一相机、光照、角度、帧数、阴影和接触点规范，再按几何家族批量生成或导出，不建议逐颗临场手画。

## 工作流示例

下面以“新增一颗点籽树类骰子”为例。它只是示例，不代表 F012 已进入正式生产包。

### 一、输入规则卡

先由策划确认最小规则卡：

| 项目 | 示例 |
|---|---|
| 骰子类型 | 点籽树 |
| 触发条件 | 命中本次指定点数 |
| 收益类型 | 本骰真实成长 |
| 结算接口 | 真实二次确认结算后入档 |
| 玩家可读短标签 | `籽 X`、`籽中成长` |
| 强调等级 | 未触发低，触发中，成长入档中高 |

这一阶段只回答“真实规则是什么”。不能在这里新增动画侧规则，也不能让表现层自己判断成长。

### 二、表现分层归类

把这颗骰子放入继承链：

| 层级 | 示例归类 | 作用 |
|---|---|---|
| 全局默认 | 固定槽位投骰、左到右停转、左到右结算点名 | 保证所有骰子节奏一致 |
| 几何家族 | 大树系方骰或刻纹方骰 | 决定是否使用特殊外形旋转模板 |
| 路线家族 | 长期成长接口 | 复用成长触发高光和入档反馈 |
| 骰子类型 | 点籽树 | 覆盖类型图标、色条、短标签和触发文本 |
| 特殊覆盖 | 无 | 不做独立完整序列帧 |

如果归类后发现只是图案和状态不同，就不新增旋转序列帧；如果轮廓变成明显树冠或根系形体，才进入大树几何家族出帧。

### 三、资源复用判定

按这个顺序判断资源：

1. 共享资源是否够用：基础循环旋转、停转预览、空白结果骰面、点数绘制。
2. 家族资源是否够用：成长光效、叶片或年轮短高光、大树系色条。
3. 类型资源是否必须新增：点籽树静态身份图标、结果阶段小图标。
4. 特例资源是否真的必要：完整独立旋转条通常不做，除非外形轮廓无法用现有家族表现。

该例的首版结论可以是：复用共享旋转和停转；复用成长家族高光；新增或复用点籽树类型图标；不做完整独立序列帧。

### 四、阶段表现填表

| 阶段 | 玩家看到什么 | 资源来源 | 配置覆盖 |
|---|---|---|---|
| 投前身份态 | 槽位里直接显示点籽树类型原图 | 类型图标 | 类型身份 |
| 起转 | 固定槽位内播放共享旋转 | 共享旋转条 | 大树系帧速略慢或相位不同 |
| 循环旋转 | 不显示长文本，不显示点数 | 共享旋转条 | 只保留轻微家族节奏 |
| 停转显点 | 从左到右停住，显真实点数 | 共享停转条 + 结果骰面 | 点籽树色条和小图标 |
| 结果保持 | 显点数、类型小图标、`籽 X` | 结果骰面 + 类型身份层 | 短标签 |
| 结算点名 | 本槽轻抬、高亮、分数飞字 | 共享结算点名 | 不抢倍率和总分焦点 |
| 触发高光 | 只有真实命中时显示 `籽中成长` 和成长入档反馈 | 成长家族特效 | 中等强调 |
| 重置 | 未达标回到等待投骰时恢复类型原图 | 类型图标 | 清除本次点数和短标签 |

这张表是 F012 的核心产物之一。它让美术知道要画什么，程序知道读什么配置，界面体验知道每个阶段的信息密度。

### 五、样张与验收

首轮不要直接做全量骰子。建议只做三类样张：

- 基础骰：证明统一流程、点数可读性和安静结果态成立。
- 点籽树：证明类型身份、成长短标签和触发高光成立。
- 一颗金币或龟龟代表：证明另一个路线家族也能套用同一工作流。

样张通过后再批量扩展到其它已实现骰子。否则会把错误规范复制到几十颗骰子上。

### 六、配置接入

程序侧应按配置读表现，而不是在类型判断里硬写动画：

| 配置字段 | 示例值 | 用途 |
|---|---|---|
| `die_type` | `PointSeedTree` | 匹配骰子类型 |
| `geometry_family` | `tree_square` | 选择几何家族资源 |
| `route_family` | `growth` | 选择触发高光模块 |
| `spin_profile` | `default_slow` | 控制旋转节奏 |
| `stop_profile` | `default_left_to_right` | 控制停转节奏 |
| `result_identity` | `tree_seed` | 结果阶段身份层 |
| `trigger_vfx` | `growth_pulse` | 真实触发时播放 |
| `fallback_profile` | `default_die` | 资源缺失回退 |

这些配置只影响表现。真实触发仍来自结算事件、结构化标签或已提交状态。

### 七、运行验证

验收顺序应保持分层：

1. 静态资源检查：资源路径存在，命名规范，能回退。
2. 表现配置检查：字段齐全，不影响随机和计分。
3. 普通未触发局：点数清楚，类型不抢戏。
4. 真实触发局：只有真实成长时才出现成长高光。
5. 出千后触发局：表现读取最终有效点数，但不刷新已固定目标。
6. 未达标继续投骰：重置为投前身份态，不残留上次标签。
7. 关闭动画或资源缺失：能回到可玩的旧展示路径。

通过后再把同一流程推广到大树系其它骰子、金币系骰子、龟龟系骰子和奇偶系骰子。

## 首个推进切片建议

### 范围

建议首批生产包只做 `F012-01 待机到显点流程`。

纳入范围：

- 待机身份态。
- 起摇预备。
- 槽位内旋转循环。
- 停转显点。
- 结果点面保持。
- 未达标或下一次投骰前的重置。

不纳入范围：

- 结算触发高光。
- 成长入档表现。
- 金币飞字和钱包入账。
- 临时小骰生成和折叠。
- 倍率盖章和通关收束。
- Boss 或宣传级特殊动画。

这些后续可以作为 `F012-02 结算触发表现` 或其它子切片处理。

### 阶段拆分

| 阶段 | 玩家核心感受 | 允许显示的信息 | 资源策略 | 不建议做 |
|---|---|---|---|---|
| 待机身份态 | 我知道每个槽位是什么骰子 | 类型原图、材质层、少量状态角标 | 直接复用 `DiceTypes` 类型图标和材质表现 | 显示长规则说明或复杂动效 |
| 起摇预备 | 我要开始投骰了 | 槽位轻抬、阴影压缩、短促反馈 | 全部骰子共享一套槽位动作 | 每种骰子单独做大动作 |
| 旋转循环 | 结果还没揭晓 | 不显示清晰点数，只保留体积、阴影和节奏 | 共享旋转条为主，几何家族少量覆盖 | 在旋转中塞机制文本 |
| 停转显点 | 结果正在被揭开 | 从左到右逐颗停住并显真实点数 | 共享停转条 + 结果骰面底图 + 程序点数 | 按点数重排或提前展示结果 |
| 结果点面保持 | 我能读清点数和骰子身份 | 真实点数、类型小图标、色条、短状态标签 | 结果骰面统一，身份层配置化 | 用数字徽章替代点数 |
| 重置 | 进入下一次投骰准备 | 清掉点数和本次状态，回到类型身份 | 回到待机身份态资源 | 残留上一手标签或触发光效 |

### 不同骰子的处理方式

不同骰子不直接对应“不同整套动画”，而是先归类到四层：

| 层级 | 解决的问题 | 示例 |
|---|---|---|
| 全局共享层 | 所有骰子共同的阶段、槽位、阴影、帧数、停转节奏 | 基础旋转、停转显点、结果骰面底图 |
| 几何家族层 | 外形轮廓差异 | 标准方骰、刻印方骰、龟壳轮廓、大树轮廓、金币轮廓 |
| 路线家族层 | 机制气质差异 | 成长、金币、龟龟、奇偶、爆发 |
| 单骰身份层 | 具体类型识别 | 点籽树图标、悬赏骰图标、慢龟图标、短标签 |

首批建议不要一次性把所有几何家族做完。可以先做：

- 基础骰：验证统一阶段和点数可读性。
- 一颗表面差异骰，例如奇偶或金币代表：验证只换身份层是否成立。
- 一颗外形差异骰，例如点籽树或龟龟代表：验证几何家族模板是否必要。

### 资源包拆分

| 资源包 | 内容 | 复用目标 |
|---|---|---|
| 共享动作包 | 起摇、旋转循环、停转显点、槽位轻抬、阴影变化 | 所有骰子 |
| 共享结果包 | 空白结果骰面、点数绘制规则、类型色条位置、小图标位置 | 所有骰子 |
| 几何家族包 | 标准方骰、龟壳、大树、金币等轮廓的旋转模板 | 同轮廓骰子 |
| 路线家族包 | 成长轻光、金币闪光、龟壳余波、奇偶印章、爆发震动 | 同路线骰子 |
| 单骰身份包 | 类型图标、短标签、状态角标 | 单个骰子类型 |

资源新增顺序应从上到下。越靠上越值得打磨，越靠下越要克制。

### 配置字段草案

首版可以先有这些字段：

| 字段 | 作用 |
|---|---|
| `die_type` | 绑定骰子类型 |
| `geometry_family` | 选择外形与旋转模板 |
| `route_family` | 选择路线气质和后续触发高光 |
| `idle_identity` | 待机身份图 |
| `spin_profile` | 旋转速度、相位、轻重感 |
| `stop_profile` | 停转时长、停顿强度、显点节奏 |
| `result_face_profile` | 结果骰面、点数样式、身份层位置 |
| `status_label_profile` | 结果保持阶段允许显示的短标签 |
| `fallback_profile` | 资源缺失时回退到基础骰表现 |

首版配置只影响表现，不参与随机、出千、得分、钱包、成长或存档。

### 推荐推进顺序

1. 先确认 `F012-01` 范围，只做待机到显点。
2. 做一张阶段规范表，明确每个阶段允许显示什么、禁止显示什么。
3. 做一张骰子家族表，把当前骰子先粗分为标准、龟龟、大树、金币、奇偶、爆发。
4. 选三颗代表骰做样张：基础骰、一颗只换身份层的骰子、一颗需要几何家族的骰子。
5. 样张通过后再生成共享动作包和共享结果包。
6. 程序只接表现配置和资源引用，保持 F009 / F006 的只读边界。
7. 用 Unity 运行验证普通未触发、出千后显点、未达标重置、资源缺失回退。
8. 最后再批量扩展到所有已实现骰子。

### 关键验收

- 玩家在待机时能识别骰子类型。
- 玩家在旋转时感到所有骰子属于同一套游戏风格。
- 玩家在停转后能第一眼读清真实点数。
- 非基础骰能保留身份，但不遮挡点数。
- 特殊骰不因为普通投骰阶段过度表演而抢结算触发的戏。
- 新表现关闭或资源缺失时仍可回到当前可玩的展示路径。
- 视觉层不改变真实结果、槽位顺序、出千、计分、钱包、成长、临时小骰或存档。

## 已拒绝 / 已后置想法

- 不建议首轮就为每个骰子类型制作完整独立序列帧动画；这会在当前类型数量下迅速变成资源和验收黑洞。
- 不建议让类型差异主要发生在投骰旋转阶段；旋转阶段应以统一节奏和结果悬念为主，类型差异应更多在结果稳定和结算触发时兑现。
- 不建议让动画层拥有规则判断；所有触发和收益仍应来自真实结算结果。
- 不建议把“每种骰子要有自己的表现”理解为“每种骰子都要从起转到结算画一整套独立序列帧”。
- 不建议让所有骰子外形差异都大到需要独立旋转条；骰子类型图标仍应把机制符号融入骰子本体，但大多数类型应共享可复用的几何家族。

## 待回答问题

- 基础骰标准序列帧需要覆盖哪些阶段：是否只覆盖起转循环和停转，还是也覆盖投前待机、显点吸附和结算轻抬。
- 类型差异插槽分几层：类型图标、色条、材质层、短标签、触发特效、路线家族特效是否都纳入 F012 首版。
- 每种骰子的表现配置档字段需要确认：是否包括共享帧引用、路线家族、旋转节奏、停转强调、结果身份层、结算高光等级、触发特效和回退资源。
- 表现配置继承层级需要确认：全局默认、几何家族、路线家族、骰子类型和特殊覆盖分别允许改哪些字段。
- 几何家族如何划分需要确认：首版至少应区分标准方骰、龟龟系、大树系、金币系、奇偶印章系和爆发系，还是先只做标准方骰加 1 个代表性特殊家族。
- 序列帧生产方式需要确认：使用 3D / 2.5D 统一模板批量渲染，还是使用 2D 分层贴图和运行时变形合成。
- F012 是否要生成新的动画配置表，还是扩展现有 `roll_feedback_config.csv`。
- 首版是否只制定规范和生成基础骰 / 一两个类型样张，还是直接批量规划全部已实现类型。
- F012 与 F009 的边界如何划分：F009 是当前固定槽位序列帧首版实现，F012 更像后续统一资产规范和类型扩展规范。

## 设计理由

当前骰子类型已经覆盖基础、龟龟、金币、奇偶、大树、赌徒、双倍、材质和词缀等多个方向。如果每次新增骰子都重新设计整套动画，玩家会看到很多互相不兼容的动效；如果所有骰子只换皮，又会看不出类型机制。统一动画语法可以把投骰和结算的基础节奏固定下来，让玩家先相信流程，再通过类型差异理解构筑。

这个方向也符合当前项目的实现边界。F009 已经把骰子留在固定槽位内，用序列帧表现旋转和停转；F006 / F007 已经要求结算表现只读真实 `ScoreDice()` 后的提交态。F012 应把这些已经成立的边界提升成规范，而不是重新打开物理投骰、随机结果或结算公式。

## 职能备注

### 策划

策划需要定义哪些反馈属于基础流程，哪些反馈属于类型触发。普通骰和普通结果应保持安静；只有真实触发金币、成长、临时小骰、出千收益、倍率成立或通关时才提高强调等级。

### 美术

美术重点是先统一骰子的体积、边缘、受光、阴影、点数可读性和类型图标嵌入方式。类型差异应融入骰子本体或结果标签，避免把骰子变成“骰子 + 场景道具说明”的构图。

### 界面体验

界面体验需要保证不同类型骰子在同一槽位体系内可读。运动阶段不应显示过多文本或完整机制说明；稳定结果和结算阶段用短标签表达触发原因，详细解释放到详情或日志。

### 程序

程序应把动画做成阶段机和配置驱动。表现层消费真实投骰阶段、槽位顺序、最终点数、类型身份、材质显示和结算展示事件；不得调用或替代 `ScoreDice()`，不得重新生成随机结果，不得重复写钱包或成长。

## 风险登记

| 风险 | 重要原因 | 可能缓解方式 | 状态 |
|---|---|---|---|
| 统一后类型同质化 | 玩家可能看不出不同骰子的机制身份 | 保留类型色条、图标、短标签和触发高光插槽 | 已识别 |
| 差异过多导致风格重新分裂 | 每种骰子都加独立大动作会破坏统一节奏 | 将专属差异限制在结果和结算触发阶段 | 已识别 |
| 资源量失控 | 已实现骰子数量多，逐颗全动画成本过高 | 使用共享基础序列帧和路线家族特效模块 | 已识别 |
| 表现变成纯表演 | 动画如果不绑定真实触发，会掩盖决策密度 | 只读真实结算事件，强调等级绑定实际收益 | 已识别 |
| 与 F009 边界混乱 | F009 已经有首版过程动画实现 | F012 定义后续统一规范和资产扩展，不重复改写 F009 目标 | 已识别 |

## 候选验收

- 玩家能看出所有骰子使用同一套投骰阶段：投前身份、起转、循环、停转、显点、结果保持和结算点名。
- 基础骰结果阶段点数最安静、最清晰，不出现多余类型标签抢焦点。
- 非基础骰在稳定结果阶段保留类型身份，但不遮挡点数。
- 龟龟、金币、大树、奇偶和爆发类骰子在触发时有各自可识别的短反馈。
- 普通投骰阶段不会因为类型差异太多而变得混乱。
- 动画关闭或资源缺失时，仍可回退到当前可玩的 2D / 程序绘制路径。
- 任意动画都不能改变真实骰面、结算顺序、出千结果、钱包、成长、临时小骰或存档。
