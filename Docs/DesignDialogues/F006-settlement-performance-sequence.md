## 当前摘要

本次讨论聚焦后续表现优化，重点是让结算演出更爽，同时把越来越复杂的计分、金币、成长、临时小骰和贿赂流程整理成明确顺序。当前已确认：左到右结算不应绑定投骰后按点数重排的展示顺序，而应绑定玩家投骰前可调整的固定骰袋槽位；结果锁定后不再重排，直接按槽位顺序展示和结算。用户确认先按“整机负载过载 75% + 克制 CRT 屏幕击穿 25%”制作 V6B-R6 五档样片，但随后明确退回：最高档仍不够强烈，没有形成压抑后的爆发。V7 以“锁死—吸光—真空—击穿—余震”首次获得用户对“瞬间击中”和“爆发前寂静”的正向反馈；V7-R1 随后把结算主体校准到游戏 `1.0x`，并由用户确认“这个版本没问题”，正式冻结为最高暴击母版。V7-R2 按同一 7.9 秒评审标尺派生未达到反向泄压、过关单拍闭环、超过双拍继电、远超过可控整机过载；用户以“可以，继续推进吧”允许五档方向进入运行时。V7-R3 现已沿既有 `TargetSettle` 队列接入五档独立终局、最高档真空击穿、程序音效与降闪缩放，并通过 Unity 双分辨率自动捕获；当前只等待用户在真实整手和扬声器环境中回归，不宣告最终体验验收完成。

## 原始用户意图

用户希望后续优化表现，尤其是结算时的结算演出必须有爽感；随着计算内容增多，需要明确计算顺序和流程。用户进一步指出，如果当前结果会按点数重排，而左到右结算又读取展示顺序，就会影响真实结算结果。随后用户确认新逻辑：摇骰子之前可以调整骰子的槽位，摇动完毕出现结果后骰子不再重排，直接计算结果。

## 问题框定

- 玩家问题：玩家需要看懂分数、金币、成长和补分为什么按这个顺序发生，并能把高分归因到自己的构筑和槽位安排。
- 生产问题：结算内容正在变多，如果展示顺序、预览顺序和真实计算顺序不一致，后续实现和调试会持续出错。
- 当前系统张力：点数排序有助于看牌型，但一旦与流式钱包、贿赂和词缀读取绑定，会把随机点数额外变成随机结算顺序。
- 目标体验：玩家投骰前能调整槽位，结果锁定后看到一条稳定的账本过账队列，逐颗触发并最终盖章结算。

## 讨论时间线

### 第 1 轮：结算演出与流程拆分

- 用户方向：后续表现优化最重要的是结算演出要爽，同时需要明确计算顺序和流程。
- Codex 回应：建议把结算定义成“账本连锁兑现”，后台先生成不可变的结算账本，动画只播放账本；表现分为逐骰小爆点、路线中爆点和总账大爆点。
- 形成的洞察：爽感不只是最终数字跳大，而是要让玩家看到每颗骰子、每个路线效果和倍率如何一步步把结果推高。
- 决策状态：倾向

### 第 2 轮：识别点数重排风险

- 用户方向：指出如果每次结果会重排，左到右结算会对结算结果产生影响。
- Codex 回应：确认该风险成立；如果按点数重排后的展示顺序结算，会影响猪猪、鎏印、金币后缀、贿赂和后续读取钱包的效果。
- 形成的洞察：按点数重排如果参与计算，会把“随机点数”额外变成“随机顺序”，不适合作为流式钱包和贿赂的默认规则。
- 决策状态：已确认风险

### 第 3 轮：固定槽位顺序

- 用户方向：确认修改逻辑：摇骰子之前可以调整骰子的槽位；摇动完毕出现结果后，骰子不再重排，直接计算结果。
- Codex 回应：将左到右结算改为当前骰袋槽位顺序，并同步代码和文档；结果锁定、孤证重摇和出千重摇都保持槽位不变。
- 形成的洞察：固定槽位把顺序从随机副产物变成玩家可规划的前置决策，适合承载流式钱包、贿赂和后续词缀触发。
- 决策状态：已确认

### 第 4 轮：首版结算表现方案

- 用户方向：要求给出一版结算演出的表现优化方案。
- Codex 回应：提出“账本过账式三段演出”：结果锁定后保持六个固定槽位，逐槽触发单骰分、金币、材质和词缀；龟龟、大树、金币和贿赂作为路线高光慢半拍；最后用牌型倍率盖章、目标条冲刺和通关印章完成大爆点。
- 形成的洞察：结算爽感需要有层级。普通触发必须快，路线关键触发必须被看见，最终数字必须一次性落地；否则玩家要么看不懂来源，要么嫌动画拖慢。
- 决策状态：倾向

### 第 5 轮：基于当前实现的演出强化

- 用户方向：当前结算表现仍然太素，希望结合现有项目再给一版优化方案。
- Codex 回应：确认当前代码已具备左侧双数字、逐颗高亮、基础分和倍率跳涨，但缺少“开场定格、路线归因、倍率重击、通关收束”的舞台层次；建议不先改计分规则，而是在现有 OnGUI 原型上补一层可配置演出节奏和紧凑账本反馈。
- 形成的洞察：当前问题不是数字没有变化，而是玩家看不到每个爆点的价值排序。下一版应让普通骰快速入账，让路线骰、牌型倍率和达标瞬间成为明确的注意力峰值。
- 决策状态：倾向

### 第 6 轮：确认方向并询问推进流程

- 用户方向：认可“五拍结算舞台”的整体思路，并询问后续推进流程。
- Codex 回应：建议按“确认范围、生成生产包、生产包审查、分职能实现、截图与游玩验收、反馈迭代”的顺序推进；当前不直接实现，先把 F006 从设计倾向整理成可执行生产包。
- 形成的洞察：该需求已经从泛泛表现优化进入可生产阶段，但仍需要先明确首版范围，防止把音效、专属路线特效和跳过系统一次性塞进同一实现包。
- 决策状态：可推进生产包

### 第 7 轮：积分塔之外的分档视觉冲击

- 用户方向：当前结算阶段仅由积分塔承担震撼效果仍然不够，希望继续沿用未达、过关、超过、远超过、暴击五档，并参考《炉石传说》《小丑牌》等成功卡牌游戏增强视觉冲击。
- Codex 回应：把问题诊断为“强度分级已经存在，但舞台响应面过窄”。提出四个候选方向：整机负载过载、构筑家族签名、竞技场环境共振、CRT 屏幕击穿；推荐首版以整机负载为骨架、克制叠加屏幕击穿，高档位再预留家族签名。
- 形成的洞察：《小丑牌》值得借鉴的是逐触发可读与多反馈通道叠加，《炉石传说》值得借鉴的是重大事件必须拥有与事件身份绑定的独特表现；不应照搬赌场包装、全屏常驻特效或每颗骰子的独立大动画。
- 决策状态：推荐组合已获准制作动态样片

### 第 8 轮：A+D 五档动态样片

- 用户方向：认可推荐方案，要求开始制作动态样片。
- Codex 回应：沿用 R5 的固定机位、六骰固定槽位、7 次贡献事件和 4.8 秒时间轴，制作 V6B-R6 五档动态样片；积分塔继续负责入账，新增顶部 HUD、中央骰盘框、底部操作区、左右机柜和右侧终端的整机回路反馈，后三档再逐级加入 CRT 同步撕裂。暴击档使用“负载顶点 → 全机欠压 → 二段复电 → 余辉冷却”的四段呼吸。已输出五条 1280×720/60fps 带声单档视频、同步对比片、顺序带声对比片和四张审查板，并完成逐条解码、帧数和可读性检查。
- 形成的洞察：相邻档位不能只靠亮度区分，必须改变响应范围、持续时间和节奏结构。尤其暴击通过短暂欠压制造反差，比持续堆亮更有重量；CRT 只应承担高档峰值切口，不能成为常驻滤镜。
- 决策状态：样片已生成，等待用户接受 / 退回 / 要求变体

### 第 9 轮：最高档爽感不足，要求先打穿上限

- 用户方向：V6B-R6 效果仍不够强烈，达不到爽感要求，尤其缺少压抑之后的爆发；后续直接制作最高档，如果最高档不能成立，就不继续向下派生其它档位。当前先给修改方案，不立即生成新样片。
- Codex 诊断：问题不在于特效层数量，而在于宏观动态范围不足。R6 全机欠压约 0.15 秒，整屏有效压暗仅约两成；爆发仍主要由亮线、继电灯和 CRT 横带叠加，画面轮廓、空间尺度、主次关系和声音能量没有发生足够大的断层，因此更像更亮的故障闪烁，而不是机器先承压、再失控释放。
- 修改方向：V7 最高档改成“锁死—吸光—真空—击穿—余震”五段。延长蓄压与近静默窗口，真正熄灭非核心区域，把能量收束到单一高温核心；爆发时用局部曝光击穿、全机大幅后坐、单一主冲击波、两段复电和低频 / 机柜 / CRT 三层声响形成同步断层；结果文字延后到爆发后出现。首条 V7 有意把上限做得略高，先确认爽感上限，再回调并派生低档。
- 决策状态：R6 最高档已拒绝；V7 修改方案待用户确认

### 第 10 轮：授权并完成 V7 最高档动态样片

- 用户方向：确认按修改方案开始制作。
- Codex 执行：只生成一条 5.2 秒、1280×720、60 fps 的 V7 最高档带声样片。最后一笔贡献后依次进入锁死、吸光、近黑 / 近静默真空、局部击穿、整机后坐、二段复电和余震；结果文字延后到爆发之后。同步输出 R6 / V7 画面对照、R6 → V7 顺序带声对照、五阶段审查板和三张关键帧，并完成逐条解码、帧数、临时文件和声压反差检查。
- 形成的洞察：V7 已在可测量层面建立真空到爆发约 17.7 dB 的平均声压断层，视觉也从近黑单核心切换到全屏贯穿；但“是否真正达到爽感基线”必须由用户观看带声动态样片确认，不能由静帧或技术自检代替。
- 决策状态：V7 最高档样片已生成，等待用户接受 / 退回 / 要求变体

### 第 11 轮：方向命中，要求按游戏正常速复看

- 用户方向：V7 有一瞬间真正击中玩家，尤其认可爆发前寂静带来的后续爆炸反差；但压抑时长感觉偏快，要求确认是否由倍速制作导致，并查看游戏正常速版本。
- Codex 诊断：V7 的 5.2 秒 MP4 为 60 fps 正常编码，没有后期倍速；但 7 次贡献平均约 `0.30 秒/击`，而当前运行时 `1.0x` 配置为普通贡献 `0.55 秒/击`，因此导演样片主体确实接近游戏 `1.8x`。
- 执行：生成 V7-R1 正常速版本。投骰与停轮保持原速；从结算开始到爆发释放点按 `2×` 拉长；爆炸、整机后坐和余震保持原速。总长为 7.9 秒，爆发点约 6.86 秒，并附 V7 / V7-R1 同屏节奏对照、顺序带声对照和五阶段审查板。
- 决策状态：V7 的压抑—爆发结构获得方向认可；V7-R1 正常速节奏等待用户回归

### 第 12 轮：正常速最高档通过，开始向下派生

- 用户确认：“这个版本没问题，后续结合这个示例作为最高暴击的分数结算展示来做其他几个档位的特效展示。”
- 决策：V7-R1 的游戏 `1.0x` 时间轴、压抑前静默、单次击穿和余震顺序冻结为最高暴击母版；后续较低档不得修改或稀释该母版，也不得提前使用其近黑真空与全屏击穿。
- 执行：生成 V7-R2 未达到 / 过关 / 超过 / 远超过四条带声源样片，并以 `反向泄压 → 单拍闭环 → 双拍继电 → 可控整机过载 → 真空击穿` 建立五档语义。五档共用 `7.9 秒 / 60 fps / 1.0x` 评审标尺，同时输出同步视觉对比与顺序带声对比。
- 决策状态：V7-R1 已通过并冻结；V7-R2 四档已生成，等待用户接受 / 退回 / 要求变体

### 第 13 轮：五档源方向通过，接入 V7-R3 运行时

- 用户确认：“可以，继续推进吧”。
- 边界解释：该反馈直接承接 V7-R2 五档样片交付；Codex 明确说明将其视为“五档源样片通过并进入 Unity 运行时”，且不改真实计分、奖励或存档语义。
- 执行：沿既有 `TargetSettle` 展示事件接入五档终局。前四档分别为反向泄压、单拍闭环、双拍继电和可控整机过载；最高档独占 `2.20s` 的锁死、吸光、近黑真空、单次击穿和余震。同步接入程序生成的蓄压 / 击穿 / 余震声音、统一演出倍率与降闪缩放。
- 验证：Unity 2019.4.33f1 编译为 `0 error`，自动 Play Mode 校验五档分类与时长，并在 `1920×1080 / 1280×720` 捕获五档签名和最高档三关键相位；另完成 `0.606×` 目检。
- 决策状态：V7-R2 源方向通过；V7-R3 实现与自动验证通过，等待用户实机体验回归

## 探索过的方案

| 方案 | 核心思路 | 优点 | 风险 | 状态 |
|---|---|---|---|---|
| 按点数重排后左到右结算 | 结果锁定后按有效点数从小到大展示并结算 | 看牌型直观 | 顺序随机化，影响金币流和贿赂，玩家难以预期 | 已拒绝 |
| 固定骰袋槽位结算 | 投骰前可调整槽位，结果锁定后不重排，按槽位从左到右结算 | 顺序可规划，预览和真实结算一致，适合表现账本队列 | 需要额外提供点数摘要或牌型提示帮助看牌型 | 已确认 |
| 双视图展示 | 主结算保留槽位，另加点数摘要辅助阅读 | 兼顾顺序和牌型阅读 | 增加界面密度 | 倾向 |
| 账本过账式三段演出 | 逐槽过账、路线高光、倍率盖章与目标冲刺 | 结算来源清楚，爆点有层级，适合明亮账本桌游风 | 需要控制时长，并提供加速或跳过边界 | 倾向 |
| 五拍结算舞台 | 结果定格、逐槽点名、路线小爆点、倍率盖章、目标冲刺与收束 | 可以复用现有双数字和逐颗入账，同时补足情绪节奏 | 首版需要克制特效数量，避免 OnGUI 表现过载 | 倾向 |
| 整机负载过载 | 积分塔继续入账，最终档位把响应扩散到骰台、侧框、顶栏和终端 | 通用、清楚、最适合复用当前隐藏五档 | 若每击都全屏响应会疲劳，必须只在结果释放时扩场 | 推荐候选 |
| 构筑家族签名 | 高档位按本手主要贡献家族播放独特硬件语法和声音签名 | 能庆祝玩家构筑，重复游玩辨识度最高 | 混合家族归因复杂，美术与音频成本高 | 二期候选 |
| 竞技场环境共振 | 让背景机台、灯架、玻璃反射和场外声场回应结果 | 不遮挡六骰即可显著扩大舞台 | 需要新增环境分层，容易扩大 F021 场景范围 | 候选 |
| CRT 屏幕击穿 | 用同步撕裂、磷光拖影、边缘错色、短冻结和镜头承击表达高档 | 资产成本低、峰值很强、符合旧街机语言 | 易眩晕或廉价化，必须提供减弱选项并限制到高档 | 辅助候选 |
| V6B-R6 整机回路最高档 | 短欠压、二段复电、机框继电灯与 CRT 撕裂 | 已扩展到整机并完成可运行样片 | 压暗、停顿、位移和声场反差都不足，爆发读成故障闪烁 | 已拒绝 |
| V7 最高档“锁死—吸光—真空—击穿—余震” | 先清空亮度和声音，再以单一主爆点改变整机轮廓与空间尺度 | 已证明爆发前寂静能够形成玩家爽点 | 内部时标约等于游戏 `1.8x`，压抑掠过偏快 | 方向命中、节奏待校准 |

## 已确认决策

- 投骰前的 `Ready` 阶段允许玩家调整骰子的槽位顺序。
- 结果锁定后，投掷区保持当前骰袋槽位顺序，不按有效点数重排。
- 孤证重摇和出千重摇只改变骰面结果，不改变槽位顺序。
- 左到右结算指当前骰袋槽位顺序，不是点数排序后的临时展示顺序。
- 猪猪、鎏印、金币后缀、贿赂和后续读取钱包的效果都按当前骰袋槽位从左到右执行。

## 倾向方向

- 后续结算演出应做成账本过账式队列：逐颗骰子触发，路线重点效果慢半拍，总分、倍率和通关目标最后集中落地。
- 为了补偿不再按点数重排带来的牌型阅读成本，结果区可以增加小型点数摘要或牌型提示，但该辅助视图不参与计算。
- 后台应先生成本次结算账本，动画只播放账本；跳过动画、加速播放和真实结算都读取同一份结果。
- 首版表现节奏倾向分为三段：第一段逐槽过账，第二段路线高光，第三段倍率盖章和目标条冲刺。
- 普通骰只用短促缩放、浮字和账本行入账；路线关键触发使用更明确的印章、金币飞行、成长光效或小龟链条。
- 大树成长建议在分数落定后再作为“成长入档”表现，避免玩家误以为成长已经影响本次基础点数。
- 当前 F007 实现已完成“数字会跳”的底座，下一步不应推翻布局，而应把结算期分成更清楚的五拍：开盅后短暂停格、当前槽位点名、路线效果插入、倍率章重击、目标数字滚动到最终状态。
- 逐槽点名时，骰子旁只出现一条最重要的短反馈，例如 `+6`、`盖章 6`、`金币 +1`、`成长待入档`；低价值细节继续放入日志，不在主投掷区铺开。
- 路线小爆点首版优先做四类：金币入账、龟龟临时小骰、大树成长、倍率成立。其它材质和词缀先复用普通入账反馈。
- 通关收束要和未达标收束分开：通关用目标数字越线、王室印章或账本盖章；未达标只保留安静的剩余目标提示，避免失败前演得像胜利。
- 后续推进流程倾向为：先确认首版范围，再由 F006 生成生产包，随后做生产包审查，最后拆给界面体验、美术和程序执行。
- 当前新增方向推荐采用三层结构：现有来源骰因果层保持不变；结算过程中新增连续整机负载包络；最终按隐藏五档释放不同范围的机台响应。积分塔从唯一高潮载体改为主要入账锚点。
- 首版推荐组合为“整机负载过载 + 克制 CRT 屏幕击穿”：未达只做卸压，过关只扩到骰台，超过扩到终端与侧框，远超过加入机柜承击和同步扰动，暴击先短欠压再进行两段复亮与局部屏幕击穿。
- 构筑家族签名建议只在远超过与暴击出现，且先按四个主体家族做代表性样张；中立、双阵营和混合构筑先使用通用整机反馈，避免归因错误。
- 五档仍是内部表现参数，游戏内不显示档位名、百分比或阈值，也不增加金币、倍率、奖励或存档字段。
- 用户已确认先以“整机负载过载 + 克制 CRT 屏幕击穿”制作五档动态样片；该确认仅覆盖源文件方向探索，尚不等于批准运行时实现。
- 用户已确认停止继续打磨其余四档，下一轮只验证最高档；最高档必须先建立明确的压抑后爆发，成立后才允许向下派生档位。
- 用户已确认按 V7 修改方案制作最高档样片，允许本轮把总长放宽到约 5.2 秒，并使用近黑 / 近静默真空段验证压抑后的爆发；该确认仍仅覆盖源样片，不等于批准运行时实现。
- 用户已确认 V7 的爆发前寂静方向有效，并要求后续节奏评审以游戏 `1.0x` 为基准；导演样片不得再用接近 `1.8x` 的内部时标冒充正常速。
- 用户已确认 V7-R1 正常速版本没有问题，并指定它作为最高暴击分数结算展示母版；近黑、近静默、全屏击穿和强后坐不向较低档下放。
- 用户已确认 V7-R2 五档源方向可以继续推进，允许拆入 Unity 运行时；授权不包含真实计分、奖励、随机、玩法数据或存档变更。
- V7-R3 运行时的统一 7.9 秒不再作为每档硬时长；逐骰继续跟随游戏正常速，五档最终收束分别使用 `1.70s / 1.55s / 1.65s / 1.85s / 2.20s`。
- 其余四档按“未达到反向泄压 / 过关单拍闭环 / 超过双拍继电 / 远超过可控整机过载”制作源样片；确认只覆盖源样片生产，不等于批准 Unity 接入。

## 已拒绝 / 已后置想法

- 拒绝把按点数排序后的展示顺序作为真实结算队列，因为它会让同一骰袋在不同随机点数下产生额外顺序差异。
- 后置完整结算演出生产包；当前先完成顺序规则和最小可用槽位调整，后续再扩展动画、音效、统计和高光表现。
- 拒绝 V6B-R6 当前最高档作为后续五档母版：其压抑与爆发反差不足，不继续在该强度上做加法。

## 设计理由

`骰子王` 的核心幻想是把随机骰面转化成可控计分系统。固定槽位顺序让玩家在投骰前做出可理解的顺序安排，后续金币流、贿赂和词缀读取都能被解释为“账本从左到右过账”。这比按点数重排更适合长线扩展，因为后续计算内容越多，越需要一个玩家可预期、程序可复现、动画可复播的顺序来源。

## 职能备注

### 策划

左到右顺序成为玩家可规划规则，不再是显示排序副产物。后续新增任何出手内金币、消费、转分或成长效果，都应说明是否读取当前槽位顺序。

### 美术

结算演出可使用槽位编号、账本行、高亮框、金币入账、印章落点和目标条冲刺表达顺序。不要让点数摘要喧宾夺主。

### 界面体验

主骰展示保持六个槽位稳定。结果锁定后不重排；需要用牌型提示或点数摘要帮助玩家识别三同、顺子、全奇和全偶。首版槽位调整可以采用点击两颗骰子交换。

结算期建议中心保持六个槽位不动，右侧或下方出现紧凑账本条目。账本条目不应展开成长篇日志，只显示“槽位、触发名、分数或金币变化”。点数摘要放在牌型提示附近，作为辅助阅读，不参与计算。

下一版界面重点是让“当前槽位”成为唯一主焦点。高亮框、骰子轻抬、浮字和左侧计分器脉冲应按同一个节拍发生；不要同时让所有骰子、日志和目标数字一起闪。

### 程序

当前实现方向是直接使用 `dice` 列表顺序作为槽位顺序、展示顺序和结算顺序，并在安全存档节点序列化为骰袋顺序。不要在开盅、出千或结算前按有效点数排序。后续如果新增结算账本对象，应从同一槽位顺序生成。

建议后续实现先生成只读结算账本，账本条目至少包含槽位、骰子引用、触发类型、分数变化、金币变化、倍率变化、成长变化和是否高光。动画只消费账本，不在播放中重新计算。

基于当前代码，可以先沿用 `RunScoreCounterStep` 和 `scoringDice`，补一个展示层事件列表。首版事件不需要改变 `ScoreDice()` 结果，只把已冻结的基础分、倍率、贿赂补分、金币和成长拆成演出节点。加速或跳过时必须直接应用同一份最终提交态。

## 风险登记

| 风险 | 重要原因 | 可能缓解方式 | 状态 |
|---|---|---|---|
| 牌型阅读变弱 | 不按点数重排后，玩家更难一眼看出顺子或同点 | 增加点数摘要条和牌型提示，但不改变结算队列 | 已识别 |
| 槽位顺序过强 | 金币流和贿赂依赖顺序，玩家可能觉得必须频繁调槽 | 让槽位调整操作很轻，并用角标明确顺序 | 已识别 |
| 投资预算与槽位交换边界 | 当前投资在小关开始锁定，之后换槽位不应重分预算，也不应把投资扣款误写入小关开始存档 | 投资预算绑定骰子实例，槽位只改变结算顺序；槽位交换不触发完整运行存档 | 已识别 |
| 结算动画过长 | 龟龟临时小骰、词缀和倍率都可能增加播放时长 | 普通触发快速播放，路线高光慢半拍，提供后续加速方案 | 已识别 |
| 表现遮挡信息 | 浮字、印章和金币飞行过多会盖住骰面和目标条 | 同一时刻只突出一个主焦点，低价值触发合并为账本小行 | 已识别 |
| 爽感和真实性脱节 | 如果倍率或通关演出提前播放，玩家可能误以为未结算内容已经入账 | 后台先冻结提交态，动画只按确定节点播放，不做临时猜测 | 已识别 |
| 全场反馈疲劳 | 骰台、终端、边框和屏幕如果每颗骰都强响应，会让普通结算也像暴击 | 逐骰阶段只积累负载，整场扩散只在最终档位释放 | 已识别 |
| 家族签名归因错误 | 混合与双阵营构筑可能无法自然选出唯一主题 | 首版后置家族签名，或只在主要正贡献来源明确时触发，否则使用通用反馈 | 已识别 |
| 屏幕击穿可访问性 | 高频闪烁、错色和镜头震动可能造成眩晕或疲劳 | 限制在高档短峰值，设置独立减弱开关，保留声音与机柜局部反馈 | V7-R3 已实现缩放，待用户回归 |
| 爆发只靠加亮 | 特效数量增加但前后亮度和空间状态接近，玩家只会感到画面更忙 | 先主动熄灭常亮层和声场，再把释放集中到一个主爆点 | R6 已发生，V7 必须修正 |
| 真空段过短 | 低于玩家形成预期的感知窗口时，只会读成卡顿或闪屏 | 最高暴击按 V7-R1 保留完整声场下潜和近静默核心，并用收束运动说明这是故意蓄力 | V7-R1 用户已通过并冻结 |
| 样片时标快于游戏 | 动态方向可能在样片中成立，进入 `1.0x` 后却产生完全不同的节奏与耐受度 | 所有后续评审片标注对应演出倍率；至少提供游戏 `1.0x` 版本 | V7-R1 已关闭；V7-R2 共用 `1.0x` |

## 候选验收

- `Ready` 阶段可以交换两颗骰子的槽位。
- 开盅后结果区保持投骰前的槽位顺序，不按点数重排。
- 出千重摇后，骰面和预览更新，但槽位顺序不变。
- 猪猪、鎏印、金币后缀和贿赂按槽位从左到右结算，较早金币可以被较晚贿赂读取，较晚金币不会回头改变较早对象。
- 在安全存档节点后继续游戏，骰袋槽位顺序保持为保存时顺序；小关中途交换不应破坏“继续游戏回到小关开始”的存档边界。
- 玩家能在结算后说出本次高分主要来自哪 1 到 3 个触发节点。
- 普通结算不应明显拖慢节奏；路线爆点和通关爆点可以短暂停顿。
- 龟龟临时小骰超过展示上限时，用折叠账本行表达数量和总分，不逐颗拖长播放。
- 玩家能明显感知五拍节奏：结果定格、逐槽入账、路线触发、倍率盖章、目标收束。
- 没有路线触发的普通手也应比当前更有节奏，但不应出现过多特效。
- 通关和未达标的收束反馈必须视觉上不同。
