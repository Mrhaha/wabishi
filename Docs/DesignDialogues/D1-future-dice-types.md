## Current Summary

本次讨论聚焦在 `骰子王` 后续可增加的骰子种类。当前方向已经收敛为：先完善已有基础流派，再做流派之间的交织。猪猪、龟龟、奇数、偶数等现有骰子目前更像各流派的入口骰，还缺少把资源转换成战力、放大路线收益、形成中后期选择的衍生骰。新增骰子应先补足各流派内部的生成、转换、放大和稳定结构，同时不破坏当前结算边界、市场容量和首章可理解性。当前已确认：奇数流核心是追同点牌型；偶数流核心是差 2 视为顺子，用更好的凑顺能力发展顺子构筑；金币流选用国库骰、贿赂骰、投资骰；龟龟流选用壳匠、巢穴、慢龟；F001-C 奇偶二版已用孤证、盖章、半步、轨道完成首版验收。F001-D 大树首版已按用户最新指令收敛为只实现园丁和灌溉：园丁采用“有就加”的爆发成长版，灌溉保留为命中辅助；果实、嫁接、幼苗否决，年轮、树冠、苗圃旧草案不作为当前实现依据。2026-06-12 新增判断：当前 build 优化的核心问题不只是单路线缺组件，而是各 build 之间过于独立，导致玩家成型后缺少互相串联的爆点和爽感；后续应设计一批交融玩法骰，用已有路线的事件、资源或牌型结果去衔接其它路线。
交融玩法的参考方案已初步分成四层：结算链条可视化、市场交融引导、桥接骰候选池、关卡规则临时抬价。首批建议优先做二段交融骰，而不是引入全新大系统；候选方向包括金币与龟龟、金币与大树、奇偶与大树、出千与成长或爆发的连接。最新更强倾向是整体重构骰子类型表达：把“某 build 的固有组件”改成“触发条件 + 泛用收益”。例如金币类骰统一产出金币但触发条件不同；龟龟类骰统一产出额外骰子但触发条件不同。这样单颗骰不会只服务唯一 build，而是能被任何需要该泛用收益或能满足该触发条件的构筑吸收。金币收益族的首批触发池已开始整理：用户已提出指定点数、最大点数、不同牌型、回合结束、利息加倍五个主方向；后续需要从更多触发条件中筛出一批低频高收益、中频中收益和稳定低收益的组合，避免金币经济膨胀成万能最优路线。用户进一步澄清：问题不是要让金币骰节奏变得像大树骰，也不是要直接绑定大树成长事件；真正目标是让金币玩法与大树玩法的交接形式更柔和。应避免“大树成长一次，金币 +1”这种硬接口，改用中间层承接，例如果实订单、树契、估值、抵押、成熟评级、季节收成或市场兑现。大树玩法提供可被金币玩法读取的资产状态或市场价值，而不是每次成长直接吐金币。龟龟类一度按“额外骰子收益族”整理，但用户进一步指出这不够爽：龟龟和大树都应是爽点来源的重头戏，龟龟必须承担连锁爆发和积分表演。后续龟龟类应从“泛用补骰”升级为“连锁引擎”：额外骰子不是零散收益，而是结算演出的主角、连锁深度的载体和分数跳涨的来源。最新再修正：龟龟不能为了追爽感变成“多拿龟龟才有用”的自闭环。龟龟更通用的正确方向是放大其它骰子的收益：每颗龟龟骰应挂到非龟骰、相邻槽位、某个已触发事件或某类收益上，把别人的成功转成临时小骰余波。龟龟仍然负责连锁表演，但连锁来源应来自其它 build 的成功，而不是龟龟自己互相堆配件。

最新补充集中在金币收益族的第一版包装：指定点数、最大点数、牌型、回合结束、利息、出千实际重摇、临时小骰数量、槽位顺序和削减其它骰子点数都可以成为金币触发。当前倾向是把它们分成三档：低频高额的风险合约，中频中额的构筑奖励，稳定低额的经济底座。2026-06-13 用户确认这九个金币骰都需要进入后续范围，不再从九个里筛掉一部分；后续应通过 `$wabish-production-pipeline` 生成“金币收益族泛用化”生产包，并在生产包内部按实现风险拆分批次、规则卡和验收项。

2026-06-15 用户进一步确认：龟龟骰子后续优化的主体思路，是打破各个类型骰子的壁垒，把骰子从“固定流派组件”转为更通用的机制提供者。该方向不只适用于龟龟，也应成为后续骰子重构的总框架：骰子提供可跨构筑读取的触发条件、收益类型或结算机制；龟龟的重点则是把其它骰子的成功事件转成临时小骰余波和连锁积分表演，而不是继续形成只服务龟龟自己的封闭套装。

同日进一步确认：龟龟首批五个候选 `背壳龟 / 同潮龟 / 桥壳龟 / 顶壳龟 / 铅壳龟` 可以继续往规则卡细化。`d1 / d2 / d3` 作为临时小骰规格说明：`dN` 表示一颗只会掷出 `1..N` 的临时小骰，`d1` 固定为 1，`d2` 随机 1 或 2，`d3` 随机 1 到 3；它不是生成数量。

同日再确认：龟龟仍需要一个通用底层机制。当前 `Turtle / SlowTurtle` 已经存在固有生成链，适合作为“龟龟余波链 / 壳链”的机制底座：实体龟龟从自身点数启动一段递减临时小骰链，新龟龟候选则用相邻非龟骰、同点、顺子、高面、出千等外部事件启动同一套链。生成物应先按“龟系临时骰”处理，用于计分、表演和数量兑现读取，但默认不触发实体骰类型效果，避免多龟互读形成自闭环。

同日继续修正：未来壳链不应沿用当前“每一节按实际掷点再除以 2”的快速衰减作为默认体验。用户希望龟龟链路更长、更有滚分表演。当前倾向是把“本节实际掷出的点数”和“下一节壳链规格”拆开：实际点数只决定得分，链条长度由壳链种子、逐级衰减规则和最大链深控制。

同日实现确认：用户同意按“最大值逐级衰减”的壳链思路修改当前 `Turtle / SlowTurtle` 生成逻辑，后续龟龟候选都依赖这套底层。当前实现口径为：`Turtle` 用 `floor(v / 2)` 得到壳链种子，`SlowTurtle` 用 `ceil(v / 2)` 得到壳链种子，之后统一按 `dN -> d(N-1) -> ... -> d1` 生成龟系临时骰；每节实际掷点只决定本节得分，不影响下一节是否继续。

2026-07-01 用户提出龟龟吸附重做草案：龟龟可以吸附到左侧骰子，把自身壳值或期望转移到宿主各个点面，售卖宿主时连同吸附龟卖价一起回收；候选包括金钱龟、小小龟、双倍龟、幸运龟、磁力龟和呼朋唤友龟。当前判断是：该方向更贴合 `V0.2` 四族分工里的“龟龟 = 吸附 / 压缩六槽”，但它会明显改变龟龟从“临时小骰余波”到“宿主改造”的身份。若继续推进，应先把吸附边界写清楚：左邻绑定时机、宿主上限、整袋吸附上限、龟龟是否仍占骰袋容量、是否允许吸附龟读取吸附龟、售卖回收价、壳值是否使用普通 1-6 期望，以及呼朋唤友这类群体成长是否永久入档。首版倾向把绑定触发放在离开市场，出手结束只做金币、小壳、磁力和族群 payoff，避免结算中自动吸附导致玩家难以预期。

同日优化草案：吸附龟龟首版建议定位为“宿主改造 / 六槽压缩”，不是替代当前壳链的全部龟龟身份。吸附使用左邻非龟宿主，龟龟变成宿主的“壳层”后离开实体骰袋；壳值使用 `max(1, floor(来源骰当前六面平均值))` 加到宿主全部面，宿主卖价只叠加真实吸附龟固定卖价，基础壳和临时壳不增加卖价。首版建议每个宿主最多 3 个龟龟特性壳，整袋最多 4 个龟龟特性壳，同名特性不叠加；基础壳只作为点面成长层，单独设上限，且不触发首领龟或呼朋唤友龟。七颗候选的职责压缩为：金钱龟提供稳定金币，幼壳龟提供宿主慢成长，双倍龟一次性强化最低面，幸运龟作为高阶唯一控制结果，磁力龟把基础壳分发给其它骰，呼朋唤友龟奖励本回合新发生的真实龟龟吸附，首领龟作为高阶核心复制其它吸附但不递归、不复制卖价。

同日用户进一步确认：金钱龟效果表述应规范为“结算时”触发；小小龟同样按“结算时”触发。用户明确追求吸附龟龟的爽感，因此首版不做吸附上限限制，也不做不叠加限制；吸附数量和同名特性叠加带来的强度风险应进入验收观察，而不是在规则层预先砍掉。同时，每个骰子吸附后，悬浮页必须显示对应吸附内容说明，让玩家能看到宿主当前有哪些龟龟壳、每个壳的效果、卖出时会回收哪些真实吸附龟卖价。该确认推翻上一条草案中“宿主最多 3 个特性壳、整袋最多 4 个、同名不叠加、基础壳不触发部分读取”的限制建议；后续生产包应以无上限、可叠加、重展示和重验收为准。

同日新增奇偶改版备案：奇偶族不适合照搬金币族的“统一产出资源”，也不应继续只围绕全奇、全偶、纯奇同点或纯偶顺子做专用件。更合适的共同收益是“点数结构校准”：读取全体实体骰已经可见的奇偶比例、同奇偶间隔、相邻差一、同点、顺子、出千后奇偶变化或已有牌型结果，提供一次牌型借位、小额固定分、重摇复核或槽位兑现。首批备选应优先使用低理解成本规则，例如平衡骰、桥格骰、同栏骰、换栏骰和验算骰；当前 F001-C 的孤证、盖章、半步、轨道仍是已验收事实，未来改版备案不直接推翻现有实现。

同日进一步修正：用户认为“某种牌型命中后给骰子加成，并形成持续性养成”的方向更适合放到大树型骰子里。奇偶类不应承担长期成长或命中养成，而应更有区分度：偏当手的相位操作、槽位接线、少数/多数票决、出千换相和临时牌型解释。上一轮偏“命中条件 -> 固定收益”的平衡骰、同栏骰需要降级或重做；奇偶后续优先探索能让玩家在摆位、出千和保留结果上产生判断的短时机制。

同日规则压缩：用户要求奇偶改版继续简化，不能让玩家读长说明。上一轮的“相位、票决、接线”等表述只保留为设计层语言，玩家侧首批候选必须过短规则卡测试：市场卡一句话，结算标签 2 到 4 字，不新增长期状态。随后用户指出 `夹缝骰` 的“三槽夹心”规则不直观，说明短不等于好懂；该候选降级。最新用户指定保留的奇偶短规则候选为 `异邻骰 / 同邻骰 / 补全骰 / 复核骰 / 翻号骰 / 守号骰`，并新增一个出千后奇偶必变的控制骰候选，暂命名为 `转号骰`。后续流程已确认：这 7 颗先产出接触图做美术和符号可读性评审，通过后再转程序实现；未通过前不进入代码或正式数据表。后续第二版接触图已通过，最终图标、程序接入和 Unity Play Mode 验证已完成。

同日新增大树专用性修改备案：大树后续也应从“只服务大树路线的成长套件”转向更通用的长期机制提供者。当前不推翻已实现的 `Tree / Gardener / Irrigation` 代码事实，但 F011 实现时旧三件套应从市场下架。后续候选读取更广的可见触发，例如高面命中、牌型成立、出千重摇、相邻骰触发和市场持有；最新裁决不再使用发芽或成长进度，而是触发后直接让本骰真实成长。首批不建议直接按成长层数换分，也不建议写成“成长一次给金币”的硬接口。

同日用户进一步明确：后续给出的骰子设计应优先参考固定触发点，而不是随意设计触发条件。当前触发点白名单确认为：指定点数、指定牌型、最高点数、回合结束、金币、出千和槽位触发。大树重设应围绕这七类触发来做，触发效果后直接真实成长，表现为触发骰自身全部面 `+1` 和现有 `Growth +1`。基于该约束，大树候选重新整理为 `点籽树 / 牌谱树 / 冠层树 / 年轮树 / 肥料树 / 修枝树 / 根系树`，分别对应七类触发；该记录最初作为生产包来源，随后已进入 F011 程序静态接入。

- 用户问题：九颗双阵营骰已经进入原型后，下一步是否继续丰富各个家族骰子，还是应转向其它工作。

### 第 62 轮：骰子分级与扩池体验

- 用户问题：随着骰子池持续增加，希望建立一套分级方案，保证市场抽取、路线成型和玩家理解仍能维持正常体验。
- 现状诊断：正式随机池已有 `44` 枚，而市场每次只展示 `3` 枚。近似均匀抽取时，某一指定骰在一批三格货架中出现的概率只有约 `6.7%`；如果构筑依赖某个唯一命名核心，扩池后会越来越像等货，而不是构筑。当前章节段总权重已经形成约“前期 `T1 49% / T2 46% / T3 5%`、中期 `33% / 47% / 20%`、后期 `24% / 48% / 28%`”的基础梯度，主要缺口不是再加一层稀有度，而是阶级含义、货架职责和构筑补位没有形成完整合同。
- Codex 建议：保留 `T0 / T1 / T2 / T3`，但把 `T1` 定义为可独立起步的入口件，把 `T2` 定义为读取一个已有条件的联动或转换件，把 `T3` 定义为改变构筑上限或运行方式的核心 / 终结件；低阶不等于弱，高阶也不应只是数值更大。每颗骰另外增加产出、转换、放大、兑现、桥接之一的市场职责，市场按玩家当前已有资源与缺失职责做轻度补位，而不是只按同家族加权。
- 三格市场草案：第一格为稳定入口位，优先 `T1` 与低依赖 `T2`；第二格为构筑补位位，建议约 `60%–70%` 从当前家族、双阵营桥接或缺失职责中抽取，其余保留全局探索；第三格为转折位，按章节抽取 `T2 / T3`，连续付费刷新未见 `T3` 时下一次第三格真正保底 `T3`。同批不重复，候选充足时相邻刷新避免原样重复。

### 第 7 轮：F001-C 奇偶候选重做

- 用户方向：F001-C 第一版奇数骰全都不过关，重复率太高，计分算法偏复杂；偶数骰同样不够明确、不够有趣。需要重新设计一版。
- 设计诊断：第一版问题在于过度使用“统计同点数量 / 连线数量，再按公式加分”。这会让多颗骰子只是在不同条件下做同一种加分，玩家记住的是公式而不是行为。
- 重做原则：奇偶二版优先使用单一可见触发和清楚结果，例如自动重摇、固定分数、顺子辅助、固定轨道奖励；避免 `数量 * N` 作为主要表达。
- 奇数新方向草案：`孤证骰` 负责“落单自动重摇一次”，`盖章骰` 负责“有同点就按固定高分计”，`陪审骰` 作为高阶候选负责“有同点时让一次小配对变成有效判定”。
- 偶数新方向草案：`半步骰` 负责“顺子判定时偶数可向低一格借位”，`轨道骰` 负责“2/4/6 齐备时触发固定轨道奖励”，`列车骰` 或同类候选负责“正常顺子成型后的简单固定收益”。
- 决策状态：倾向草案，尚未确认；需要用户先判断整体方向是否比第一版更有趣、更清楚。

### 第 8 轮：确认低理解成本方向

- 用户方向：二版思路可以接受，但重点是不要引入新的理解成本；游戏整体规划是易上手难精通，应该在玩家已经理解的投骰、同点、顺子和出千中做花样。
- 设计诊断：奇偶新骰应避免新增一套外部资源、复杂计数器或额外牌型表；理想规则是一句话可懂，深度来自骰袋组合、出千选择、市场取舍和关卡规则交互。
- 设计原则：首层表达简单，玩家看到标签就知道发生了什么；二层深度来自“什么时候买、怎么搭配、是否出千追结果、是否转路线”。

### 第 9 轮：F001-D 大树候选重做

- 用户方向：F001-D 第一版大树草案不通过。园丁骰不符合设定，应该让大树成长更快；年轮设计一般、缺少创意；树冠和苗圃太复杂，变成纯公式计算，不符合游戏调性。
- 设计诊断：第一版把大树路线做成“读取成长层数换分数”的数值兑现，玩家记住的是公式，不是树被照料、生长、嫁接、结果的过程。大树路线应强调长期资产的可见变化和事件，而不是把成长总量折算成若干分。
- 重做原则：大树补强骰优先改变成长过程或制造可见事件，例如额外成长、代替命中、结出临时果实、把成长嫁接到别的骰、幼苗变成大树。除非非常必要，不使用“全骰袋成长总和 / 最高成长层数 -> 加分”的纯公式。
- 新方向草案：`园丁骰` 改为让自然命中的大树额外成长；`灌溉骰` 可用自己的点数替大树命中目标；`果实骰` 奖励成熟大树命中后产生临时果实；`嫁接骰` 在大树成长时让自身面组也发生小幅永久成长；`幼苗骰` 或重做后的苗圃方向不再是市场光环，而是在投骰中变成新大树。

### 第 10 轮：F001-D 第二版筛选反馈

- 倾向规则：`灌溉骰` 保留为命中辅助：灌溉骰自身结果如果等于某棵未自然命中的大树命中点，可让其中 1 棵大树成长 1 次；出千可重摇灌溉骰来追目标，但不刷新大树目标。

### 第 12 轮：build 交融问题提出

- 用户方向：这个单元讨论 build 优化。当前问题是各个 build 之间太过独立，爽感很弱；后续会出一些交融玩法的骰子类型，用来衔接各个类型的骰子。
- 设计诊断：前一阶段把金币、龟龟、奇偶、大树等路线拆清楚是必要的，但现在每条路线大多只在自己的资源里闭环，玩家买到某条路线后，后续市场选择容易变成“继续买同类更强件”，缺少跨路线触发、借力和连锁兑现。
- 倾向原则：交融骰不应把所有路线混成万能加分，而应明确连接两个已有玩法面，例如“金币支付触发龟龟收益”“同点牌型帮助大树命中”“临时小骰数量转化为一次出千价值”“大树成长给某类骰面组临时抬点”。
- 体验目标：玩家需要看到一个 build 不是独立运行，而是在结算中把前一个路线事件传给另一个路线组件，形成可读的二段或三段爆点。
- 决策状态：确认问题，交融骰池待设计。

### 第 13 轮：交融优化参考方案

- 用户方向：希望获得一些参考思路和优化方案，用于解决构筑之间过于独立、爽感偏弱的问题。
- 参考结构：不要只做“甲路线加乙路线分数”，而应做“事件来源 -> 兑现对象”的链条。事件来源可以是金币入账、临时小骰生成、同点或顺子成型、大树成长、出千重摇和最高面命中；兑现对象可以是单骰分、成长机会、临时小骰质量、出千价值或市场购买倾向。
- 优化层级：第一层是结算展示，把已有触发串成账本链；第二层是市场引导，货架中出现能连接当前路线的商品；第三层是桥接骰，直接把两条路线的事件连接起来；第四层是关卡规则，临时提高某些交融组合的价值。
- 候选池方向：优先探索金币与龟龟、金币与大树、奇偶与大树、出千与成长或爆发四组连接。暂不建议首版做三路线以上的万能骰。
- 决策状态：参考方案，待用户筛选方向。

### 第 14 轮：从 build 组件改为触发与收益双轴

- 用户方向：之前玩法一直存在 build 难成型，或者龟龟骰子对其它 build 基本没用的问题。后续想整体修改，打破 build 壁垒并增加泛用性。例子是金币类型都获得金币，但触发条件不同；龟龟类都获得额外骰子，但触发条件不同。目标是让每颗骰都能提升整体构筑，而不是只作为某个 build 的固有组件。
- 设计诊断：这个方向比单独增加桥接骰更接近根因。旧结构的问题是“流派标签决定骰子用途”，导致玩家没凑齐整套前，单颗骰经常像死组件；新结构把骰子拆为“触发条件”和“泛用收益”，能让一颗骰同时被多个构筑读取。
- 倾向框架：骰子身份由双轴构成：一轴是触发条件，例如同点、顺子、高面、低面、出千、首投、末投、命中目标、金币入账；另一轴是收益类型，例如金币、临时小骰、单骰分、成长、点数控制、出千价值。构筑不再是固定套装，而是玩家围绕自己更容易触发的条件，选择想要的收益类型。
- 设计判断：该方向基本能解决 build 难成型和跨 build 死件问题，但必须保留骰子类型的可读身份。若所有骰子只剩“不同条件给泛用收益”，会有同质化风险；需要让每个收益族仍有独特节奏和限制。
- 决策状态：强倾向，待系统化拆分规则。

### 第 15 轮：金币收益族触发池备案

- 用户方向：金币类先统一为“获得金币”，通过不同触发条件区分。用户提出五个主方向：指定点数给金币，风险最大但给得最多且指定点必须存在于骰子面组；最大点数给金币；触发不同牌型给金币；回合结束给金币；利息加倍。希望继续补充备案方向。
- 设计诊断：这五个方向已经覆盖了金币族的主干，但它们的触发频率差距很大。指定点数属于低频高额，最大点数和牌型属于中频构筑触发，回合结束和利息加倍属于稳定经济系统触发。稳定触发如果给得太多，会让金币类从“构筑收益”变成“必买经济底座”。
- 备案触发池：低点补偿、同点伙伴、孤点、顺子成立或顺子缺口、出千实际重摇、出千后点数变大或变小、首次出手、最后一次出手且通关、未达标补偿、超额达标、连续触发、槽位顺序、前序金币入账、钱包阈值、市场返利、负收益换金币、临时小骰数量、大树成长。
- 倾向分层：基础泛用金币可使用指定点、最大点、低点、首次出手；构筑触发金币可使用同点、孤点、牌型、出千、临时小骰数量、大树成长；经济系统金币可使用利息加倍、钱包阈值、市场返利、回合结束。
- 风险判断：回合结束、利息加倍、未达标补偿和高钱包滚利风险最高，需要低额、封顶、每小关限次或章节后置；指定点、低点补偿、末投通关等低频触发更适合作为高回报金币骰。
- 决策状态：备案池，待筛第一批金币收益族候选。

### 第 16 轮：金币与大树软融合

- 用户方向：金币触发池里很多方向显得太硬，和大树的融合会更好。
- 设计诊断：单纯把金币做成“任意条件触发 + 获得金币”，容易变成机械触发器。大树与金币的融合更自然，因为两者都能表达长期资产：种下本金、照料成长、成熟收成、利息复利。这样金币收益不是凭空发放，而是像大树成长后的收成。
- 倾向框架：金币与大树的连接优先围绕四类事件：大树成长时给金币、大树成熟后给稳定收益、金币投入换成长机会、大树成长提高利息或市场价值。
- 设计边界：不要复活已否决的“果实生成临时小骰”方向；如果使用果实或收成意象，应只产出金币或经济收益，不生成额外骰子，避免和龟龟路线重叠。
- 决策状态：强倾向；第一批金币交融候选优先从大树融合里筛。

### 第 17 轮：修正为大树式柔化，而非大树硬绑定

- 用户方向：上一轮理解过于直接。用户真正想表达的是，之前的金币触发条件如果直接和大树绑定也会很硬，需要柔化。
- 设计诊断：正确方向不是“某颗大树成长就给金币”，而是让金币骰拥有类似大树的时间结构。金币收益可以先种下条件或本金，经过若干次投骰、命中、保留或通关后成熟，再收成金币。这样大树 build 会因为长期成长、高点和保留资产而自然适配，但金币骰本身不要求玩家必须拥有大树骰。
- 倾向框架：把金币触发从即时判定改成中间态：播种、蓄账、成熟、收成、复利。触发条件仍可以来自指定点、最大点、牌型、回合结束、利息等，但不应全部当场发钱；部分触发应推进成熟进度，成熟后才给金币。
- 设计边界：避免“有大树 / 大树成长 / 园丁触发 / 灌溉触发就给金币”的直接规则。可以保留大树意象和长期节奏，但不要让金币族变成大树族的附属组件。
- 决策状态：已被第 18 轮进一步修正；节奏借鉴仍然偏离用户重点，用户真正关注的是金币玩法与大树玩法的交接形式。

### 第 18 轮：金币与大树交接需要中间层

- 用户方向：不是想让金币骰子的节奏和大树骰子类似，而是金币骰子和大树玩法交接的形式不能太硬。`大树成长一次，金币 +1` 这种融合方式不成立。
- 设计诊断：金币与大树的融合点不应是“成长事件直接触发金币”，而应是“成长结果形成资产状态，再由金币玩法在另一个时机读取或兑现”。这样玩家看到的是树变得值钱、能抵押、能签订单、能在市场收成，而不是每次成长旁边弹一个金币数字。
- 倾向框架：增加一个交接层。可选交接层包括：`果实` 作为可售资源、`树契` 作为提前签下的收益条件、`估值` 作为大树成长后的市场价值、`抵押` 作为用成熟大树换临时资金、`评级` 作为利息或折扣依据、`季节收成` 作为阶段性结算。
- 设计边界：大树仍然做成长和点数资产，金币玩法负责市场、利息、订单和兑现。两者通过资产状态或市场行为连接，不通过每次成长事件直接加金币。
- 决策状态：强倾向；第一批金币与大树融合候选应优先使用中间层交接。

### 第 19 轮：龟龟类额外骰子候选池

- 用户方向：需要一版龟龟类骰子的待选方案。
- 设计诊断：新版龟龟类不应继续理解为“只有龟龟 build 会买的专属套件”，而应定义为“额外骰子收益族”。共同收益是生成临时小骰，差异来自触发条件和临时小骰质量。这样同点、顺子、高点、低点、出千、金币等构筑都可以用龟龟类补分，但仍保留龟龟的结算数量爽感。
- 候选池：`伏龟` 低点触发补小骰，`顶龟` 最高面触发补中骰，`同巢龟` 有同点伙伴时补小骰，`桥龟` 顺子或顺子缺口触发补骰，`铅龟` 出千实际重摇后补骰，`钱龟` 支付金币换临时骰，`回声龟` 读取前序临时骰事件追加 d1，`主龟 / 慢龟` 保留点数递减链作为核心来源。
- 倾向分层：稳定小补使用 d1 或 d2；构筑触发使用 d2 或 d3；花金币或出千这种有成本的触发可以给 d3 或短链；连锁扩张只允许每颗骰每次结算触发一次，避免临时骰互相触发闭环。
- 设计边界：临时小骰默认不参与主牌型、不继承材质和词缀、不触发实体骰类型效果；是否计入壳匠或未来“数量兑现”要单独定义。壳匠类更适合作为“临时骰兑现族”，不直接混进所有生成类骰子。
- 决策状态：待选池，建议先筛 4 到 6 颗首批候选。

### 第 20 轮：龟龟升级为连锁爽点和积分表演核心

- 用户方向：上一版龟龟方案不够爽。用户希望龟龟和大树都是爽点来源的重头戏；龟龟必须有连锁爽感，且龟龟的积分表演也是重点。
- 设计诊断：如果龟龟只被定义成“不同条件生成一颗临时小骰”，它会退化成泛用补分工具，无法承担核心爽点。龟龟的主价值应该是连锁过程本身：一颗骰子裂出小骰，小骰继续推动链条，链条再被壳匠、回声、巢穴或终点结算放大，玩家看到基础分数字连续跳涨。
- 倾向框架：龟龟类拆成五类角色：`链源` 负责启动链条，`续链` 负责延长链条，`分叉` 负责把一条链变多条链，`回声` 负责读取已生成的小骰补余波，`兑现` 负责把链长或链尾变成明显分数跳涨。首版不应只做触发池，而应保证至少有链源、续链、分叉和兑现。
- 积分表演要求：龟龟结算应有独立表演层。主骰入账后，临时小骰沿来源骰展开为链条；基础分数字逐颗跳涨；链条到达壳匠、回声或终点时出现一次更大的跳涨；超过展示上限时不简单隐藏，而用折叠链节和连续小跳表达“还在滚”。
- 设计边界：保留防爆上限，但上限应服务节奏而不是削弱爽感。可以使用每条链最大深度、每颗实体骰每次最多触发一次、全局额外骰展示折叠、只允许主龟/慢龟产生自然递归等规则。
- 决策状态：强倾向；第 19 轮“额外骰子收益族”降为底层收益说明，龟龟首要定位改为连锁爽点与积分表演核心。

### 第 21 轮：避免龟龟连锁重新闭环

- 用户方向：第 20 轮的逻辑没有问题，但仍然出现了之前的问题：build 闭环、局部零件无用、核心拿到没有配件也无用。
- 设计诊断：如果把龟龟拆成纯链源、纯续链、纯分叉、纯兑现，会形成套装依赖。玩家拿到壳匠但没有小骰来源会失效，拿到链源但没有续链或兑现又不够爽，这正是需要解决的构筑壁垒。
- 倾向框架：每颗龟龟骰都应采用“双层能力”：基础层是自带一个小型链源或稳定临时骰收益，保证单颗可用；协同层是在已有临时小骰、链长、链尾、同点、顺子、金币或出千环境下升级为续链、分叉、回声或兑现。
- 设计规则：不做纯核心件，不做纯配件，不做纯兑现件。任何龟龟候选都必须回答两个问题：单颗买到时能不能产生一次可见小爽点；已有链条时能不能把爽点放大。
- 结构目标：把“核心 + 配件”改成“每颗都是小核心，互相变成配件”。这样市场里任意一颗龟龟都不是死件，多颗组合时才出现更大的连锁和积分表演。
- 决策状态：强修正；龟龟首批候选必须按“自带底线 + 协同升级”重写。

### 第 22 轮：龟龟应放大其它骰子收益

- 用户方向：第 21 轮仍有问题，因为又变成追多龟龟的玩法。用户要的龟龟更通用：每个龟龟骰子都应可以放大其它骰子的收益，而不是龟龟骰子自己整成闭环。
- 设计诊断：第 21 轮解决了“单颗龟龟死件”，但没有解决“龟龟仍然只和龟龟互相追”的问题。正确目标不是让龟龟彼此成套，而是让龟龟成为其它 build 的余波放大器。奇数同点、偶数顺子、大树高面、金币支付、赌徒高点、出千重摇等成功事件，都可以被龟龟转成临时小骰连锁。
- 倾向框架：龟龟候选应采用“寄托对象 + 余波小骰”的模板。寄托对象可以是左侧非龟骰、右侧非龟骰、本次最高分非龟骰、刚触发牌型的骰、刚消费金币的骰、刚出千重摇的骰。龟龟不复制对方完整效果，只把对方的成功转换成小骰余波。
- 设计规则：首批龟龟优先放大非龟骰；临时小骰不触发龟龟；龟龟之间不互相读取，或只允许高阶稀有骰读取一次。这样保留龟龟的积分表演，但不鼓励玩家只追多龟龟。
- 结构目标：龟龟在任意 build 中都能成为“加厚器”。玩家已有同点、顺子、高面、金币或出千路线时，买一颗龟龟就能让该路线的成功多一段可见滚分，而不是要求继续买龟龟配件。
- 决策状态：强修正；龟龟首批候选从“自成连锁”改为“其它骰收益放大器 + 龟龟式积分表演”。

### 第 23 轮：金币收益族九类候选包装

- 用户方向：当前骰子过于独立，后续金币类骰需要更泛用。用户提出九个金币备案方向：指定点数给金币、最大点数给金币、触发不同牌型给金币、回合结束给金币、利息加倍、被出千并实际重摇后给金币、临时小骰数量达到门槛给金币、槽位触发、削减其它骰子点数获得金币。
- 设计诊断：这九类候选比单纯扩金币路线更有价值，因为它们把金币收益挂到已有可理解事件上。指定点数和最大点数接点数控制与高面；牌型接奇偶与同点顺子；出千接重摇决策；临时小骰数量接龟龟；槽位接左到右结算顺序；削点换金币接高点、大树和风险取舍。真正风险在于回合结束和利息类过于稳定，必须低额、封顶或后置。
- 包装候选：`悬赏骰` 追本骰已有面组中的指定点，低频高额；`顶金骰` 掷出自身当前最高点给金币，中频中额；`牌税骰` 按本次最终牌型等级给金币，构筑触发；`收账骰` 本次出手确认结算结束给少量金币，稳定保底；`复利骰` 通关结算时复制部分利息，经济系统触发；`铅票骰` 本骰被确认出千且实际重摇后给金币，出千触发；`壳税骰` 本次真实生成临时小骰数量达到门槛后给金币，龟龟触发；`柜台骰` 读取左到右槽位顺序或相邻槽位事件给金币，位置触发；`伐木骰` 临时削减其它实体骰的单骰分或有效点数来换金币，风险触发。
- 首批倾向：如果要马上进入生产包，建议从 `悬赏骰 / 顶金骰 / 牌税骰 / 铅票骰 / 壳税骰 / 伐木骰` 中筛 4 到 6 颗。`收账骰` 和 `复利骰` 更适合作为经济系统压力测试候选，不宜和第一批高触发金币骰同时上线。`柜台骰` 值得保留，但要先确定槽位规则是“相邻读取”还是“前序读取”，否则 UI 说明会变重。
- 结算边界倾向：金币统一按当前骰袋槽位从左到右写入钱包；除 `伐木骰` 的高阶版本外，金币骰默认不改变主牌型识别。削点换金币首版建议只削本次单骰分，不改有效点数，避免它同时变成控点工具；如果后续要做真正改有效点数，应作为高阶变体单独生产。
- 决策状态：备案方案已包装；尚未确认首批生产范围和具体数值。

### 第 24 轮：确认九颗金币骰都进入后续范围

- 用户方向：九个金币骰子都需要。后续可以通过其它 skill 推进流程并产生生产包。
- Codex 回应：记录为确认范围，不在本轮直接生成生产包。后续应切换到 `$wabish-production-pipeline`，以 D1 设计对话记录为来源，创建“金币收益族泛用化”生产包；必要时再用 `$wabish-packet-review` 审核生产包，之后交给程序、美术或界面体验对应流程推进。
- 形成的洞察：九颗骰子都保留，并不意味着同一实现批次一次性上线。更稳妥的方式是一个总生产包覆盖九颗金币骰的设计目标、通用结算边界和共享 UI 语言，再在执行计划里拆成高风险 / 中风险 / 稳定经济三组子切片。
- 决策状态：已确认九颗进入后续范围；规则数值、批次拆分和具体实现边界待生产包确定。

### 第 25 轮：确认龟龟后续优化的泛用机制方向

- 用户方向：这个模块主要探讨龟龟骰子后面的优化方向；当前已确认主体思路是打破各个类型骰子的壁垒，把骰子全部转成更通用的机制提供者。
- 设计诊断：这不是单纯给龟龟再补几颗同路线组件，而是确认第 14 轮以来的“双轴重构”作为更高层原则。旧结构按流派分组件，容易让龟龟、金币、奇偶、大树各自封闭；新结构应让骰子提供“触发条件、收益类型、结算机制”这类可被不同构筑吸收的能力。
- 龟龟定位：龟龟后续不再主要追求“多龟龟互相套娃”，而是负责把非龟骰、相邻槽位、牌型成立、金币支付、出千重摇、高面命中或其它成功事件转成临时小骰余波。龟龟仍保留连锁表演和滚分爽点，但连锁来源要更多来自其它 build 的成功。
- 设计边界：泛用不等于无条件强。每颗骰仍需要清楚说明触发条件、收益类型、读取对象、每次结算上限、是否参与壳匠类兑现、出千后是否刷新、是否进入左到右结算流，以及市场章节和价格限制。
- 决策状态：主体方向已确认；具体龟龟候选池、规则卡、旧龟龟骰是否重做、以及通用机制接口仍待后续生产包或下一轮讨论细化。

### 第 26 轮：龟龟泛用机制备选案

- 用户方向：希望获得一批龟龟机制的好备选案，用于后续筛选。
- 设计诊断：首批龟龟机制不应追求“所有触发都能读”，否则会变成必买加分件；也不应只读龟龟临时骰，否则会退回自闭环。更合适的候选应围绕当前已经可读的玩家决策：槽位顺序、同点、顺子、高面、出千、金币支付和首次成功事件。
- 候选池：
  - `背壳龟`：读取左侧相邻非龟实体骰；若左邻本次单骰分高于自身有效点数对应的基础分，生成 1 颗 d2 临时小骰。若差值达到高档，可升级为 d3。核心价值是把加分类收益、固定分、投资、国库、材质等统一读成“左邻被强化过”。
  - `同潮龟`：当本骰或相邻非龟骰有实体同点伙伴时，生成 d2；如果最终形成三同或更高同点牌型，升级为 d3。核心连接奇数追同、盖章、铜骨和同点面组。
  - `桥壳龟`：最终实体骰形成顺子，或由半步参与形成顺子时，生成 d3。核心连接偶数顺子、半步和顺子面组。
  - `顶壳龟`：相邻非龟骰掷出自身当前面组最高点，或有效点数达到高面阈值时，生成 d3。核心连接高面模板、大树成长、赌徒高点和顶金类收益。
  - `铅壳龟`：相邻非龟骰被确认出千并实际重摇后，生成 d3。核心连接出千选择，让玩家在重摇目标之外考虑相邻龟龟的余波收益。
  - `租壳龟`：本次结算前序或相邻非龟骰发生钱包消耗时，生成 d2 或 d3。核心连接投资、贿赂和后续金币支付，但经济闭环风险较高。
  - `首响龟`：本次结算第一个非龟类型、材质或词缀成功触发后，生成 d1 或 d2。它是最泛用的原型，但 UI 需要清楚显示“首个成功事件”，否则玩家会觉得规则抽象。
  - `潮汐龟`：本次结算出现两种以上非龟事件族时，按事件族数量补 d1，最多 3 颗。它能奖励混合 build，但更适合中后期或 Boss 后市场，不适合首批。
- 首批建议：优先从 `背壳龟 / 同潮龟 / 桥壳龟 / 顶壳龟 / 铅壳龟` 中筛 4 到 5 颗。`租壳龟 / 首响龟 / 潮汐龟` 保留为第二批或高阶候选。
- 决策状态：候选案，未确认规则数值，未进入生产包。

### 第 27 轮：首批五个候选可继续，统一 dN 表达

- 用户方向：首批里的五个候选可以；同时询问 `d1 / d2 / d3` 的含义。
- 解释口径：`dN` 是临时小骰规格，不是数量。`1 颗 d2 临时小骰` 表示生成一颗小骰，真实结算时随机掷出 `1` 或 `2`；`d1` 固定为 1，`d3` 随机 1 到 3。这个表达沿用常见桌游骰子记法，也贴合当前龟龟临时小骰的“最大点数”逻辑。
- 设计用途：用 `d1 / d2 / d3` 给触发难度分层。稳定触发只给 d1 或 d2；同点、顺子、高面等构筑触发可给 d2 或 d3；出千重摇这种消耗机会的触发可以给 d3。这样玩家能直观看到“越难触发，小骰质量越高”。
- 决策状态：首批五个候选进入规则卡细化范围；`dN` 作为内部规则表达可用，最终 UI 文案仍需决定是否直接显示 `d2`，还是显示“小骰 1-2 点”。

### 第 28 轮：确认龟龟需要统一壳链底座

- 用户方向：龟龟还是要有一个通用机制，哪怕是生成出来的龟龟骰子；同时询问当前是否已有固有生成链。
- 当前事实：已有固有链。`Turtle` 按当前点数 `v` 生成最大值为 `floor(v / 2)` 的临时小骰，若最大值小于 1 则停止；生成的小骰掷出实际点数后继续按同样递减逻辑生成下一颗。`SlowTurtle` 使用 `ceil(v / 2)`，但在 `v <= 1` 时停止，避免无限链。`Nest` 在有效龟龟链之后补 d1，`Shellsmith` 读取龟系临时小骰数量兑现加分。
- 设计判断：这条现成递减链应上升为龟龟族的统一机制底座，可命名为“壳链”或“龟龟余波链”。旧 `Turtle / SlowTurtle` 是自启动链源，新 `背壳龟 / 同潮龟 / 桥壳龟 / 顶壳龟 / 铅壳龟` 则是外部事件链源；它们不各自发明临时骰规则，而是统一调用壳链接口。
- 接口草案：一段壳链至少记录链源事件、触发骰、种子规格 `d1 / d2 / d3`、递减规则、链深或展示上限、是否计入壳匠类数量兑现、是否允许被巢穴补骰。默认生成物是“龟系临时骰”，参与单骰加分和表演，但不参与主牌型、不继承材质词缀、不触发实体骰类型效果、不能被出千选择。
- 决策状态：确认龟龟需要统一壳链底座；具体命名、Shellsmith/Nest 是否读取外部事件生成的壳链、链深上限和 UI 展示方式仍待生产包规则卡细化。

### 第 29 轮：壳链需要比除半链更长

- 用户方向：希望龟龟保持更长链路；当前每次 `/2` 的维度感觉很快结束，不足以支撑龟龟的连锁表演。
- 设计诊断：当前链短的根因不是 `d1 / d2 / d3` 数值太低，而是“下一节规格读取本节实际掷点”。例如 `d3` 一旦掷出 1，链马上停止；即使龟龟起手点数高，链也经常只滚 1 到 2 节。
- 首选修正：把壳链改成“最大值逐级衰减”。链源先决定种子规格 `dN`，每一节实际从 `1..N` 掷分，但下一节规格按 `N-1` 衰减，而不是按本节实际掷点除半。示例：`启动 d3 壳链` 展开为 `d3 -> d2 -> d1`；实际掷出的 1、2、3 只决定本节加几分，不会让链提前早死。
- 规则表达建议：区分“生成 1 颗 d3 临时小骰”和“启动 d3 壳链”。前者只是一颗小骰；后者才展开为 `d3 -> d2 -> d1`。这样不推翻第 27 轮对 `dN` 的解释，也能给龟龟专属机制留出更长表演。
- 风险边界：如果壳匠仍按每颗龟系临时骰 +1，稳定长链会显著抬高壳匠收益；因此生产包必须同步定义壳匠读取上限、按链计数还是按壳节计数、巢穴是否只按链源补 1 次。
- 决策状态：确认“未来壳链需要更长”这一体验目标；推荐方案为“最大值逐级衰减 + 最大链深上限”，具体数值和旧骰重做方式待规则卡定稿。

### 第 30 轮：壳链底层进入当前原型实现

- 用户方向：认可最大值逐级衰减方案，可以按这个思路修改龟龟骰子的生成逻辑，后续依赖这个底层。
- 实现口径：当前 `Turtle / SlowTurtle` 仍只作为链源，不新增外部事件龟龟。`Turtle` 用有效点数 `floor(v / 2)` 计算壳链种子；`SlowTurtle` 用有效点数 `ceil(v / 2)` 计算壳链种子；两者之后都调用同一套壳链底层，按 `dN -> d(N-1) -> ... -> d1` 展开。
- 结算边界：临时骰仍是龟系临时骰，只追加单骰分，不参与主牌型、不继承材质词缀、不触发实体骰类型效果、不能被出千选择。壳匠、巢穴、壳税继续读取真实生成的临时小骰数量；这会提高龟龟链数量，需要后续实测。
- 程序同步：`Assets/Scripts/DiceKingDemo.cs` 已将真实结算和预览估算改为同一壳链模型，并把短标签从旧“慢衰”改成“壳链 / 慢壳链”口径。`PROJECT_CONTEXT.md`、`GAME_FLOW.md`、`DICE_ARCHETYPES.md` 已同步当前行为。
- 决策状态：已进入当前原型实现；仍需 Unity Play Mode 验证真实生成数量、预览估算、壳匠加分、壳税触发、折叠展示和结算节奏。

### 第 31 轮：奇偶泛用化备案

- 用户方向：询问奇偶骰子改版如何打破专用性并配合全体骰子，希望参考已经改版后的金币类和龟龟类。
- 设计诊断：金币族的共同收益是钱包金币，龟龟族的共同收益是壳链或临时小骰余波；奇偶族如果也强行做成资源产出，会和金币或龟龟抢身份。奇偶更自然的公共价值是“点数结构校准”：帮助现有骰面、牌型、槽位和出千结果被更好地组织，而不是新增一套奇偶资源。
- 推荐方向：保留奇偶作为牌型和点数结构的胶水。候选规则优先读取全体实体骰的同点、差一、同奇偶间隔、奇偶数量比例、相邻槽位和出千奇偶变化，收益只给一次牌型借位、小额固定分、重摇复核或槽位兑现，不新增全局牌型表。
- 备选候选：
  - `平衡骰`：当前实体骰奇偶数量为 `3 / 3` 时，本骰获得固定中高单骰分；用于奖励混合骰袋，而不是纯奇或纯偶。
  - `桥格骰`：本骰与相邻实体骰点差为 1 时，可在顺子识别中提供一次桥接或小额单骰分；用于把任意构筑的相邻摆位接入顺子。
  - `同栏骰`：本骰与任意实体骰同奇偶且点差为 2 时获得小额收益；如果形成 `1 / 3 / 5` 或 `2 / 4 / 6` 三节点，可升级一次固定分。
  - `换栏骰`：本骰被确认出千并实际重摇后，若奇偶发生变化，获得小额固定分或一次牌型校准；用于让出千目标选择更有结构判断。
  - `验算骰`：自然开盅后，如果本骰是当前唯一奇数或唯一偶数，可自动重摇自己一次；用于泛化孤证的“复核”感觉，但必须限制每次出手一次且出千后不触发。
- 不推荐方向：不要新增“奇偶能量”“奇偶币”或多段奇偶计数器；不要把全奇、全偶继续扩成更多专用倍率；不要让奇偶改版直接改变所有骰子的有效点数，否则会快速破坏当前简化牌型表。
- 决策状态：备案池，未确认规则、数值和生产范围；若继续推进，应先筛 3 到 5 颗规则卡，并决定是否作为 F001-C 后续子包还是新的奇偶泛用机制包。

### 第 32 轮：奇偶不做持续养成，改走相位操作

- 用户方向：牌型命中后给骰子加成这个点可以保留，但如果它形成持续性的养成，应放进大树型骰子。奇偶类机制需要更有意思、更有区分度。
- 设计诊断：这是一次重要边界修正。奇偶如果继续做“某牌型命中 -> 本骰变强”，会和大树的长期成长抢功能；如果只是“满足奇偶条件 -> 固定加分”，又会退回收益表。奇偶应负责当手结构，不负责长期积累。
- 修正原则：奇偶机制优先做短时判断和解释权，例如本手是否反相、谁是少数派、相邻槽位是否接线、出千是否改变奇偶、一个破坏全奇/全偶的异类是否被临时处理。不新增长期字段，不写入 `DiceData`，默认不改变长期面组。
- 优先候选：
  - `接线骰`：读取左右相邻实体骰。若左右邻奇偶相反，本骰接上线路，获得一次局部收益或为顺子识别提供一次桥接。重点是投前摆位，而不是牌型命中加成。
  - `反相骰`：如果本骰是唯一破坏全奇或全偶的异类，可在副牌型判定中临时反相一次；不改变显示点数、有效点数或长期面组。
  - `票决骰`：读取当前奇偶多数和少数。本骰若站在少数阵营，可获得一次高质量即时收益；若站在多数阵营，则只提供稳定低收益。重点是让混合骰袋也有选择。
  - `换相骰`：本骰被确认出千并实际重摇后，如果奇偶发生变化，触发一次换相信号，可影响自身本次分数或相邻骰的局部判定。重点是让出千目标选择多一层判断。
  - `夹缝骰`：当本骰左右邻同奇偶、自己为相反奇偶，形成 `奇-偶-奇` 或 `偶-奇-偶` 夹缝时触发。它鼓励玩家有意识地排布槽位，而不是追纯奇或纯偶。
- 降级候选：`平衡骰` 和旧版 `同栏骰` 如果只是奇偶比例或同奇偶命中后加固定分，区分度不足；除非改成票决、反相或接线机制，否则不进入首批。
- 决策状态：用户已确认奇偶不承担持续养成；具体候选仍是备案，未确认规则数值和生产范围。

### 第 33 轮：奇偶规则压缩，避免长说明

- 用户方向：思路可接受，但规则必须简化，不能让玩家读长文本。
- 设计诊断：上一轮的“相位操作、票决、接线”是设计层语言，不能直接出现在玩家规则卡。奇偶首批应只保留一眼可判定的场景：左右夹缝、补全全奇或全偶、出千后奇偶变化。
- 短规则候选：
  - `夹缝骰`：两边同奇偶，本骰相反，8 分。
  - `补全骰`：只差本骰时，补全全奇/全偶。
  - `换相骰`：出千后奇偶变了，8 分。
- 降级处理：`票决骰` 暂缓，因为“多数/少数阵营”需要额外解释；`接线骰` 并入 `夹缝骰`；`反相骰` 改名为 `补全骰`，避免玩家理解“反相”。
- 文案标准：市场卡只写一句核心规则；结算中只显示 `夹缝 +8`、`补全`、`换相 +8` 这类短标签；不解释相位、不新增计数器、不引入长期成长。
- 决策状态：确认简化标准；具体数值仍是占位，生产包阶段再调。

### 第 34 轮：夹缝降级，补充奇偶短规则备案

- 用户方向：没有理解 `夹缝骰：两边同奇偶，本骰相反，8 分`，希望再给更多备案。
- 设计诊断：`夹缝骰` 原意是 `奇-偶-奇` 或 `偶-奇-偶` 的三槽摆位奖励，但这需要玩家同时看左右两颗和本骰关系；如果还要解释例子，说明它不适合作为首批短规则。奇偶候选应优先使用一邻、全局数量或出千变化这类更直观的判断。
- 夹缝处理：`夹缝骰` 暂时降级，不删想法，但不作为首批推荐。若未来保留，应改成更像界面一眼能提示的版本，例如只读左邻或只读右邻。
- 新备案池：
  - `异邻骰`：左邻奇偶不同，本骰 6 分。
  - `同邻骰`：左邻奇偶相同，本骰 6 分。
  - `补全骰`：只差本骰时，补全全奇/全偶。
  - `复核骰`：本骰破坏全奇/全偶时，重摇一次。
  - `翻号骰`：出千后奇偶变了，8 分。
  - `守号骰`：出千后奇偶没变，6 分。
  - `转号骰`：出千重摇必变奇偶。
  - `独门骰`：本骰是唯一奇/偶时，8 分。
  - `少数骰`：本骰属于少的一边，8 分。
  - `平衡骰`：三奇三偶时，本骰 8 分。
  - `隔邻骰`：左邻点差为 2，本骰 7 分。

### 第 37 轮：大树专用性泛用化备案

- 用户方向：大树骰子后续也需要修改专用性，希望先获得一批备案方向。
- 设计诊断：当前大树首版的 `园丁 / 灌溉` 都高度围绕 `Tree` 自身命中成长，路线记忆清楚，但容易变成“大树只买大树补件”的封闭套装。后续应保留大树的长期资产感，同时让其它构筑可以通过高面、牌型、出千、槽位或市场行为接入成长机制。
- 备案原则：大树的泛用收益不应变成临时小骰，避免和龟龟重叠；也不应直接变成钱包金币，避免金币硬绑定。更合适的共同收益是成长进度、成熟阈值、护荫支持、资产估值或市场兑现。

  - `冠层骰`：相邻非大树骰掷出自身当前面组最高点时，本骰获得成长进度；满一定进度后自身成长一次。连接高面模板、赌徒高点、顶金类收益和已成长大树。

  - `修枝骰`：本骰被确认出千并实际重摇后，如果本次结算通关或形成更高牌型，本骰获得成长进度。连接出千决策和成长收益，避免无脑为成长乱出千。
  - `护荫骰`：本骰达到成熟阈值后，左邻或右邻首次触发自身类型效果时获得小额单骰分，或让本骰获得成长进度。连接槽位摆放和其它路线触发，但每次结算必须限次。
  - `树契骰`：市场阶段或通关阶段读取成熟大树的资产状态，转成折扣、抵押额度、利息修正或订单兑现；不在成长瞬间直接发金币。连接金币玩法，但通过市场或阶段结算隔一层。
  - `年轮骰` 重做方向：不再按成长层数直接加分，而是把若干次真实成长转成一次成熟节点；成熟节点改变下一次触发质量，例如高面更容易被读取、护荫更强或市场估值更高。
- 初步筛选：首批更推荐 `冠层骰 / 年鉴骰 / 修枝骰`，因为它们分别接入高面、牌型和出千三类已有决策，界面负担较低。`护荫骰 / 树契骰 / 年轮骰` 更适合第二批，因为它们牵涉成熟阈值、市场读取或邻位支持，容易增加状态。

### 第 38 轮：确认触发点白名单并重设大树骰子

- 用户方向：要求把几个固定触发点记录在项目工程里，后续设计参考这些触发点，不要随意设计；并基于这些触发点重新设计大树骰子。
- 已确认触发点白名单：指定点数、指定牌型、最高点数、回合结束、金币、出千、槽位触发。
- 设计诊断：第 37 轮的大树备案仍偏“散点候选”，容易继续临时发明触发。现在应先固定触发来源，再把大树收益统一成长期成长接口。这样后续设计会从“我当前能触发什么”出发，而不是从“再给大树想一个新花样”出发。
- 大树统一收益建议：大树类骰优先产出 `成长进度`，由生产包决定 `2`、`3` 或更高进度兑现 1 次真实成长。低频触发可直接成长或给更多进度；稳定触发如回合结束和金币只给低进度。真实成长仍只在二次确认结算后入档。
- 七触发大树候选：
  - `点籽树`：指定点数触发。每次出手生成或读取一个指定点，本骰命中后获得成长进度。它是当前 `Tree` 的可读重做入口。
  - `牌谱树`：指定牌型触发。小关或出手指定一个现有牌型，最终形成该牌型时获得成长进度。它让奇偶、追同和顺子构筑接入成长。
  - `冠层树`：最高点数触发。本骰掷出自身当前面组最高点时获得成长进度。它连接高面模板、成长后的大树、赌徒高点和顶金类机制。
  - `年轮树`：回合结束触发。真实结算结束时获得少量成长进度。它是稳定低额成长底座，必须用较高阈值或每小关限次控制强度。
  - `肥料树`：金币触发。本次真实结算出现金币入账、金币支付或通关利息结算时获得成长进度。它让金币路线滋养大树，但大树本身不直接产金币。
  - `修枝树`：出千触发。本骰被确认出千并实际重摇后，如果结果变好、形成目标牌型或帮助通关，则获得成长进度。它把出千变成成长取舍，避免无脑重摇。
  - `根系树`：槽位触发。左邻或右邻实体骰触发自身类型效果后，本骰获得成长进度。它让投前摆位和左到右结算顺序参与成长。
- 初步推荐首批：`点籽树 / 牌谱树 / 冠层树 / 修枝树`。它们分别覆盖指定点数、指定牌型、最高点数和出千，玩家能从当前系统直接理解。`年轮树 / 肥料树 / 根系树` 先后置，因为稳定性、经济闭环和固定站位风险更高。

### 第 39 轮：大树七触发详细方案草案

- 用户方向：希望获得相对详细的大树重设方案，后续设计可以按该方案细化，而不是只停留在触发点备案。
- 总体定位：大树不应再被理解成一条只买大树补件的封闭套装，而是一类“长期成长接口”。七类触发点负责回答“什么事件能让树长”，大树收益负责保持“成长进度、真实成长、成熟节点、长期面组资产”。
- 统一成长模型草案：
  - 每颗大树类骰保留 `成长层数` 作为长期资产；生产包可新增 `成长进度` 作为中间状态。
  - 默认建议 `3` 点成长进度兑现 `1` 次真实成长；低频或有成本触发可给 `2` 点进度，稳定触发只给 `1` 点进度。
  - 真实成长只在二次确认后的真实结算中写入，表现为本骰所有面 `+1` 且 `成长层数 +1`；不影响本次已经锁定的有效点数、牌型和分数。
  - 如果首版不想改存档，可先让成长进度只在小关内暂存，达到阈值才写入现有成长层数；如果要保留“差一点成熟”的长期资产感，则生产包必须新增持久进度字段并提高存档版本。
- 首批详细候选：
  - `点籽树`：每次出手固定一个指定点，优先从本骰当前面组中抽取；本骰最终有效点等于指定点时获得 `2` 点成长进度。出千不刷新指定点，因此玩家可以用出千追目标。定位是当前 `Tree` 的可读重做入口。
  - `牌谱树`：出手或小关开始指定一个已有牌型大类，例如三同、顺子、全奇、全偶；最终牌型命中时获得 `2` 点成长进度。更强同类牌型可以兼容命中，例如四同、五同、六同可视为命中三同目标。定位是让奇偶、追同和顺子构筑接入长期成长。
  - `冠层树`：本骰掷出自身当前面组最高面时获得 `1` 点成长进度；若生产包需要更强版本，可要求它同时也是本次实体骰最高有效点，改为获得 `2` 点进度。定位是连接高面模板、已成长大树、赌徒高点和顶金类触发。
  - `修枝树`：本骰被确认出千并实际重摇后，如果新结果相对旧结果有明确改善，或帮助形成目标牌型、完成通关，则获得 `2` 点成长进度。定位是把一次出千机会变成成长取舍，而不是鼓励玩家无脑重摇树。
- 后置候选边界：
  - `年轮树` 只能做稳定低额慢成长，建议真实结算结束时若本骰本次没有其它成长进度，则获得 `1` 点进度，并且每小关限一次或使用更高阈值。
  - `肥料树` 读取金币变化但不创造金币；首选读取左到右结算中已经发生的钱包入账、支付或通关利息事件，每次结算最多获得 `1` 点进度。
  - `根系树` 读取相邻槽位的明确类型触发，首版建议只读左邻且每次结算最多一次，避免形成固定最优站位和难以解释的“任意成功事件”。
- UI 和结算表达：短标签优先显示触发来源和进度，例如 `籽5`、`谱:顺`、`冠顶`、`修枝`、`芽 2/3`；真实成长时再播放一次 `成长入档`，展示面组从旧值跳到新值。主界面不应放长解释，详细原因进结算日志或牌型 / 骰子详情。
- 首批建议：先做 `点籽树 / 牌谱树 / 冠层树 / 修枝树` 四颗进入规则卡和接触图，不同时做七颗。`Tree / Gardener / Irrigation` 首版实现先保留为当前原型事实；等新四颗验证后，再决定是替换旧 `Tree`，还是把 `Gardener / Irrigation` 改成成长进度放大器和指定点支持骰。
- 决策状态：详细方案草案已记录，但仍未进入生产包、代码、数据或正式市场。正式执行前还需要补齐价格、章节、面组合法性、预览估算、存档字段、短规则文案和反滥用上限。

### 第 40 轮：肥料、年轮和根系口径修正

- 用户修正：`肥料树` 不读金币变化次数，改成和当前金币存量相关，类似利息且不加上限；`年轮树` 不按小关限一次，而是每次出手限一次；`根系树` 需要重新设计。
- `肥料树` 新口径：每次真实结算结束时读取当前钱包金币存量，按利息式分段换算成长进度，例如 `floor(当前金币 / 肥料步长)`，不设置单次成长进度上限。它不创造金币、不消费金币，也不读取金币变化次数，而是奖励玩家保留本金。具体 `肥料步长` 应和基础利息口径或生产包配置对齐，避免当前测试钱包数值直接炸穿成长节奏。
- `年轮树` 新口径：每次真实结算结束时获得 `1` 点成长进度；每次出手最多触发一次，不再按小关限一次。它不要求本次没有其它成长，但因为稳定性高，后续应通过章节、价格、成长阈值或市场权重控制强度。
- `根系树` 新方案：结果锁定后按本骰有效点数决定读取方向，奇数读取左邻，偶数读取右邻；目标邻位骰在本次真实结算中触发明确类型效果时，根系树获得 `2` 点成长进度，每次出手最多触发一次。边缘槽位若目标方向没有邻骰，则本次不触发。
- 根系读取范围：首版只读取实体骰的可见类型效果，例如金币入账或支付、临时小骰生成、成长进度或成长入档、自动重摇、出千收益、单骰分替换或加分、牌型借位或补全；不读取基础骰、纯面组限制、纯被动概率分布或材质触发。
- 设计判断：这个根系方案让玩家需要在投前摆位时考虑“根系树两侧分别放什么”，又因为方向由本骰奇偶决定，不会退化成永远把最强触发骰放在左边。奇偶控制、出千和槽位顺序都能参与该骰的成长判断。
- 决策状态：三项口径已作为大树重设最新草案同步到 `DICE_ARCHETYPES.md`；仍未进入代码、数据或正式市场。

### 第 41 轮：根系树机制简化

- 用户反馈：上一版 `根系树` 仍然偏复杂，奇数读左邻、偶数读右邻会让玩家同时判断点数、方向和邻位触发，超过短规则骰应承担的理解负担。
- 最新短规则：`根系树` 改为“相邻骰触发效果，本骰长芽”。左邻或右邻任一实体骰在本次真实结算中触发自身类型效果时，根系树获得 `2` 点成长进度，每次出手最多触发一次。
- 结算边界：根系树不再按自身奇偶决定方向，也不要求玩家判断左读还是右读。边缘槽位仍可读唯一相邻骰。若左右两侧都触发，也只获得一次成长进度。
- 读取范围简化：玩家侧只需要理解“相邻骰触发了类型效果”。程序侧生产包可用内部触发标签判断；基础骰、纯面组限制、纯被动概率分布和材质触发不计入。
- 设计判断：这一版牺牲一部分方向控制深度，换来更好的可读性。玩家仍有真实摆位决策：把根系树放在会触发效果的骰旁边，或夹在两个触发骰之间提高稳定性。
- 决策状态：根系树最新草案已同步到 `DICE_ARCHETYPES.md`；仍未进入代码、数据或正式市场。

### 第 42 轮：根系树相邻触发判定

- 用户问题：根系树的短规则可以接受，但需要明确“怎么判定左右效果触发了”。
- 判定原则：根系树不读取玩家肉眼猜测，也不读取所有被动规则，而是读取结算内部的“类型触发标签”。相邻实体骰只有在真实结算中实际产生可见类型事件，才算触发。
- 可计入事件：金币实际入账或支付、临时小骰生成、自动重摇实际发生、成长进度或成长入档、出千专属收益、条件型单骰分替换或加分、牌型借位或补全实际被采用、赌徒爆发或归零等。
- 不计入事件：基础点数计分、主牌型倍率、基础骰无效果、奇数或偶数骰的纯面组限制、双倍这类无条件常驻修正、材质触发、预览估算、未被采用的候选牌型。
- 结算顺序：先收集本次所有非根系骰的类型触发标签，再让根系树读取左邻和右邻标签；左右都触发也只获得一次成长进度。根系树由此产生的成长进度不再作为其它根系树的触发来源，避免根系互相连锁。
- 玩家侧表达：市场短规则仍保持“相邻骰触发效果，本骰长芽”；结算短反馈可以显示 `根:左`、`根:右` 或 `根:邻触发`，详细来源进入结算日志。
- 决策状态：根系判定规则已同步到 `DICE_ARCHETYPES.md`；仍未进入代码、数据或正式市场。

## Confirmed Decisions

- F020 中立八骰名单已确认：光牙、二重唱、触发器、王冠、接济、空袭、盟约、缝合。
- 中立只统计猪猪、恶魔、龟龟、海盗四个实体主体家族；吸附龟成为壳层后不再计龟龟家族，只统计宿主自身家族。
- 二重唱和触发器只处理主体家族真实结算事件，排除市场、离场、中立、纯被动和复制事件。
- 光牙永久全脸 +2，不回改本次锁定点数；王冠读取各主体家族最高最终有效点的向下取整均值作为额外单骰分。

- 本次讨论优先处理“各流派完善”，之后再讨论流派之间的交织。
- 奇数流派优先围绕相同点数牌型做文章。
- 偶数流派优先围绕顺子做文章，可以探索“差 2 的数字也能构成顺子”的偶数专属表达。
- 金币流候选中，国库骰和贿赂骰方向被认可。
- 金币流派第一批候选选择：国库骰、贿赂骰、投资骰。
- 龟龟流候选中，壳匠方向被认可。
- 龟龟流中，壳匠更适合按临时小骰数量给分，贴合链式爆发。
- 龟龟流派第一批候选选择：壳匠骰、巢穴骰、慢龟骰。
- 大树流派第一批实现范围收敛为园丁骰和灌溉骰。
- F001-D 第一版草案已被用户否决：园丁需要改为成长加速，年轮、树冠、苗圃的公式化兑现不符合当前调性。
- F001-D 第二版筛选：灌溉方向保留；果实、嫁接、幼苗否决；园丁需要更爆，倾向采用“有园丁就让自然命中的大树额外成长”的叠加方向。
- F001-D 首版执行决策：只实现园丁和灌溉，不再为本版补 1-2 个额外候选。
- 当前 build 优化的主要问题被确认：各流派之间过于独立，导致成型后的串联感和爽感不足。
- 后续需要设计交融玩法骰，用来衔接不同骰子类型和 build，而不是继续只扩每条路线的内部组件。
- 更强的整体修改方向是把骰子从“某 build 的固有组件”改为“触发条件 + 泛用收益”的双轴结构。
- 金币类骰的共同收益可以统一为获得金币，但通过不同触发条件区分；龟龟类骰的共同收益可以统一为获得额外骰子，但通过不同触发条件区分。
- 后续骰子优化的主体思路已确认：打破各类型骰子的封闭壁垒，把骰子转向更通用的机制提供者，由触发条件、收益类型和结算机制共同定义价值。
- 龟龟后续优化的核心不是继续做龟龟自闭环，而是让龟龟把其它骰子或其它 build 的成功事件转成临时小骰余波和连锁积分表演。
- 龟龟首批五个候选进入后续规则卡细化范围：`背壳龟`、`同潮龟`、`桥壳龟`、`顶壳龟`、`铅壳龟`。这只确认候选集可继续，不代表具体数值和实现范围已定稿。
- 龟龟需要统一的通用底层机制。现有 `Turtle / SlowTurtle` 的递减临时小骰生成链应作为“壳链 / 龟龟余波链”底座；新龟龟候选默认只是用不同可见事件启动同一套链，而不是各自生成一套孤立小骰规则。
- 未来壳链的体验目标应比当前每次 `/2` 更长、更稳定；当前倾向是把实际掷点和下一节规格拆开，用“最大值逐级衰减”替代按实际掷点快速除半。
- 当前原型已按最大值逐级衰减更新底层：`Turtle` / `SlowTurtle` 分别用 `floor(v / 2)` / `ceil(v / 2)` 决定壳链种子，后续统一展开为 `dN -> d(N-1) -> ... -> d1`。
- 金币收益族后续备案方向包含九类触发：指定点数、最大点数、牌型、回合结束、利息、出千实际重摇、临时小骰数量、槽位顺序和削点换金币。
- 九个金币收益族骰子都需要进入后续生产范围：`悬赏骰`、`顶金骰`、`牌税骰`、`收账骰`、`复利骰`、`铅票骰`、`壳税骰`、`柜台骰`、`伐木骰`。
- 奇偶改版的备案方向是“点数结构校准”，而不是新增奇偶资源或继续强化纯奇、纯偶专用路线；候选池包含平衡、桥格、同栏、换栏和验算等低理解成本方向。
- 奇偶首批玩家规则必须短卡化：不使用“相位、票决、接线”等抽象词；`夹缝骰` 因理解成本降级。当前用户指定保留的候选池为 `异邻骰 / 同邻骰 / 补全骰 / 复核骰 / 翻号骰 / 守号骰 / 转号骰`。
- 七颗奇偶短规则骰的后续顺序已确认：先出接触图并通过评审，再转程序实现；该流程已执行，当前已完成 Play Mode 验证。
- 后续推进方式确认：通过 `$wabish-production-pipeline` 从 D1 记录生成生产包；本设计对话不直接改代码、不直接更新正式骰子定稿。
- F010 奇偶短规则骰生产包已创建；第一阶段规则卡和接触图门禁、最终资源、程序接入和 Unity Play Mode 验证均已完成。
- 大树专用性修改进入备案方向：后续大树类骰应尽量读取高面、牌型、出千、槽位或市场资产等已有可见事件，触发后直接真实成长，表现为本骰全部面 `+1` 和现有 `Growth +1`；当前只确认备案池，不确认具体程序范围。
- 后续骰子设计的触发点白名单已确认：指定点数、指定牌型、最高点数、回合结束、金币、出千和槽位触发。后续候选应优先从这七类触发中选择或组合，避免随意添加孤立触发。
- 大树重设备案改为七触发结构：`点籽树 / 牌谱树 / 冠层树 / 年轮树 / 肥料树 / 修枝树 / 根系树` 分别对应七类触发，统一服务直接真实成长和长期面组变化；F011 已静态接入运行时，仍待 Unity 验证。

## Leaning Direction

- 骰子阶级应表达“适合进入构筑的阶段、前置依赖和规则复杂度”，不表达简单的强弱替代关系：`T1` 是可独立起步的入口件，`T2` 是需要一个已有条件的联动 / 转换件，`T3` 是改变上限或运行方式的核心 / 终结件，`T0` 继续只给基础骰和贡品骰等系统骰。
- 每颗正式市场骰除家族和阶级外，再增加一个主要构筑职责：产出、转换、放大、兑现或桥接。市场补位优先识别玩家缺少的职责，不只增加已经最多家族的权重，避免软引导把路线越推越窄。
- 三格市场分别承担稳定入口、构筑补位和高阶转折；建议第二格保持约 `60%–70%` 软倾向与 `30%–40%` 全局探索，第三格的付费刷新保底改为真正的 `T3` 保底，不再把常见的 `T2` 与 `T3` 合并成同一高阶结果。
- 当前 `44` 枚完整池先通过重新定级、职责标签和三格货架合同治理。完整池增长到约 `60` 枚以上时，再把完整图鉴池与每局市场名录分离，每局按家族和职责配额固定抽取约 `26–30` 枚并允许玩家查看。
- 低阶骰在后期仍应是高阶引擎所需的高效率输入，不允许形成“见到高阶就无脑替换低阶”的装备淘汰链；同一路线的生存和基本成型也不应依赖唯一命名 `T3`。
- 双阵营骰优先采用“两个家族标签 + 一句单向桥接规则”，不把两颗单家族骰的完整技能合并到一颗骰上。
- 黑帆魔蝠改为每次存在旧货架来源的刷新都吞吃 1 颗即将被替换的骰子，不再限制本市场首次；多颗黑帆各取不同来源，首次生成货架不触发。
- 本次拟添加名单收敛为 `黑帆魔蝠 / 驮粮龟 / 贡猪 / 黑市小鬼 / 血契船长 / 清仓猪 / 补给猪 / 共食魔龟 / 托底龟` 九颗。
- 暂不为了双家族六对全覆盖而补龟龟 / 海盗；上一轮未入选候选返回备选池，寄生魔龟和战利品龟维持淘汰。
- 双阵营骰作为实体时同时被两个家族识别；吸附成为壳层后不再提供家族标签，宿主也不继承双阵营身份。
- 同一次桥接只结算一次数值。允许同时发布两个家族事件标签供各自监听器读取，但复制、吸附壳复制和桥接衍生事件不能再次进入桥接，避免递归。
- 为保持 F020 的四族终局难度，当前推荐议会类统计采用“一颗实体最多占一个家族席位”；双阵营骰可以灵活补任意一侧，但不能一颗占两席。
- 优先为现有基础流派补齐内部结构：入口骰、资源转换骰、路线 buff 骰、稳定骰和中后期 payoff 骰。
- 新骰子应保持“一眼知道它为什么存在”的类型身份。
- 候选设计需要同时说明结算边界、出千交互、市场定位和 UI 表达。
- 金币流需要“金币如何转为战力”的骰子，但应避免把本回合刚由猪猪生成的账本金币立刻消费成分数，防止形成短闭环。
- 龟龟流需要围绕临时小骰的 buff 或 payoff，而不是只继续增加生成数量。
- 奇数/偶数流需要自己的衍生收益，避免只停留在提高全奇/全偶概率。
- 大树流可以围绕“命中更容易”“成长更值钱”“成长后有成熟奖励”“保护长期资产”四个方向补齐。
- 大树重做后应优先围绕成长事件和可见状态变化：额外成长、替树命中、结果、嫁接、幼苗变树，而不是读取成长层数直接加分。
- 大树泛用化后应把共同收益定义为“长期成长接口”，而不是“大树专属补件”。其它构筑可以通过高面、牌型、出千或槽位直接触发真实成长，但写入长期面组必须只发生在真实二次确认结算后。
- 大树和金币的连接继续保持中间层原则：可以读取成熟状态、资产估值、树契或市场兑现，不直接写成“成长一次给金币”。
- 大树和龟龟的分工应保持清楚：大树负责长期资产和成熟节点，龟龟负责当手临时小骰余波；大树候选不要用生成临时小骰作为主要收益。
- 大树七触发重设的首批更适合选择 `点籽树 / 牌谱树 / 冠层树 / 修枝树`。它们触发清晰、和现有投骰/牌型/出千关系直接，较少引入长期额外状态。
- `年轮树 / 肥料树 / 根系树` 仍更适合作为第二批观察项，但口径已修正：年轮按每次出手限一次，肥料按当前金币存量和标准金币步长做利息式直接成长且不加上限，根系改为相邻骰触发效果后直接成长，优先保证玩家一眼能读懂。
- 金币流需要从“按金币直接加分”扩展到更像经济行为的组件，例如投资、贷款、保险、竞价、税单、利息挪用等。
- 奇数流需要追同工具链：制造同点、奖励同点、保护同点结果、提高同点 payoff。
- 偶数流需要顺子工具链：差 2 视为顺子、补缺口、延长顺子、奖励顺子长度或顺子稳定性。
- 龟龟流需要链式工具链：增加链长、提高链上小骰质量、奖励链尾、奖励本次生成数量。
- 交融玩法骰应优先连接已有稳定事件：钱包金币变化、临时小骰数量、同点或顺子牌型、大树命中成长、出千重摇、最高面命中。
- 交融骰的首版目标是形成二段连锁，不追求无限连锁；例如一个路线产生资源，另一条路线把资源兑现为单骰分、成长机会、出千价值或牌型稳定。
- 交融骰应有明确路线归属和市场定位，避免成为任何 build 都必买的泛用强件。
- 优先采用“事件来源 -> 兑现对象”的设计模板：先定义哪件事发生，再定义谁把它变成收益。
- 交融优化不只靠新增骰子。市场推荐、结算展示和关卡规则可以先降低路线割裂感，再用桥接骰提供长期构筑深度。
- 第一批交融候选应控制在 4 到 6 颗，覆盖 2 到 3 组路线连接即可，避免一次性把所有路线打通。
- 新版骰子池应从“流派专属组件表”转为“触发条件表 + 收益类型表”。玩家构筑时可以选择自己擅长触发的条件，也可以选择自己当前缺少的收益。
- 奇偶泛用化倾向把共同价值定义为点数结构、牌型胶水和出千复核；首批不新增奇偶能量、不新增全局牌型、不把所有骰子直接改点。
- 收益类型必须足够泛用：金币能服务市场和金币转分，临时小骰能服务总分、壳匠类数量收益和未来结算链，成长能服务长期点数和牌型，点数控制能服务同点和顺子。
- 触发条件必须足够有性格：同点适合奇数和盖章，顺子适合偶数和半步，高面适合成长和偏置面组，低面适合低面模板，出千适合赌徒和补救，金币入账适合经济链。
- 仍应保留骰子类型的一句话身份，避免玩家只看到一堆“条件触发器”。
- 金币收益族应按触发频率分层：低频高额、中频中额、稳定低额。不能让回合结束、利息加倍这类稳定触发给出和指定点数、末投通关同级的收益。
- 第一批金币收益族建议优先保留用户提出的指定点数、最大点数、牌型、利息加倍，再从低点补偿、出千实际重摇、同点伙伴、槽位顺序或前序金币入账中挑 2 到 4 个做补充。
- 金币收益族的第一批生产候选更适合选“玩家需要做决定或构筑配合”的触发：指定点数、最大点数、牌型、出千实际重摇、临时小骰数量和削点换金币。稳定经济类先降低优先级。
- 九颗都保留后，生产包应按风险拆批而不是按重要性筛掉：高风险组为 `复利骰 / 收账骰 / 柜台骰 / 伐木骰`，中风险组为 `牌税骰 / 壳税骰 / 铅票骰`，基础触发组为 `悬赏骰 / 顶金骰`。
- 指定点数候选必须从本骰当前面组中抽取或选择，不能出现本骰永远掷不到的目标点。若面组有重复点，目标可以按独立点数池抽取，避免重复面导致目标概率和 UI 解释混乱。
- 牌型给金币应读取本次最终牌型结论，不新增小牌型表。首版可按三同、顺子、四同以上、全奇和全偶等现有结果分档，且每颗牌型金币骰每次出手只结算一次。
- 回合结束给金币适合作为保底或低额经济骰，不适合作为高收益核心；否则玩家无须构筑触发条件也能稳定滚经济。
- 利息加倍应只发生在通关结算阶段，并限制同类叠加。首版更安全的表达是复制一部分利息或提高利息上限，而不是无限乘法复利。
- 出千给金币只认确认出千后本骰被选中并实际重摇；进入出千界面、选中后取消、未被选择或结果未重摇都不触发。
- 临时小骰数量给金币应读取真实结算生成的小骰数量，预览只显示估算；临时小骰不反向触发金币骰，避免龟龟与金币闭环。
- 槽位触发优先做相邻读取或前序读取，而不是固定站位无条件触发。当前结算顺序已经是玩家可调整的左到右槽位，适合把顺序变成轻策略。
- 伐木类削点换金币首版建议削本次单骰分，不改变有效点数或主牌型识别。真正削有效点数的版本会同时成为控点骰，应该作为高阶候选另行评审。
- 与龟龟、大树、奇偶的交融可以先通过金币触发条件实现，例如“本次生成临时小骰达到一定数量给金币”“本次有大树成长给金币”“本骰有同点伙伴给金币”，而不是另写一批复杂桥接规则。
- 金币与大树的融合重点是交接形式，不是金币骰自身节奏。不要把规则写成“成长一次给 1 金”。
- 更好的交接是：大树成长形成某种资产状态，金币玩法在市场、利息、订单、抵押或阶段结算中读取这个状态。
- 可以使用中间对象降低硬绑定感：果实不是临时小骰，而是可售资源或订单目标；树契不是直接奖励，而是提前约定的兑现条件；估值不是金币收入，而是影响利息、折扣或抵押额度。
- 大树玩法仍然负责“树变强”，金币玩法负责“强树变值钱”。两者之间隔一层资产或市场动作，融合会更自然。
- 龟龟类可以使用“额外骰子收益族”作为底层统一收益，但不能停留在泛用补骰。龟龟的首要身份应是连锁引擎和积分表演核心。
- 龟龟类触发条件建议分四档：低点或首投等稳定小补，同点或顺子等构筑触发，出千或金币支付等成本触发，前序临时骰等连锁触发。
- 临时小骰质量必须随触发难度分层。稳定触发只给 d1 或 d2；构筑触发给 d2 或 d3；有成本触发可以给 d3 或短链；连锁触发必须限次。
- 壳匠类不应默认和所有新龟龟生成源绑定。更安全的做法是把壳匠归为“临时骰数量兑现族”，再单独决定它读取全部临时小骰，还是只读取龟系临时小骰。
- 首批龟龟候选不应平均铺开触发条件，而应保证完整连锁结构：至少一个高质量链源、一个续链、一个分叉、一个回声和一个兑现。
- 龟龟积分表演应成为验收标准的一部分。只实现生成小骰但没有逐颗跳涨、链条展开、终点爆点和折叠余波，不算达到龟龟路线目标。
- 龟龟和大树都应是爽点来源：大树偏长期资产和成长爆发，龟龟偏当回合连锁和结算爆发。
- 第 21 轮的“双层能力”只能解决死件问题，但仍会诱导追多龟龟。最新方向应把龟龟从“自成连锁”改成“其它骰收益放大器”。
- 龟龟候选的主要对象应是非龟骰或其它 build 的成功事件，例如同点成立、顺子成立、高面命中、金币支付、出千重摇、赌徒高点。
- 龟龟不应复制对方完整效果，只产生小骰余波。这样既能放大收益，又避免把其它路线的核心机制偷走。
- 龟龟之间默认不互相读取，临时小骰也不触发龟龟。龟龟连锁表演来自其它骰成功后的余波展开，而不是龟龟互相套娃。
- 龟龟积分表演仍然重要：当被寄托的骰子成功时，龟龟在其后展开小骰链，让该 build 的成功多一段“滚分尾音”。
- 后续具体候选应先按“触发条件、收益类型、机制接口”拆分，再决定它叫金币骰、龟龟骰、大树骰还是其它骰；类型名称服务阅读，不再决定唯一 build 归属。
- 龟龟首批机制优先使用相邻读取和现有牌型读取。它们能让玩家在市场和投骰前产生清楚决策：买哪颗龟、放在哪个槽、是否为了触发相邻余波而调整出千目标。
- 每颗新龟龟首版建议每次真实结算最多触发 1 次；稳定触发给 d1 或 d2，构筑触发给 d2，出千或高面这类成本 / 低频触发给 d3。首批不做分叉、回声和二次触发；若使用 dN 作为壳链种子，只沿基础递减规则展开，并设置展示和链深上限。
- 如果采用“最大值逐级衰减”，规则卡必须区分 `生成 dN 临时小骰` 和 `启动 dN 壳链`：前者只生成一颗，后者展开为 `dN -> d(N-1) -> ... -> d1`。
- `dN` 只表示一颗临时小骰的点数范围，不表示生成 N 颗。`d1` 固定 1 点，`d2` 为 1 到 2 点，`d3` 为 1 到 3 点。规则卡里可以用 `dN` 快速写，面向玩家的 UI 文案可再转成“小骰 1-2 点”等更直白表达。
- “壳链 / 龟龟余波链”默认生成龟系临时骰：它参与单骰分、结算表演和被数量兑现读取的候选池，但默认不参与主牌型、不继承材质词缀、不触发实体骰类型效果、不成为出千目标。任何例外都必须在规则卡里显式写出。
- 外援临时小骰默认不触发龟龟，也默认不自动计入旧壳匠；如果需要计入，应先把壳匠改造成更明确的“临时骰数量兑现族”，再单独定上限。
- 吸附龟龟草案应先作为 `V0.2` 龟龟的另一条分支验证，不直接删除当前壳链。壳链负责当回合结算表演，吸附负责长期宿主改造和六槽压缩，两者是否共存需要在纸面测试后确认。
- 吸附触发首版建议放在离开市场或出手结束后的成长阶段，不在当前槽位计分中途改变宿主面组，避免预览和真实结算出现难以解释的差异。
- 左邻吸附是最低实现成本方案：玩家把龟龟放在目标骰右侧，触发时绑定左侧最近的非龟实体骰。普通吸附默认不读取其它吸附龟；只有首领龟可以读取真实龟龟吸附事件，且复制事件不再触发复制。
- 吸附值建议使用来源骰当前六面平均值向下取整且最低为 1；这和恶魔吞吃的期望转移口径一致。若普通 `1/2/3/4/5/6` 龟龟直接给宿主全部面 `+3` 仍过强，后续可把基础龟壳值降为 2 或让幸运龟不再提供壳值。
- 宿主卖价只叠加真实吸附龟的固定卖价，不读取宿主成长溢价、基础壳层数、首领复制壳或呼朋唤友奖励，防止买入、吸附、卖出、刷新形成套利。
- 小小龟和磁力龟生成的基础壳只改宿主点面，不算真实龟龟吸附，不触发首领龟、呼朋唤友龟或其它读取“吸附龟数量”的效果。
- 最新确认覆盖上一轮安全限制：首版不设置宿主吸附上限、整袋吸附上限或同名叠加限制。强度、结算长度和信息量风险进入验收观察，不作为生产包的阻塞项。
- 金钱龟玩家侧短规则统一为：结算时，获得 1 金。
- 小小龟玩家侧短规则统一为：结算时，宿主吸附 1 颗基础骰子。
- 悬浮页必须展示宿主吸附列表、每个吸附龟效果、基础骰吸附层、吸附带来的点面变化和卖出时回收项。
- 黑帆魔蝠的强化方向已确认：每次刷新时吞吃 1 颗即将被替换的非空货架骰，不再只限本市场第一次付费刷新。
- 现版寄生魔龟与战利品龟退出推荐池；前者过于直白，后者状态统计偏复杂。
- 驮粮龟与贡猪继续保留为双阵营候选，具体数值和实现范围尚未确认。
- 黑市小鬼与血契船长进入双阵营保留池；具体叠层、负费用、购买监听和目标选择仍待规则卡阶段确认。
- 清仓猪、补给猪、共食魔龟和托底龟进入本次拟添加双阵营名单；核心方向确认，具体数值和事件边界尚未确认。
- 本次拟添加双阵营清单共 9 颗，暂不加入龟龟 / 海盗组合。

- `牌税骰` 是否只按现有最终牌型给金币，还是允许不同牌型有不同金币档位？档位越多，界面说明越重。
- `复利骰` 是复制已封顶利息，还是提高利息上限后再结算？这会明显影响金币经济强度。
- `柜台骰` 的槽位触发应读左侧相邻骰、右侧相邻骰、全部前序骰，还是指定槽位？不同方案对玩家换位操作的压力不同。
- `伐木骰` 削减的是本次单骰分、有效点数，还是长期面组？首版倾向只削本次单骰分，长期削面风险过高。

## Design Rationale

当前产品方向强调玩家通过骰子类型理解构筑，而不是在大量隐藏组合里猜规则。新增骰子种类如果只提供更高分数，会削弱市场选择和路线记忆；如果过度复杂，又会增加 UI 说明和结算理解负担。因此本轮讨论应优先围绕每条既有路线的内部缺口展开。先让猪猪、龟龟、奇数、偶数、大树和赌徒各自形成足够完整的路线，再做路线交织，能降低学习负担，并让后续桥接骰有真实可连接的对象。

当前单路线已经有若干可用组件后，新的设计重点应从“每条路线缺什么”推进到“路线之间如何传球”。如果每个 build 只在自身内部累积，玩家会很快把它看成独立装配表；交融骰的价值在于让结算出现可解释的链条，例如先由金币、同点、成长或临时小骰制造一个事件，再由另一颗骰把这个事件兑现。这样可以增强爽感，同时保持底层规则仍然来自玩家已经理解的骰子类型。

“触发条件 + 泛用收益”的双轴结构可以进一步降低成型门槛。玩家不再需要抽到某条路线的完整专属套件才开始成型；只要已有构筑能稳定触发某类条件，就可以拿任何读取该条件的金币骰、临时小骰骰、成长骰或点数控制骰。这会让市场选择从“我是不是缺某个固定组件”变成“我当前最容易触发什么，以及我缺哪种收益”。这个方向能解决龟龟类骰只服务龟龟 build、金币类骰只服务金币 build 的割裂问题。

已确认的泛用机制方向应作为后续候选筛选的上位标准。骰子名称可以继续保留流派风味，但规则不能只回答“它属于哪条路线”，而要回答“它读取什么事件、提供什么机制、谁能使用这个机制”。龟龟族在这个框架里最适合承担“临时小骰余波”和“结算连锁表演”，用其它构筑的成功事件作为链源，避免重新回到龟龟互相追龟龟的套装闭环。

## Role Notes

### Design

先为每条基础流派定义缺口，再设计候选骰。建议每条路线至少区分：资源入口、资源转换、路线放大、稳定/保底、终局 payoff。

### Art

后续若进入生产，每种骰子需要一个可读的类型图标和结果态短标签。视觉主题应延续明亮账本桌游风。

### UI/UX

新增骰子不能依赖长说明文本。运行时状态应尽量压缩为短标签、目标点、计数器或明确的点数变化。

### Dev

当前实现源头是 `Assets/Scripts/DiceKingDemo.cs`。正式实现前需要核对枚举、市场 CSV、保存字段、结算预览和出千逻辑。

| 牌型金币文本过重 | 如果每种牌型都有独立金币档位，玩家需要同时记牌型倍率和金币收益表 | 首版只读取最终牌型等级或少量大类，界面短标签显示本次触发结果 | 已识别 |

## Candidate Acceptance

- 每颗正式市场骰必须同时标明家族、获取阶级和一个主要构筑职责；同一颗骰可以响应多个事件，但市场补位只使用一个主职责，避免标签失控。
- `T1` 必须在没有同家族核心时也能立即产生可理解价值；`T2` 最多依赖一个清晰前置条件；`T3` 可以依赖成型基础，但不能成为路线通关所必需的唯一命名零件。
- 首次购买非基础骰后，后续两次市场看到至少一个同家族补强、双阵营桥接或缺失职责候选的目标概率应达到约 `70%`，同时保留转路线的全局候选。
- 每批三格货架至少有一个玩家当前可立即使用的候选；不能三格同时都是缺少前置条件的高阶核心。
- 候选充足时同批不得出现相同类型，相邻付费刷新不应原样重复整批；连续刷新保底必须记录真实 `T3` 是否出现。
- 若启用每局市场名录，名录必须在开局或首次进入市场时可查看，并保证任何被投放的主路线至少具备入口、转换 / 放大和兑现三个职责，不允许随机出一条残缺路线。
- 每个新增骰子能用一句话说明核心定位。
- 每个新增骰子至少服务一个明确路线或路线连接。
- 每个新增骰子的结算边界包含：是否改变有效点数、是否参与牌型、是否能被出千影响、是否有长期状态、是否进入市场。
- 第一批扩展不要求完整实现，只要求筛出值得进入生产包的候选。
- 每颗交融骰必须说明它连接的两个玩法面，以及玩家在市场或出千中因此多出的一个实际决策。
- 第一批交融骰的结算链条不超过两段，避免首版直接进入复杂连锁系统。
- 新版骰子候选必须同时写明“触发条件”和“收益类型”。
- 新版骰子候选还必须写明“机制接口”：读取谁的事件、在结算顺序中何时触发、是否能被其它收益族读取、是否有每次结算上限。
- 一颗骰子如果离开原本标签 build 后完全没有价值，就不符合新方向。
- 一颗骰子如果任何 build 都无脑购买，也不符合新方向。
- 龟龟候选如果不能给玩家带来至少一个市场、槽位或出千决策，不应进入首批。
- 龟龟候选如果需要解释“事件族”“成功事件”“结算接口”这类抽象词，首批应改写成更可见的触发条件。
- 规则卡中使用 `dN` 时必须同时记录玩家侧显示文案，避免玩家把 `d3` 误解为“生成 3 颗小骰”。
- 龟龟规则卡必须写明是否启动壳链、种子规格、递减规则、最大链深、是否计入壳匠类兑现、是否可被巢穴补骰、以及生成物在 UI 中是否显示为龟系临时骰。
- 金币收益族候选必须写明触发频率档位、单次金币上限、同类是否叠加、是否进入左到右钱包流，以及失败时是否清空收益。
- 涉及出千、临时小骰、槽位和削点的金币骰，必须额外写明预览和真实结算是否一致，避免玩家以为能在确认前锁定收益。
- “九颗都需要”是生产范围确认，不等于九颗都已具备可实现规则。生产包必须为每颗骰补齐规则卡，未补齐前不能进入程序实现。
- 奇偶泛用化候选必须写明它只影响牌型识别、单骰分、出千复核还是槽位兑现；默认不改变有效点数，除非生产包单独确认并重做牌型风险评估。
- 如果候选包含“牌型命中后让骰子成长、升级或保留加成”，默认应转入大树型骰子或成长型候选池，不作为奇偶首批机制。
- 大树泛用化候选必须写明触发事件、成长收益类型、是否写入长期面组、是否需要成熟阈值、是否可被金币或龟龟读取，以及每小关或每次结算的触发上限。
- 新增或重做骰子的触发条件默认应从七类白名单中选择：指定点数、指定牌型、最高点数、回合结束、金币、出千和槽位触发。白名单外触发必须单独说明为什么不能复用现有触发点。
- 大树重设候选必须先标明对应哪一类白名单触发，再写收益；不得先想收益再临时补触发。
- 大树候选不得把主要收益做成临时小骰；如果需要接入龟龟，应通过高面、成长完成或成熟节点作为龟龟读取事件，而不是由大树直接生成龟系临时骰。
- 奇偶首批候选的玩家侧市场文案必须压缩为一句短规则；如果需要解释设计术语，暂不进入首批。
- 奇偶结算标签优先控制在 2 到 4 个字加数字，例如 `夹缝 +8`、`补全`、`换相 +8`。
