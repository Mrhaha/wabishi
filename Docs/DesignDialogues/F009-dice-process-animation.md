## 当前摘要

F009 聚焦主流程中骰子的过程动画表现。最新方向已从骰盅、抛投和程序摇晃进一步收敛为“固定槽位内的序列帧旋转”：玩家出手前，六颗骰子在各自槽位显示类型身份；玩家按 `Space` 后，每颗骰子在原槽位播放旋转序列帧；随后从左到右逐步停下，显示真实骰面点数，再按槽位进行定点结算和跳分。表现层只读取当前投骰结果和结算事件，不改变骰面随机、`ScoreDice()`、出千、分数、存档或骰子规则。

## 原始用户意图

用户先提出后续需要更好的骰子动画表现，包括进入游戏和结算计算时的表现，并询问使用 Codex 应如何实现。随后用户进一步明确：想要的是骰子的摇晃、反转、过程表现等，不是简单形变。用户认可首版目标后，要求创建 F009 设计对话记录，首版目标是 2.5D / 3D 骰子入场、翻滚、开盅定格和结算点名；当前只沉淀范围和验收标准，不写代码。

## 问题框定

- 玩家问题：当前骰子如果只作为静态 UI 图片或简单抖动，玩家难以感到“真的把骰子掷出、等待结果、看到结果落定”，投骰前后的期待感和结算归因都偏弱。
- 生产问题：需要在不破坏现有 `F004` 投骰配置、`F006` 只读结算事件队列和 `F007` 积分塔的前提下，新增一层可验证的骰子过程表现。
- 当前系统张力：真实翻滚感很有吸引力，但如果让物理模拟决定骰面，会破坏当前构筑、随机、出千和结算边界；如果继续只用 OnGUI 变形，又达不到用户想要的过程动画。
- 目标体验：玩家进入小关时看到六颗骰子按槽位落桌；按 `Space` 起摇后看到骰盅和骰子有真实摇晃/反转暗示；开盅后骰子翻滚并定格到真实结果；结算时当前骰子被点名、轻抬或短翻，和左侧积分跳涨形成同一节拍。

## 讨论时间线

### 第 1 轮：提出后续骰子动画需求

- 用户方向：希望后续无论进入游戏还是结算计算，都有更好的骰子动画表现，并询问使用 Codex 如何实现。
- Codex 回应：建议沿用 Wabish 的生产节奏，先把动画表现拆成设计记录和生产包；入场 / 投骰表现衔接 `F004`，结算表现衔接 `F006 / F007`。
- 形成的洞察：骰子动画不应孤立实现，而应服务当前投骰状态机、固定槽位顺序和只读结算事件。
- 决策状态：倾向

### 第 2 轮：明确不是简单形变

- 用户方向：追问是否可以做出想要的动画表现，例如骰子的摇晃、反转和过程表现，而不是简单形变。
- Codex 回应：确认可以做到，但不应继续只靠当前 OnGUI 图片抖动；要达到真实翻滚感，需要新增 2.5D / 3D 骰子表现层或翻滚帧动画层。
- 形成的洞察：F009 的关键生产判断是“视觉真实翻滚”和“玩法真实结果”必须分离。动画可以表现得像实体骰，但最终点数必须对齐当前随机结果。
- 决策状态：已确认方向

### 第 3 轮：确认首版推进方式

- 用户方向：认可首版目标，并询问后续如何推进流程。
- Codex 回应：建议先确认 F009 首版范围，创建设计对话记录，再生成生产包，随后按小片实现入场、投骰、开盅、结算点名和路线小爆点。
- 形成的洞察：首版不应追求最终美术和完整物理系统，而应先验证骰子是否有“实体过程感”和是否能稳定对齐真实结果。
- 决策状态：已确认流程

### 第 4 轮：创建设计对话记录

- 用户方向：要求创建 F009 骰子过程动画表现的设计对话记录，首版目标是 2.5D / 3D 骰子入场、翻滚、开盅定格和结算点名，先不写代码，先沉淀范围和验收标准。
- Codex 回应：新增本设计对话记录，明确首版范围、非范围、候选实现路径、职能备注、风险和候选验收。
- 形成的洞察：F009 已具备生成生产包的基础，但生产包前仍可补一轮技术路线选择：程序 3D 方块、2.5D 翻滚帧动画或混合方案。
- 决策状态：可生成生产包

### 第 5 轮：首版实装观感回炉

- 用户方向：反馈当前动态表现太丑、特别混乱，要求先给出整体优化思路，重点考虑美感、游戏节奏和对玩家的刺激。
- Codex 回应：判断问题不是单一动画参数，而是缺少动画导演层；下一版不应继续把骰盅、骰子、点数和结算效果同时推到前景，而应按阶段只保留一个主焦点。
- 形成的洞察：F009 当前程序 2.5D 方块只能作为技术验证，不足以承担最终美感。首版回炉应优先重建节奏和视觉层级，再决定是否继续使用程序方块、2.5D 帧动画或正式骰子资源。
- 决策状态：待用户确认

### 第 6 轮：取消骰盅阶段的方向

- 用户方向：提到云顶之弈一类更直接的骰子表现，并明确希望取消骰盅阶段。
- Codex 回应：认可这会显著降低画面混乱度，并建议把 F009 主路线从“遮挡蓄势开盅”改成“无盅直投、短滚动、落点定格、结算点名”。
- 形成的洞察：骰盅虽然能制造遮挡和悬念，但在当前主流程里会与骰子本体、点数和结算反馈争夺视觉焦点。对 `骰子王` 这种高频投骰游戏，更适合让骰子本身承担动作主角，用短促滚动和落点停顿制造刺激。
- 决策状态：倾向

### 第 7 轮：固定槽位原地立体翻转

- 用户方向：进一步明确不需要抛投概念，希望骰子固定在槽位中；玩家出手时骰子就在原地旋转，旋转要有立体翻转感觉，不是平面旋转；结束后停下并显示骰面点数，然后开始定点结算。
- Codex 回应：确认该方向更适合当前游戏，因为槽位顺序本身承载出千、结算、金币流和构筑归因；动画不应让骰子离开槽位或制造位置混乱。
- 形成的洞察：F009 的首版主表现应从“真实投掷模拟”转为“槽位内实体骰翻面”。重点不是让骰子在桌面上跑，而是在固定位置表现出厚度、转轴、侧面、阴影和最终吸附。
- 决策状态：倾向

### 第 8 轮：序列帧旋转播放器

- 用户方向：明确当前需要的是序列帧动画，而不是在骰子上继续做摇晃动作；每回合开始各槽位显示各类骰子的样子，敲击 `Space` 后每颗骰子播放旋转序列帧，随后缓慢停下，从左到右逐步停止，再依次开始跳分。
- Codex 回应：将 F009 首版实现方向从“程序方块连续形变”改为“OnGUI 序列帧播放器”：运动由离散帧号驱动，骰子不离槽，不抛投，不做横纵摇晃；加力窗口结束后锁定真实结果；`Stopping` 阶段按槽位从左到右停住并逐颗显点；结算跳分继续复用 F006 / F007 的左到右提交态。
- 形成的洞察：对当前主流程来说，最关键的是“稳定槽位 + 类型可读 + 帧动画节奏 + 左到右停转 / 跳分”。序列帧比程序摇晃更容易做出可控节奏，也更方便后续替换为正式美术帧图。
- 决策状态：已执行首版

## 探索过的方案

| 方案 | 核心思路 | 优点 | 风险 | 状态 |
|---|---|---|---|---|
| 继续 OnGUI 位移 / 缩放 / 旋转 | 在当前骰子 UI 上加强抖动、位移、浮字和缩放 | 成本最低，可复用现有绘制 | 很难形成真实翻滚和实体过程感，容易继续像 UI 抖动 | 已拒绝作为首版主方向 |
| 2D 翻滚帧动画 | 使用多帧序列表现骰子翻滚、弹跳和翻面 | 比完整 3D 轻，容易控制最终画面 | 需要准备多套帧或程序生成帧；不同骰子类型、材质和点数适配成本可能上升 | 候选 |
| 2.5D / 3D 骰子表现层 | 在主流程上叠加独立骰子视觉层，使用方块旋转、弹跳、阴影和最终面定格 | 最接近用户想要的摇晃、反转和过程表现；最终点数可强制对齐真实结果 | 需要处理与当前 OnGUI 主界面的叠加、遮挡、相机和资源 | 已确认首版方向 |
| 固定槽位序列帧旋转 | 骰子不离开槽位，按离散序列帧表现正面、侧面、背面、压缩面和停转显点 | 最清楚保留槽位顺序和结算归因，节奏可控，适合高频投骰和后续替换正式帧图 | 如果帧图或程序帧画得粗糙，会显得像临时占位 | 最新执行方向 |
| 物理模拟决定骰面 | 让刚体骰子真实碰撞并由物理结果决定点数 | 观感真实 | 会破坏当前随机、出千、构筑和结算可控边界，调试困难 | 已拒绝 |
| 全面重做主流程为 3D 桌面 | 将整个主游戏场景替换为 3D 桌面和空间 UI | 上限最高 | 范围过大，会拖慢当前机制验证和 UI 迭代 | 已后置 |

## 已确认决策

- 本案编号为 `F009`。
- 首版目标是 2.5D / 3D 骰子过程动画表现，不是继续只做简单 UI 形变。
- 首版关注三个玩家可见时刻：槽位内翻转、停转显点、结算点名。
- 当前阶段只创建设计对话记录，不写代码，不改数据，不改主流程文档。
- F009 首版不改变骰面随机、概率、出千、结算、目标分、金币、存档或骰子效果。
- 骰子动画最终必须对齐已经确定的真实骰面结果；动画表现不能成为新的规则来源。
- 结算点名应读取 `F006 / F007` 既有结算展示思路：真实结算先冻结，动画只播放已提交状态。

## 倾向方向

- 新增独立的骰子视觉表现层，例如 `DiceVisualController` 方向；它消费当前骰袋、投骰阶段、最终骰面和结算展示事件，而不是拥有玩法规则。
- 首版优先验证“固定槽位内的立体翻面”是否成立，再决定是否补最终美术资源。
- 入场阶段：进入小关时，六颗实体骰可以直接出现在固定槽位，或用很短的淡入 / 轻落位表达，不再强调抛入桌面。
- `Ready` 阶段：骰子按固定槽位轻微待机，保留类型身份和槽位顺序，允许后续槽位调整表达。
- `Rolling` 阶段：骰子不离开槽位，只在自身格内做原地立体翻转；运动面不显示可读点数。
- `Stopping` 阶段：骰子按轻微错峰停止，最终正面吸附到真实点数，顶部点数和状态标签必须与真实结果一致。
- `ResultDecision` 阶段：骰子稳定停在槽位上，点数、类型色条、材质和短状态标签清晰可读。
- `Scoring` 阶段：当前结算骰轻抬、短翻或盖章点名，和左侧基础分、倍率、当前积分跳涨保持同一节拍。
- 龟龟、金币、大树和倍率等路线小爆点可继续使用 F006 的事件队列扩展，不在 F009 首版里强行一次做完所有专属特效。
- F009 后续生产包应把“程序验证层”和“美术最终资源层”拆开，避免首版卡在资产制作。

## 回炉优化方向：固定槽位、原地立体翻转、显点、点名

当前回炉方向调整为取消骰盅和抛投概念，把一次投骰拆成四个明确节拍。骰子始终留在各自槽位内，动画只改变自身朝向、厚度透视、阴影和强调状态：

1. `起转`：按下 `Space` 后，六颗骰子在原槽位同时或轻微错峰启动立体翻转，给出“这一手开始了”的明确反馈。
2. `旋转`：骰子围绕横轴、纵轴或斜轴做槽位内翻面，显示顶面、侧面和阴影变化；运动面不显示可读点数，只保留空白面、类型色或背面。
3. `停转显点`：骰子轻微错峰减速，最终正面吸附到真实点数，并给一个短暂停顿让玩家读牌型。
4. `定点结算`：结算阶段一次只强调当前入账槽位，其它骰子降噪，当前骰轻抬、描边、短闪或盖章，并和左侧计分跳涨同步。

### 美感原则

- 主视觉应更接近“明亮账本桌游风的实体棋子”，少用程序立方体硬扭曲和点数贴面拉伸。
- 点数只在可读阶段出现；运动阶段可以显示空白面、类型色、剪影或背面，不显示会乱飘的 pip。
- 每个阶段只允许一个强运动焦点：起转期看整组槽位启动，旋转期看槽位内立体翻面，结果期看点数，结算期看当前点名骰。
- 六颗骰子不应离开槽位高速乱跑；最多用错峰、翻面节拍和短吸附来制造实体感。
- 类型身份应靠小角标、色条、图标和结算标签保底，不要压在点数上方抢读数。
- 骰盅不再作为首版默认阶段；如保留，只能作为背景道具、菜单装饰或极少数特殊事件表现。
- 原地旋转必须表现出厚度：至少要有顶面、侧面、受光变化和接触阴影，不接受单张骰面图片绕中心平面旋转。

### 节奏建议

- 普通投骰从第一次按 `Space` 到结果可读，目标控制在约 `1.5` 到 `2.0` 秒。
- 加力窗口约 `0.6` 到 `0.9` 秒，主要制造旋转速度、桌面反馈和输入反馈。
- 取消开盅等待和落点等待，改为停转前约 `0.10` 到 `0.15` 秒的“最后半圈”停顿，让玩家产生“马上停住”的期待。
- 结果揭示约 `0.35` 到 `0.55` 秒；六骰按两波或从左到右小错峰停转显点。
- 结果稳定后至少保留约 `0.25` 到 `0.4` 秒安静阅读时间，再进入结算点名。
- 结算点名默认每颗约 `0.15` 到 `0.22` 秒；爆发牌型、通关、金币大入账或成长入档可以额外加短促强调，但普通手不加烟花。

### 玩家刺激来源

- 刺激不来自“所有骰子都乱动”，而来自“原地起转、速度提升、最后半圈、真实点数亮出、命中反馈”的递进。
- 玩家敲 `Space` 应得到明确反馈：旋转速度、桌面震动、短音效和提示脉冲增强，但这些反馈不影响随机结果。
- 稀有或高价值结果才放大：四同以上、顺子、全奇 / 全偶、通关、接近失败时贿赂救场、金币大入账、大树成长入档、龟龟壳链长尾。
- 类型专属表现放在“结果 / 结算”阶段更清晰：龟龟走壳链余波，金币走金币飞向钱包，大树走成长脉冲，奇偶 / 盖章走几何校准或盖章，赌徒 / 双倍走短促爆点。
- 表现必须帮助玩家归因到构筑和槽位，而不是只提供一段与规则无关的表演。

### 技术倾向修正

- 当前程序 2.5D 方块不再建议作为最终默认表现，只保留为规则对齐和回退验证层。
- 下一版首选“动画导演层 + 2.5D 实体骰资源 / 帧动画 / 简化 3D 渲染”的混合路线。
- 表现层应显式维护阶段：`StageEnter / SlotSpinStart / SlotSpinLoop / SlotSpinStop / ResultHold / SettlementCallout`。
- 所有时长、错峰、缓动、强调等级应配置化，继续保持只读真实结果，不调用或替代 `ScoreDice()`。
- 若暂时不补最终美术资源，也应先替换掉生硬的 pip 贴面和程序方块扭曲，至少用更规整的骰面底图、阴影、空白运动面和结果吸附来保证观感。

### 回炉验收口径

- 任意截图中，玩家能立刻判断当前阶段是起转、加力、停转显点、看结果还是结算某颗骰。
- 首版默认没有骰盅阶段；如果画面出现骰盅，只能是弱背景道具，不承担主运动焦点。
- `Shaking` / `Rolling` 期间没有完整点数和高强度类型标签同时抢焦点。
- 投骰期间骰子不能离开自身槽位，不能打乱玩家对槽位顺序和后续结算顺序的理解。
- 旋转必须有立体感：能看到顶面 / 侧面 / 阴影变化，不能只是平面图片旋转。
- 结果点数在稳定后 `0.3` 秒内可读；类型标记不遮挡点数。
- 普通投骰节奏不拖慢高频游玩，玩家不需要等待一段与结果无关的长表演。
- 特效强弱和真实收益相关：普通结果安静，高价值结果才有明显刺激。
- 结算点名必须让玩家看懂“是哪颗骰、为什么加分 / 入账 / 成长 / 触发路线”。

### 不建议继续投入的方向

- 不建议继续主要调参当前 OnGUI 程序方块的摇晃和翻转；它容易继续表现为粗糙形变。
- 不建议继续把骰盅作为首版主阶段；它会延长流程并重新制造遮挡焦点。
- 不建议做抛投、位移滚动或碰撞模拟；它会削弱槽位顺序和结算归因。
- 不建议在旋转阶段显示六颗带完整点数的骰子同时翻滚；这会直接制造视觉噪声。
- 不建议让每种骰子在投骰阶段都做大幅专属旋转；类型差异应主要在结算阶段兑现。
- 不建议把普通一手做成长镜头；投骰是高频动作，刺激必须短、准、可跳过或可加速。

## 已拒绝 / 已后置想法

- 拒绝让物理模拟直接决定骰面点数；真实结果仍由当前随机和出千规则决定。
- 后置完整 3D 桌面重做、完整物理碰撞系统、复杂镜头运镜和全套音效混音。
- 后置每个骰子类型的专属翻滚动画；首版只要求类型身份不丢失。
- 后置最终生产级骰子模型和材质；首版可先用程序方块或临时资源验证节奏。
- 不新增长篇教程解释动画；玩家应通过短提示、槽位稳定和结果一致性自然理解。

## 待回答问题

- 首版技术路线选程序 3D 方块、2D 翻滚帧动画，还是两者混合。
- 3D 表现层如何与当前 OnGUI 主流程叠加：独立相机、渲染纹理、世界空间对象，还是先在主场景中直接摆放。
- 骰子类型身份在 3D / 2.5D 表现中如何呈现：类型色条、角标、顶部小图标、材质层，还是结算时回到现有 2D 标签。
- 首版是否要覆盖出千重摇的独立翻滚动画，还是先只覆盖自然投骰和结算点名。
- 首版是否包含音效、桌面轻震和 `Space` 加速结算，还是全部后置到下一轮手感调优。
- 是否需要把 F004 的投骰表现配置扩展为通用动画配置，还是为 F009 新增独立配置文件。
- 普通手和爆发手的动画时长目标需要在生产包中确认；普通投骰不应拖慢高频游玩。

## 设计理由

`骰子王` 的核心不是单纯赌博，而是玩家把随机结果转化为可控计分系统。更强的骰子过程动画应放大这个核心：玩家先安排槽位，按下 `Space` 触发投骰，看见骰子作为真实桌面物件运动，再看到结果稳定落入既定槽位，最后由结算点名把高分归因到构筑、槽位和路线触发。表现层越强，越需要保持玩法边界干净；否则动画会从“放大决策”滑向“替代决策”。

F009 因此不应把重心放在炫技物理模拟，而应先解决三件事：骰子像实体物件、结果可读且可信、结算归因更清楚。只要这三点成立，后续美术、音效和专属特效都能在同一表现层上迭代。

## 职能备注

### 策划

F009 不新增机制收益，不改变玩家策略。策划重点是确认动画服务哪些真实决策：槽位安排、出千选择、路线触发、倍率成立和目标达成。验收时应关注玩家能否更清楚地感知“我为什么得到这个结果”，而不是只评价动画是否热闹。

### 美术

首版可接受程序临时骰子，但方向应贴合明亮账本桌游风：干净轮廓、清楚点数、柔和阴影、轻微弹跳和桌面物件质感。正式资产后续再决定是否需要倒角骰子模型、类型图标贴花、材质层、桌面投影或极少数特殊事件装饰。不要用过度写实、赌场感或厚重暗色风格。

### 界面体验

动画不能遮挡目标分、当前分、剩余出手、出千和结算主提示。主焦点应按阶段转移：入场看槽位，投骰看固定槽位内立体翻转，停转看最终点数，结算看当前点名骰和左侧积分跳涨。首版应验证 `1280x720` 和 `1920x1080` 两个尺度下没有重要文字或骰面被遮住。

### 程序

建议把 F009 做成只读表现层。它可以读取 `rollPhase`、当前实体骰列表、最终骰面、槽位顺序、`scoringDice` 和 `SettlementDisplayEvent`，但不能调用或替代 `ScoreDice()`，不能重复随机骰面，不能重复写钱包，不能重复应用大树成长或临时小骰。若 3D 表现层加载失败或关闭，游戏应回退到当前 2D 结果展示，保证原型仍可玩。

### 跨职能

F009 的第一轮验收必须包含录屏或 Play Mode 实看。静态检查只能证明文件和结构没坏，不能证明翻滚节奏、遮挡、最终点数一致性和结算点名是否有效。

## 风险登记

| 风险 | 重要原因 | 可能缓解方式 | 状态 |
|---|---|---|---|
| 动画结果和真实结果不一致 | 玩家会失去对随机和结算的信任 | 最终帧强制吸附到真实骰面；动画只读已确定结果 | 已识别 |
| 物理系统侵入玩法 | 让物理决定点数会破坏构筑、出千和可复现调试 | 物理只作为视觉模拟，不参与规则 | 已识别 |
| 实现范围失控 | 3D 骰子、相机、UI 叠加、材质和音效容易一起膨胀 | 首版只做程序验证层和四个关键时刻 | 已识别 |
| OnGUI 与 3D 叠加冲突 | 当前主流程仍是 OnGUI，3D 表现可能遮挡或坐标不一致 | 生产包中先确认相机和坐标方案，必要时保留 2D 回退 | 已识别 |
| 动画拖慢高频操作 | 投骰是高频动作，过长动画会消耗耐心 | 普通投骰控制短节奏，爆发手才允许更长停顿；后续考虑加速 | 已识别 |
| 骰子类型身份丢失 | 3D 方块只显示点数会削弱类型可读性 | 使用类型色条、角标或结算标签保底 | 已识别 |
| 表现变成纯表演 | 动画好看但不强化决策，可能掩盖玩法密度问题 | 动画节点绑定槽位、出千、路线触发和结算归因 | 已识别 |

## 候选验收

- 进入小关时，六颗实体骰按当前槽位落桌，玩家能看出它们不是静态弹出。
- `Ready` 阶段骰子槽位稳定，后续槽位调整或结算顺序不会被动画打乱。
- 第一次按 `Space` 后进入投骰动画，骰子在固定槽位内有可见立体翻转过程。
- 加力窗口内连打只增强表现强度，不改变骰面概率、出手次数、出千次数、金币或分数。
- 停转后骰子必须定格到真实结果；顶部点数、短状态标签和结果预览一致。
- 结果锁定后仍保持投骰前槽位顺序，不按点数重排。
- 结算阶段当前骰子有明确点名动作，例如轻抬、短翻、发光边框或盖章，并和左侧计分跳涨同步。
- 出千重摇如果纳入首版，重摇骰只播放被选实体骰的翻滚，未选骰保持槽位和结果。
- 禁用或加载失败 F009 表现层时，当前主流程仍能用现有 2D 展示完成投骰、出千和结算。
- Play Mode 录屏中普通一手不显著拖慢节奏；玩家能说清楚骰子何时入场、何时停转显点、何时结算点名。
- 录屏或截图验收覆盖至少 `1280x720` 和 `1920x1080`，确认骰面、目标分、当前分和主提示不互相遮挡。
