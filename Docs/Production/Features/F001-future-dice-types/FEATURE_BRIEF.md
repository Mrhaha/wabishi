## 功能意图

F001 把编号 1 的设计对话沉淀为后续可继续细化、拆分和实现的生产包。F001 本身不是直接实现包，而是新增骰候选与规则边界的伞形规划包。

本包的生产目标是：

- 保存已确认的路线方向和候选池。
- 定义每颗新骰进入生产前必须补齐的规则卡。
- 把后续工作拆成可审查的执行切片。
- 阻止未批准候选直接进入美术、界面体验或程序实作。

第一批扩展优先完善已有基础流派：金币、龟龟、奇数、偶数和大树。路线桥接骰、全新高阶路线和复杂跨系统机制暂时后置。

## 玩家体验

玩家在中后期市场中不只是购买更贵的直接加分骰，而是能围绕当前骰袋选择更明确的路线组件：

- 金币流从“猪猪赚钱”扩展为“本金、消费和延迟收益如何转换成战力”。
- 龟龟流从“生成临时小骰”扩展为“链长、链质、数量和链尾收益”。
- 奇数流从“全奇概率”扩展为“追同点牌型和同点收益”。
- 偶数流从“全偶概率”扩展为“更容易形成顺子或偶数专属连线”。
- 大树流从“单颗成长”扩展为“命中、成长保护、成熟兑现和新树培养”。

玩家应能用一句话理解每颗新增骰为什么存在，并在市场中判断它是否服务当前构筑。

## 已确认规则

- F001 是新增骰候选与规则边界确认包，不是直接实现包。
- 本轮新增骰优先补齐各基础流派内部结构，再讨论流派交织。
- 奇数流派优先围绕相同点数牌型做文章。
- 偶数流派优先围绕顺子做文章，并可探索“差 2 的数字也能构成顺子”的偶数专属表达。
- 金币流第一批候选国库骰、贿赂骰、投资骰已经完成 A-gold 切片，并由用户确认已实现验收。
- 龟龟流第一批候选壳匠骰、巢穴骰、慢龟骰已经完成 F001-B 切片，并由用户确认已实现验收。
- 壳匠方向优先按临时小骰数量给分，贴合龟龟链式爆发。
- 大树流第一批候选为园丁骰、灌溉骰、年轮骰、树冠骰、苗圃骰。
- 苗圃骰方向采用“新入/新买入的大树骰初始成长层数 +1”。
- 每颗候选骰后续都必须补齐：触发时机、结算边界、出千交互、市场定位、界面标签和反滥用边界。

## 候选成熟度阶梯

## 规则卡模板

每颗候选骰进入 `规则草案` 前必须补齐：

| 字段 | 必须回答 |
|---|---|
| 一句话身份 | 玩家一句话理解这颗骰为什么存在 |
| 路线职责 | 服务哪条路线，补哪个缺口 |
| 触发时机 | 投骰开始、结果锁定、预览、结算、通关、市场或跨小关 |
| 效果公式 | 明确公式、上限、随机范围和取整规则 |
| 影响对象 | 是否改变有效点数、单骰分、牌型、钱包金币、本关账本、成长或临时小骰 |
| 预览 / 结算 | 预览和真实结算是否一致；随机结果如何标记预估 |
| 出千交互 | 出千是否影响结果、目标、预算、成长或延迟状态 |
| 持久状态 | 是否需要跨小关状态和存档字段 |
| 市场配置 | 阶级、买价、卖价、章节权重和是否可出售 |
| 运行界面 | 市场标签、未投掷标签、结果标签、预览行、结算反馈、日志兜底 |
| 反滥用边界 | 防闭环、防刷分、防套利、防界面过载规则 |
| 验收检查 | 可观察的通过 / 失败标准 |

## 建议默认值

- 下一步先对 `F001-D-tree-dice` 执行 `$wabish-packet-review`，审查成长收益范围、灌溉方案、苗圃市场边界和存档形状。
- 第一批新骰默认不修改全局基础牌型表；路线专属收益可以在单颗骰规则中表达。
- 龟龟系临时小骰默认只参与本次结算，不进入骰袋、不保存、不触发其它实体骰长期状态。
- 最终图标和运行资源只在单颗骰达到 `用户确认` 后开始。
- 新骰 CSV 行只在代码枚举、解析、显示、保存和市场逻辑支持后加入运行数据。

## 阻塞项

- F001-D 大树子切片仍缺少用户确认后的成长收益范围、灌溉方案和苗圃市场边界。
- 任何会改全局牌型表或有效点数识别的规则，需要单独确认后才能实现。

## 约束

- 当前骰袋容量为 6，所有实体骰默认上场；新增骰直接影响牌型概率和出千价值。
- 当前结算公式以实体骰主牌型倍率为核心，临时小骰默认只追加单骰分，不参与主牌型。
- 出千只重摇实体骰骰面，不刷新猪猪目标、大树命中点、赌徒阈值、国库快照或投资预算。
- 本关账本金币不能在同一次结算中直接消费成战力。
- 新骰子不能依赖长说明文本，运行态信息应压缩为短标签、目标点、计数器或清晰的点数/金币变化。

## 非目标 / 后置范围

- 不在 F001 伞形包中直接实现代码、接入 CSV、生成最终图标或修改运行时 UI。
- 不把路线桥接骰或全新高阶路线提前纳入第一批实现。
- 不把偶数差 2 顺子直接写入全局牌型表，除非后续明确批准。
- 不在规则未细化前修改 `Assets/Scripts/DiceKingDemo.cs`。
- 不把候选图标登记为最终运行资产。

## 职能交接

### 美术

当前只允许维护候选视觉动机和目标路径。为进入用户确认的单颗骰准备类型图标，风格延续明亮账本桌游风。未定稿骰子不要生成最终资源。

### 界面体验

为每颗候选骰定义最短运行态表达：市场短建议、未投掷身份、结果标签、结算反馈和可能的手牌参考补充。每颗骰达到规则草案时，应填写对应子包的界面体验规格。

### 程序

等待单颗骰进入程序就绪后再实现。实现前必须检查 `DiceKingDemo.cs` 的枚举、保存/读取、市场 CSV、默认面组、结算预览、真实结算、出千和图标加载。

## 不要改动

- 不改变当前首章默认开局：6 颗基础骰、通过关间市场获得其它骰子。
- 不改变当前小关、市场、失败、存档恢复主流程，除非子切片明确批准。
- 不允许本关账本金币在同一次结算中直接消费成战力。
- 不允许候选规则绕过规则卡、用户确认和角色门禁。

## 验收含义
