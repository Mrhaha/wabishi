## 功能意图

F001-A 把金币流第一批候选 `国库骰 / 贿赂骰 / 投资骰` 从候选名推进到已确认规则包。

本切片的目标仍然不是立刻实现代码，而是把金币如何转为战力写成可执行合同：

- `国库骰`：奖励持有本金，不花钱。
- `贿赂骰`：在差一点通关时花实时可用钱包金币补最终分。
- `投资骰`：小关开始自动锁定并预付预算，换取本小关持续加分。

这三颗骰必须遵守同一条硬边界：不消费、不读取同一小关内由猪猪等来源新产生的本关账本金币。

## 玩家体验

玩家进入中期后，金币不再只服务市场购买，也会成为一条清晰的战力路线：

- 如果玩家选择囤钱，国库骰让本金本身变成稳定分数。
- 如果玩家选择保留应急金，贿赂骰能把差一点的失败结果买成通关。
- 如果玩家愿意提前下注，投资骰会在小关开始自动锁定预算，并把预算转成本关内持续单骰加分。

三颗骰都应让玩家在市场里做出可理解的经济判断，而不是只看“更贵所以更强”。投资自动预算必须在市场文案、小关开始反馈和顶部钱包变化中明确展示。

## 已确认规则

- 金币流第一批候选为国库骰、贿赂骰、投资骰。
- 本切片只产出规则和职能交接，不修改 `Assets/Scripts/DiceKingDemo.cs`。
- 本切片不生成最终图标，不新增 `dice_market_config.csv` 行。
- 三颗骰都不能消费或读取同一小关内刚进入本关账本的金币。
- 三颗骰都不改变有效点数，不参与制造新牌型，不修改全局手牌识别。
- 小关开始先记录钱包快照，再锁定投资预算；国库读取原始快照，贿赂读取投资后的可用钱包。
- 国库骰没有单颗加分上限，按小关开始钱包每 10 金稳定增长。
- 投资预算自动锁定并视为预付扣除，失败或通关后都不返还。

## 建议默认值

| 骰子 | 已确认默认行为 | 设计后果 |
|---|---|---|
| 国库骰 | 读取小关开始钱包，`floor(金币 / 10)` 给单骰加分，无单颗上限。 | 鼓励囤钱，和市场消费形成取舍，不受本关猪猪收益影响。 |
| 贿赂骰 | 只在可补到通关时，玩家确认结算后自动花可用钱包金币补最终分；每金 +4 最终分，每颗每次最多 2 金。 | 形成救急骰，不增加额外按钮，但必须预览清楚。 |
| 投资骰 | 小关开始自动锁定并预付预算，预算公式为 `min(投资骰数量 * 2, floor(小关开始钱包 / 3))`；每金让本骰每次结算 +2 单骰分。 | 形成提前下注路线，和贿赂的事后救急形成差异。 |

## 约束

- 当前运行时只有 `Basic / Piggy / Turtle / Double / Odd / Even / Tree / Gambler` 八种 `DieType`。
- 当前 `DiceData` 只保存名称、类型序号、面组和成长层数；新增类型实现前必须升级读取边界。
- 当前市场数据通过 `dice_market_config.csv` 读取，新增骰数据行必须等代码支持后再加入。
- 当前小关失败直接结束本轮，投资预算失败损失不需要跨失败流程保留。
- 当前继续游戏恢复到小关开始，阶段临时预算可以在进入小关时重新计算；如果之后支持小关中途保存，需要补字段。
