## 执行顺序

1. 使用 `$wabish-art-production` 产出五材质视觉方向和资源；如程序先行，可先用材质名短标签回退。
2. 使用 `$wabish-dev-implementation` 实现市场生成、材质数据、结算接入、运行记录清空、新存档字段和文档同步。
3. 界面体验按 `UI_UX_SPEC.md` 检查商品卡、卖出区、结果反馈和出千反馈。
4. 执行 Play Mode、运行记录清空、市场刷新、结算和 UI 截图验收。
5. 根据刷新样本和结算样例决定后续是否启用面组和材质组合加价。

## 建议默认值

- 每颗新生成的商店骰必带一种材质。
- 当前没有玩家，F002 实现允许清空现有运行记录；不为旧 `DiceData` 做长期兼容包袱。
- 随机点面使用模板池，不使用完全自由随机。
- 骰子类型是第一优先级：奇数族只生成奇数面，偶数族只生成偶数面。
- 极端同点面第 1-2 章不出现，第 3 章后低权重出现，价格修正为 `+7`。
- 材质价格修正：官铁 `+3`、鎏印 `+2`、明釉 `+3`、铅封 `+2`、铜骨 `+3`。
- 材质权重：官铁 `26/24/22`，鎏印 `22/20/18`，明釉 `18/18/18`，铅封 `18/18/18`，铜骨 `16/20/24`。
- 独特骰卖出价首版按类型固定回收。
- 面组和材质组合加价未确认，后置为调参项。

## 阻塞问题

无硬阻塞。以下为实现后需注意的非阻塞缺口：

- 面组模板和价格修正尚未实测。
- 材质权重尚未调参。
- 面组和材质组合加价未确认，待市场刷新样本和结算样例后决定。

## 验证计划

- 连续刷新市场至少 30 次，记录类型、面组、材质、价格组合是否可读。
- 购买每种材质至少 1 颗并进入小关验证触发。
- 检查出千后铅封只对被选择的实体骰生效。
- 检查鎏印账本金币通关后并入钱包，失败不保留。
- 检查铜骨不读取临时小骰。
- 检查 F002 实现前清空或重置旧运行记录后，新游戏和继续游戏状态正确。
- 按 `Docs/UI_ACCEPTANCE_FEEDBACK_WORKFLOW.md` 做市场界面截图验收。
