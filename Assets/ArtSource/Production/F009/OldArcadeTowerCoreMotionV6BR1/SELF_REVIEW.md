# F009 V6B-R1 老式街机塔芯动态样片自审

审核日期：2026-07-21  
审核结论：`不通过，需整体补全演出流程`  
运行边界：继续停留在 `Assets/ArtSource/`，禁止据此接入 Unity。

## 核心判断

`60fps / 90` 帧只证明文件没有技术抽帧，不能证明演出完整。当前样片把完整结算错误压缩成了一个孤立的“积分灯珠入塔—塔体过驱”特效片段，缺少玩家能够理解这次出手的起因、推进关系和最终结果，因此观看上像只播放了中间一截。

## 硬性验收

| 检查项 | 结果 | 证据与问题 |
|---|---|---|
| 从投骰 / 停转进入结算 | 失败 | 样片从六颗已锁点骰的静止画面直接开始，`0.07s` 即点亮第三槽；没有 V4R6 高速滚轴、左到右停靠、稳定读点和自动交接。 |
| 六槽左到右主游标 | 失败 | 可见事件固定为第三槽 → 第一槽 → 第五槽，不存在第一至第六槽的完整主流程，也没有显示当前主槽在局部回环结清前保持身份。 |
| 跨骰回环因果 | 失败 | 第一槽只表现为一圈更薄的追灯；画面没有建立“第三槽是主来源、第一槽是被再次触发者”的因果，因此玩家只能看到随机槽位依次闪亮。 |
| 每骰贡献直入积分塔 | 部分通过 | 三个积分灯珠包都直入左塔，没有画骰子互联；但仅覆盖三次事件，不能代表完整六骰结算。 |
| 单骰影响重量 | 警告 | `8 / 11 / 18` 颗灯珠在实际尺寸和短飞行时间内差异过小；轻、中、重主要成为粒子数量差，而没有形成清楚不同的塔体承击。 |
| 积分包来源成立 | 失败 | 灯珠包从骰子左下方附近直接出现，没有“骰面点数 / 类型效果析出并凝成贡献包”的短动作，来源与积分包之间有视觉断层。 |
| 当前分 / 目标上下文 | 失败 | 为规避数值感而把顶部、左塔目标 / 当前、右侧规则和底部阶段信息全部清空。弱化精确数值被错误执行成删除游戏上下文，画面像空壳美术测试。 |
| 连续压力包络 | 部分通过 | 三次入塔和塔内灯排确实连续累积，没有逐击清零；但 `0.07–0.83s` 过度压缩，缺少完整六槽和局部队列节奏。 |
| 五级结果反馈 | 失败 | 动态只演示一条暴击 / 过驱分支，没有验证未达、过关、超过、远超过四种收束，也没有证明同一套连续参数能够覆盖五档。 |
| 最终高潮 | 部分通过 | 积分塔局部欠压、继电器吸合、温白过驱与 CRT 撕裂符合老式街机语言；但欠压只有约 `87ms`，更像一次灯闪，且前面没有足够累压使高潮成立。 |
| 结果落位与下一状态 | 失败 | 高潮后只保留约 `0.4s` 灯排余辉，未进入 `StageClear / StageFailed` 结果状态；MP4在中间状态结束，循环 WebP / GIF 会从满灯瞬间硬跳回全暗。 |
| 完整声音叙事 | 失败 | 只有结算点击、继电器和冲击声，没有滚轴电机、逐槽停靠、自动交接、结果锁定与状态收束；平均电平约 `-31.7dB`，主峰之外的因果提示也偏弱。 |
| 老式街机材质 | 通过 | 白炽追灯、步进灯排、烟熏玻璃、继电器火花与局部同步撕裂方向成立。 |
| 骰子原位与聚焦 | 通过 | 六骰没有离槽、大幅移动或互相连线，主要视觉重量仍在左侧积分塔。 |
| 文件技术完整性 | 通过但非验收结论 | 原速 MP4 `90/90` 帧、慢放 MP4 `180/180` 帧、WebP `90/90` 帧；技术完整不能覆盖上述演出缺失。 |

## 不完整的根因

1. 把静态四格“主来源—回声—重贡献—最终高潮”直接翻译成了四个动态节点，却没有补回真实游戏的入场与离场状态。
2. 为避免明确数值感，过度删除了低关注度但必需的目标、当前分、规则和阶段信息。
3. 为追求连续，把三次事件压缩进不足一秒；连续性成立了，但完整性、因果和重量差被牺牲。
4. 只验证塔体特效材质，没有验证“六槽主游标 + 局部连锁队列 + 五级最终结果”这一整套产品语义。
5. 使用无限循环动态图作为主展示，末帧没有结果停留，循环重置进一步放大了“演出没结束”的感觉。

## V6B-R2 必须补齐的完整链路

下一版不应继续微调 R1 的灯光参数，而应先重建完整导演结构：

| 时段建议 | 必须出现的状态 | 关键要求 |
|---:|---|---|
| `0.00–1.22s` | V4R6 投骰与左到右停靠 | 复用当前固定骰体、面内滚轴和真实锁点节奏，不另造摇骰动作。 |
| `1.22–1.42s` | 稳定读点与自动交接 | 六骰结果、目标 / 当前、规则终端保持可读；不等待第二次输入。 |
| `1.42–3.65s` | 六槽主结算与局部回环 | 主游标严格从左到右；第二槽触发第一槽回声时，第二槽主身份保持，第一槽仅短促回声并把实际贡献直送积分塔；每个主槽与下一槽交叠 `60–80ms`。 |
| `3.65–4.10s` | 目标越线与隐藏等级释放 | 当前分低关注度更新；整手等级只控制积分塔负载，不弹评级文字。暴击才使用局部欠压与温白过驱。 |
| `4.10–4.80s` | 结果落位 | 右侧终端进入通过 / 未达结果，积分塔保持最终状态，给玩家完成感；不自动循环回零。 |

建议总长约 `4.6–5.0s`，以原速 `60fps` MP4 为唯一主验收载体；聊天动态图只作辅助。

## R2 画面修正要求

- 使用实际主游戏状态底图，恢复章节、生命、金币、目标、当前分、规则终端与阶段信息；这些信息保持低亮、稳定，不做大号跳分。
- 每个积分包从当前骰面中央短促析出，而不是在骰子下方凭空出现。
- 主来源聚焦、回声聚焦和主游标保持必须有三种清楚但克制的状态，禁止使用骰子间连线。
- 轻 / 中 / 重贡献不能只改粒子数：还要连续改变积分包核心比例、吸收时长、灯排推进格数、塔壳局部回弹与余辉长度。
- 五档结果共用同一基础流程，只改变最终塔体工作状态；首个动态门禁先做一条包含第二槽回环的完整暴击手，完整性通过后再派生其余四档。
- 末帧至少保持 `0.6–0.8s`，并明确停在结果状态；禁止无限循环把满灯硬切回空塔。

## 结论

V6B-R1 只可保留为“老式街机塔体特效材质测试”，不能再称为完整结算动态样片，也不能作为 Unity 接入依据。下一步应先按 V6B-R2 的完整链路重做导演样片，而不是继续修改编码、帧率或单个特效参数。
